# Karpathy 放出去一个 Agent，周一醒来发现它跑了 700 个实验

> 素材来源：B站 BV1uwTS66Eqg（五里墩茶社）+ Karpathy GitHub/推文 + Bilevel arXiv 2603.23420
> 写作：Claude Opus 4 | 润色要求：故事性强、可读性高、中文

三月初的一个周末，Andrej Karpathy 干了一件事：他写了 630 行代码，分成三个文件，然后把一个 AI agent 放出去改他的 GPT-2 训练代码。他自己去睡了。

周一早上，他打开电脑，发现 agent 已经跑了 700 个实验，其中 20 个改动是真的有用——把 GPT-2 的训练时间从 2.02 小时压到了 1.80 小时，快了 11%。更有意思的是，agent 发现了几个他自己都没注意到的 bug：QK-Norm 缺了缩放因子导致注意力分布太散，Value Embeddings 漏了正则化，banded attention 的参数没调好，AdamW 优化器的 betas 参数全是乱的。

Karpathy 在推特上说了一句后来被反复引用的话："All LLM frontier labs will do this. It's the final boss battle."——所有大模型实验室都会这么干，这是最终 Boss 战。

然后他把代码开源了。48 小时内，1100 万人看了这条推文。Fortune 杂志给这套方法起了个名字：**The Karpathy Loop**。

---

## 三个文件，一块 GPU，一个数字

这套叫 AutoResearch 的东西，结构简单到有点不像话。

整个系统就三个文件。`prepare.py` 负责数据预处理，锁死不许动。`train.py` 是训练代码，agent 唯一能改的就是这个文件。`program.md` 是给 agent 的指令，由人来写和迭代。Karpathy 自己总结成一句话：**One GPU, one file, one metric**——一块 GPU，一个文件，一个指标。

那个指标叫 val_bpb，验证集上的 bits-per-byte，越低越好。每次实验固定跑 5 分钟，不多不少。agent 的工作循环是这样的：读当前代码 → 改 `train.py` → git commit → 训练 5 分钟 → 看 val_bpb → 如果变好了就留着，变差了就 git reset 回滚。然后进入下一轮。

这个 commit/reset 的机制是整套系统的棘轮——它只能往前走，不能往后退。每一次改进都被锁住，每一次失败都被清除。agent 不需要记住什么方案试过了没用，因为代码库本身就是记忆。

`program.md` 里有一条规则特别有意思。它写着：**LOOP FOREVER**，永远循环下去，不许停下来问人类要不要继续。原文是这么写的："The human might be asleep. You are autonomous."——人可能在睡觉，你是自主的。

还有一条审美偏好写进了指令：同等效果下，更简单的方案更好。这不是废话——如果不写这条，agent 会倾向于叠加越来越多的 trick，代码会膨胀到没人看得懂。

Karpathy 在 README 开头写了一段有点科幻味道的引言："One day, frontier AI research used to be done by meat computers..."——有一天，前沿 AI 研究曾经是肉体计算机做的。他自己也承认这个项目 "Part code, part sci-fi, and a pinch of psychosis"——一半代码，一半科幻，再加一撮精神病。

## Shopify 老板一晚上干了什么

Karpathy 开源之后，第一个跟进的大人物是 Shopify CEO Tobi Lütke。

Lütke 用同样的方法跑了一晚上，37 个实验。结果是：一个 8 亿参数的模型，性能超过了他们手工精调的 16 亿参数模型。超了 19%。

你需要停下来想一秒这意味着什么。一个参数量只有一半的模型，因为训练代码被 agent 优化过，反而比大一倍的模型跑得更好。这不是"AI 帮你省点力气"的故事，这是"你以为你已经调好了，其实还差着一截"的故事。

Shopify 团队还把这套方法用到了他们的模板引擎上。93 个自动 commit 之后，渲染速度快了 53%。

## 为什么一个简单的循环能发现人类漏掉的东西

你可能会想：Karpathy 是全世界最懂深度学习训练的人之一，一个 AI agent 怎么能发现他漏掉的 bug？

答案不是 agent 比 Karpathy 聪明。答案是 agent 比 Karpathy **勤快**。

人类研究者在调参数的时候，脑子里有一个隐含的优先级列表。学习率、batch size、层数——这些是"大旋钮"，会优先调。但 AdamW 优化器的 beta 参数？QK-Norm 的缩放因子？这些是"小旋钮"，通常被认为"默认值就挺好的"，很少有人会去碰。

agent 不知道什么是"大旋钮"、什么是"小旋钮"。它没有这个偏见。它会把所有能改的东西都试一遍，每次 5 分钟，用结果说话。700 个实验，20 个有用，命中率不到 3%——但人不会去跑那 680 个"浪费掉的"实验，所以也就永远不会撞上那 20 个改进。

这就是 loop 的力量：它不需要每次都对，它只需要不停地试，并且有一个可靠的信号告诉它什么时候对了。

## Loop Engineering：五个零件，一个灵魂

五里墩茶社在视频里梳理了 Loop Engineering 的五个构件（来自 MarkTechPost 和 Rocco Liu 的总结），我觉得这个框架很值得拿来对照 AutoResearch 看。

**自动化执行（Automation）**：agent 能自动跑起来，不需要人盯着。AutoResearch 的 "LOOP FOREVER" 就是这个。

**技能（Skill）**：agent 需要具备完成任务的基本能力。这里用的是一个足够强的代码模型，它能读懂 PyTorch 训练代码，能提出合理的修改方案。

**子代理（Sub-agents）**：复杂任务可以拆成子任务分给不同的 agent。AutoResearch 原版没用这个，但后面要讲的 Bilevel 版本用了。

**连接器（Connectors）**：agent 和外部世界的接口——能跑命令、能读文件、能 commit 代码。

**验证器（Verifier）**：这是最关键的一个。没有验证器，前面四个都是摆设。

为什么验证器是灵魂？因为 loop 的本质是一个搜索过程，而搜索需要一个信号来判断方向。AutoResearch 的验证器极其简单：跑 5 分钟训练，看 val_bpb 有没有降低。这个数字不会骗人，不需要人来判断，不需要主观审美。

这也解释了为什么 AutoResearch 能用这么简单的架构做出这么好的效果——它选了一个验证器特别干净的问题。模型训练的好坏，一个数字就能说清楚。

## 当 Loop 自己开始优化 Loop

故事讲到这里，本来可以结束了。但两个研究者 Qu 和 Lu 问了一个让人头皮发麻的问题：

**如果 AutoResearch 本身也是一种研究方法，那我们能不能用 AutoResearch 来优化 AutoResearch？**

他们把这个想法写成了论文（arXiv 2603.23420），叫 Bilevel AutoResearch——双层自动研究。

原版 AutoResearch 只有一层循环：改代码 → 训练 → 评估 → 决定保留还是回滚。他们加了第二层。

**内层（Level 1）** 和原版一样：agent 改训练代码，跑实验，看指标。

**外层（Level 2）** 做的事情不一样：它不改训练代码，它改的是**内层 agent 搜索方案的策略本身**。具体来说，外层会在运行时生成新的 Python 搜索机制，用 importlib 动态注入到系统里。

中间还有一个 **Level 1.5**，负责配置层面的事，比如决定哪些参数冻住不让内层动、哪些解冻让它去探索。

听起来很抽象，看结果就清楚了：只用 Level 1（原版方案），val_bpb 降了 0.009。加上 Level 2 之后，降了 0.045。**五倍提升**。用的是同一个 LLM（DeepSeek），没有换更强的模型。

更有意思的是 Level 2 独立发现的那些搜索策略。它自己"发明"了 Tabu Search（禁忌搜索，一种避免重复走老路的优化方法）、Multi-Scale Bandit（多尺度赌博机，在不同粒度上平衡探索和利用）、Orthogonal Exploration（正交探索，确保每次尝试的方向尽可能不同）。这些都是运筹学和优化领域里已知的经典方法——但没人教它，它自己从实验结果里摸索出来的。

---

## 为什么需要第二层

这里有一个关键洞察，论文里说得很清楚：Level 1 会陷入**确定性搜索路径**。

什么意思？同一个 LLM，给它同样的代码和同样的 prompt，它倾向于提出类似的修改方案。论文里观察到，同一个失败的提案被重复提交了 22 次。agent 不是不努力——它一直在跑——但它被困在了搜索空间的一个角落里，反复撞同一面墙。

这就是为什么光改 prompt 的措辞是不够的。论文里有一句话说得很到位："Prompt-level optimization has a ceiling — you can't discover a fundamentally new search mechanism by rewording a prompt."——提示词层面的优化有天花板，你没法通过换个说法就发现一种全新的搜索机制。

Level 2 打破了这个循环。它不是在 prompt 层面做文章，而是直接写新的 Python 代码来改变搜索的逻辑。这是一种质的区别：从"用不同的话说同一件事"变成了"用不同的方法做同一件事"。

如果你觉得这有点像"AI 在写 AI"，那确实就是。只不过它写的不是一个通用 AI，而是一个针对特定搜索任务的策略模块。边界很清晰，验证器依然在把关。

## 那我能不能用这套方法写日常代码

这大概是看完 Karpathy 的东西之后每个程序员都会问的问题。

答案取决于一件事：**你的验证能不能自动化？**

如果可以，你就能建一个 loop。能自动化验证的场景包括：测试通过率、编译是否成功、benchmark 跑分、lint 检查。这些都有一个明确的、机器可读的信号告诉你"好了"还是"没好"。

不能自动化验证的场景呢？UI 好不好看、用户体验顺不顺、代码写得"优不优雅"。这些需要人类判断，而一旦需要人来看，loop 就循环不起来了。

Rocco Liu 提出过一个 7 天行动计划，建议你从小的、有明确验证信号的任务开始试。但他同时也指出了三个风险：

**理解债务**：agent 改了 93 次代码，你真的理解每一次改动在干什么吗？如果你不理解，这些代码就变成了一个黑盒。它能跑，但你不知道为什么能跑，哪天它出了问题你也不知道为什么。

**认知投降**：当你习惯了让 agent 跑 loop 来解决问题，你自己思考问题的能力会不会退化？这不是杞人忧天——当计算器普及之后，人类的心算能力确实下降了。

**破坏性优化**：agent 只看你给它的那个指标。如果你的指标不够全面，它可能会找到一些"technically correct but practically terrible"的方案——技术上指标变好了，但实际上引入了别的问题。就像那个经典笑话：你让 AI 最大化回形针产量，它把整个地球都变成了回形针。

## 这件事真正在说什么

如果往后退一步看，Karpathy Loop 和 Bilevel AutoResearch 讲的其实是同一件事：**当你有一个可靠的验证信号时，暴力搜索比你想象的有效得多。**

人类研究者有直觉、有经验、有审美，但也有盲区、有偏见、有疲劳。机器没有前面那些优点，但也没有后面那些缺点。当验证信号足够清晰的时候，"没有偏见地试遍所有可能"比"聪明地猜少数几个方向"更管用。

Bilevel 的故事还多说了一层：不光搜索本身可以自动化，**搜索策略的搜索**也可以自动化。而且不需要更强的模型——同一个 DeepSeek，只是让它在两个层面上运转，效果就翻了五倍。这个结论比 AutoResearch 本身更深远。

Karpathy 在 3 月 5 日跑完第一轮实验后说了一句半开玩笑的话："Go relax a bit and enjoy the feeling of post-AGI."——放松一下，享受后 AGI 时代的感觉。

他当然是在开玩笑。但 700 个实验、20 个人类漏掉的改进、一个周末——这件事不需要 AGI 就已经发生了。它只需要一个 loop、一个验证器、和一块 GPU。

至于你要不要也试试？先问自己那个问题：你的验证信号，是一个数字，还是一种感觉？如果是数字——放手让它跑吧。
