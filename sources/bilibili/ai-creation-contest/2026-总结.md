## 一、赛事背景

B站于2026年6月5日正式启动"Build in Bilibili · AI创造公开赛"，这是国内首个规模化的 **Build in Public (BIP)** 赛事。核心理念是"公开做产品，观众即用户"。参赛者需使用AI工具创造可运行、可交互的产品，并在B站以连载视频形式公开创作过程，观众通过投币、弹幕参与产品迭代。

**关键数据：**
- 非专业开发者占比超70%
- 千粉以内UP主占比超70%
- 投稿创作者接近5000人（开赛一个月）
- Vibe Coding相关关键词日均搜索量飙升6倍
- 话题浏览量超1.1亿

## 三、共性经验总结

### 3.1 核心技术栈分布

| 技术领域 | 采用频次 | 代表项目 |
|---------|---------|---------|
| **ASR (语音识别)** | 高 | 百聆、贾维斯、语音交互系列 |
| **LLM (大语言模型)** | 全部 | DeepSeek, GPT, Qwen, GLM, Gemini等 |
| **TTS (语音合成)** | 高 | edge-tts, ChatTTS, Kokoro, VITS |
| **VAD (语音活动检测)** | 中 | silero-vad (百聆) |
| **Agent框架** | 高 | OpenClaw (使用最多), Hermes, Codex |
| **Live2D/VRM/3D渲染** | 中 | N.E.K.O, AI-Vtuber |
| **Vibe Coding** | 高 | 无代码/低代码产品开发 |
| **Docker部署** | 中 | N.E.K.O, 百聆 |
| **WebSocket实时通信** | 高 | 语音交互全链路 |

### 3.2 做得好的方面

1. **ASR+LLM+TTS级联架构**仍是主流，模块化设计便于独立优化和替换
2. **OpenClaw** 成为最广泛使用的Agent编排底座，社区生态丰富
3. **端到端延迟优化**：百聆实现800ms级延迟，接近实时对话体验
4. **情感化设计**：最高播放量的作品普遍具有情感共鸣点（猫娘、时空电话等）
5. **Build in Public模式**：弹幕/评论直接转化为产品需求，加速迭代
6. **低门槛社区化**：Vibe Coding使非专业开发者也能构建可运行产品
7. **多模态融合**：语音+视觉+文字+Agent执行，向"类贾维斯"体验演进

### 3.3 常见问题和坑

1. **ASR准确率瓶颈**：在嘈杂环境或方言场景下，语音识别仍是薄弱环节
2. **全链路延迟**：级联架构下，ASR→LLM→TTS串行处理，首字延迟仍是挑战
3. **LLM推理成本**：大模型推理耗时长，轻量化部署（量化蒸馏）是必须面对的问题
4. **多Agent编排复杂性**：Agent间通信、任务编排、状态同步的工程复杂度高
5. **端侧部署资源限制**：移动端/嵌入式场景下，模型体积和推理速度受限
6. **语音打断体验**：实现自然打断（barge-in）需要VAD+流式处理的精细配合
7. **长期记忆持久化**：N.E.K.O的五维记忆是亮点，但多数项目缺乏完善的记忆方案
8. **开源协议混用问题**：有专门视频讲解开源Skill的前的协议检查（BV13hjq6vENB）

## 四、对我们项目的具体建议

### 项目现状
VPS服务器 + 红米K30手机 + Hermes Agent + 语音管道（ASR→LLM→TTS）

### 4.1 可直接借鉴的设计

#### ① 百聆的架构参考
- **核心借鉴**: ASR(FunASR) → VAD(silero-vad) → LLM(DeepSeek) → TTS(edge-tts)
- **建议**: 直接复用百聆的模块化架构，其支持OpenClaw工具调用的设计与我们Hermes base高度契合
- **理由**: 百聆明确标注"重点支持OpenClaw"，端到端800ms延迟是经过验证的数据

#### ② N.E.K.O的五维记忆系统
- **核心借鉴**: 工作记忆 / 近期记忆 / 事实记忆 / 反思记忆 / 人格记忆
- **建议**: 在我们的Hermes项目中引入分级记忆架构，尤其是"事实记忆"和"人格记忆"概念
- **实施**: 短期可使用Hermes的memory插件，长期可参考N.E.K.O的memory目录结构

#### ③ N.E.K.O的Agent工具执行层
- **核心借鉴**: CUA / OpenClaw A2A / 插件三层的工具执行能力
- **建议**: 红米K30作为移动端入口，调用VPS上的OpenClaw/Hermes Agent执行任务
- **理由**: 与我们已有架构一致，验证了移动端Agent + 云端Server的可行性

#### ④ 合乎周礼翻译器的BIP模式
- **核心借鉴**: 通过B站B站Toy直接体验、开源代码共享、弹幕驱动迭代
- **建议**: 我们的项目也应该在B站创建合集，公开开发过程，获取社区反馈

#### ⑤ AI-Vtuber的多平台直播集成
- **核心借鉴**: 一套系统同时支持B站/抖音/YouTube等多个直播平台
- **建议**: 我们的语音管道应该设计为平台无关的抽象层

### 4.2 要避免的坑

1. **不要低估ASR延迟** — 红米K30的端侧ASR能力有限，建议优先使用云端ASR API（如腾讯云/火山引擎/声网），延迟可控制在200ms内
2. **避免纯离线方案** — 多数参赛作品证明纯本地的LLM推理（无GPU）无法满足实时对话需求，VPS上的云端推理是正确方向
3. **不要忽视语音打断** — 对话体验的核心指标，需要在VAD层和流式TTS层配合实现
4. **避免重造轮子** — OpenClaw/Hermes生态已有成熟的工具编排能力，不应自己实现Agent调度层
5. **注意开源协议** — 复用开源代码前必须检查协议（Apache 2.0 / MIT / GPL 兼容性）
6. **避免过度工程化** — 许多高播放量作品技术并不复杂，但情感设计和展示做得好，先跑通再优化

### 4.3 推荐的下一步优化方向

#### Phase 1 — 语音管道巩固（2周）
- [ ] 参考百聆的架构，优化我们的ASR→LLM→TTS级联，目标端到端延迟 < 1.5s
- [ ] 集成silero-vad实现语音活动检测和打断能力
- [ ] 在VPS上测试DeepSeek（最具性价比）作为主LLM引擎
- [ ] 使用edge-tts或ChatTTS作为TTS后端（均免费且质量不错）

#### Phase 2 — 移动端体验（3周）
- [ ] 参考N.E.K.O的移动端适配策略，优化红米K30上的交互体验
- [ ] 实现红米K30作为语音输入/输出终端，VPS作为推理后端的分层架构
- [ ] 添加多形态Avatar支持（至少Live2D或简单的PNGTuber形态）

#### Phase 3 — Agent能力增强（4周）
- [ ] 深度集成OpenClaw作为Hermes Agent的工具编排层
- [ ] 参考N.E.K.O的插件SDK设计，建立我们自己的插件生态
- [ ] 实现分级记忆系统（参考五维记忆），提升长期对话连贯性

#### Phase 4 — BIP社区运营（持续）
- [ ] 在B站创建项目合集，公开开发过程
- [ ] 将弹幕/评论作为产品需求池，建立反馈→迭代的闭环
- [ ] 开源部分核心模块，吸引社区贡献者

#### 推荐的ASR+LLM+TTS最佳组合（基于参赛作品数据）

| 组件 | 推荐方案 | 备选方案 | 理由 |
|------|---------|---------|------|
| **ASR** | 火山引擎/腾讯云实时ASR | FunASR (自部署) | 云端延迟低，红米K30端侧不行 |
| **VAD** | silero-vad | webrtcvad | 成熟开源，延迟低 |
| **LLM** | DeepSeek | Qwen/GPT-4o-mini | 性价比最高，中文优秀 |
| **TTS** | edge-tts / ChatTTS | Kokoro-82M | 免费、质量好、延迟可接受 |
| **Agent** | Hermes + OpenClaw (A2A) | 已有 | 这是我们的优势，需坚持 |

## 五、总结

B站AI创造公开赛展示了中国AI创作者生态的蓬勃活力。对我们项目而言，最重要的几个启示是：

1. **ASR+LLM+TTS级联架构已验证可行** — 百聆800ms延迟和N.E.K.O的实时语音交互证明这条路走通了
2. **OpenClaw生态是最大公约数** — 多数Agent类作品选择OpenClaw作为底座，我们应积极拥抱
3. **情感化设计 > 技术复杂度** — 播放量最高的作品（猫娘120万、时空电话102万、合乎周礼144万）都有强烈的情感或文化hook
4. **Build in Public是加速器** — 公开开发过程可以快速获取用户反馈和社区资源
5. **移动端Agent伴侣是蓝海** — 目前赛道内成熟的移动端Agent产品极少，红米K30+Hermes的组合有差异化优势
6. **红米K30低功耗语音管道配VPS云端推理** — 与业内最佳实践一致，方向正确

> 建议立即开始：将Hermes + 红米K30的语音助手项目以B站合集形式公开，走BIP模式，边做边获得社区反馈。
