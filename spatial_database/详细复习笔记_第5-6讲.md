## ② 知识点逐条精讲

### 第一块 数据库完整性

#### 1.1 什么是数据库完整性

**定义（可直接背）**：数据库完整性（DB Integrity）是指 DBMS 应保证的数据库的一种特性——在**任何情况下**的**正确性、有效性和一致性**。

- **广义完整性**：涵盖语义完整性、并发控制、安全控制、DB 故障恢复等
- **狭义完整性**（本讲重点）：专指**语义完整性**，DBMS 通常有专门的完整性管理机制与程序处理

**为什么会产生完整性问题？**
- 不正当的数据库操作：输入错误、操作失误、程序处理失误等

**完整性管理的作用**：
1. 防止和避免数据库中不合理数据的出现
2. DBMS 应尽可能自动防止 DB 中语义不合理现象
3. 减轻应用程序员逐条 SQL 检查的负担

**DBMS 如何自动保证完整性？**
1. DBMS 允许用户用 SQL-DDL 定义完整性约束规则
2. 当有 DB 更新操作时，DBMS 自动按照约束条件检查，确保操作符合语义完整性

#### 1.2 完整性约束的一般形式

**约束四元组（OPAR 模型，重要！）**：

```
Integrity Constraint ::= (O, P, A, R)
  O：数据集合——约束的对象（列、多列/元组、元组集合）
  P：谓词条件——什么样的约束？
  A：触发条件——什么时候检查？
  R：响应动作——不满足时怎么办？
```

#### 1.3 完整性约束分类

**按约束对象分类**：

| 类型 | 说明 |
|------|------|
| 域完整性约束条件 | 施加于**某一列**上，对给定列上要更新的某一候选值是否可以接受进行判断，孤立进行 |
| 关系完整性约束条件 | 施加于**关系/table** 上，判断候选元组是否可接受，或一个关系中若干元组与另一关系中若干元组间的联系是否可接受 |

**按约束来源分类**：

| 类型 | 来源 | 关心内容 |
|------|------|----------|
| 结构约束 | 来自于**模型**的约束（函数依赖约束、主键约束/实体完整性、外键约束/参照完整性） | 数值相等与否、是否允许空值 |
| 内容约束 | 来自于**用户**的约束（用户自定义完整性） | 元组或属性的取值范围（如 Sage 在 15~40 之间） |

**按约束状态分类**：

| 类型 | 说明 | 例子 |
|------|------|------|
| 静态约束 | 要求 DB 在**任一时候**均应满足 | Sage > 0 且 < 150 |
| 动态约束 | 要求 DB 从一状态变为另一状态时满足 | 工资只能升不能降（从 1000 改为 800 违法） |

#### 1.4 SQL 实现静态完整性：CREATE TABLE

SQL 语言支持如下几种约束：
- **静态约束**：列完整性（域完整性约束）、表完整性（关系完整性约束）
- **动态约束**：触发器 Trigger

**CREATE TABLE 有三种功能**：
1. 定义关系模式
2. 定义完整性约束
3. 定义物理存储特性

**列约束（Col_constr）**：只能应用在**单一列**上，其后的 UNIQUE、PRIMARY KEY 及 search_cond 只能是单一列的约束。

**课件完整例题（必记）**：

```sql
-- 列约束示例
CREATE TABLE Student (
    S#    char(8) NOT NULL UNIQUE,
    Sname char(10),
    Ssex  char(2) CONSTRAINT ctssex CHECK (Ssex='男' OR Ssex='女'),
    Sage  integer CHECK (Sage >= 1 AND Sage < 150),
    D#    char(2) REFERENCES Dept(D#) ON DELETE CASCADE,
    Sclass char(6)
);
-- 假定 Ssex 只能取 {男，女}，1<=Sage<=150，D# 是外键

-- 表约束示例（同时含 PRIMARY KEY 和跨列 CHECK）
CREATE TABLE Student (
    S#    char(8) NOT NULL UNIQUE,
    Sname char(10),
    Ssex  char(2) CONSTRAINT ctssex CHECK (Ssex='男' OR Ssex='女'),
    Sage  integer CHECK (Sage > 1 AND Sage < 150),
    D#    char(2) REFERENCES Dept(D#) ON DELETE CASCADE,
    Sclass char(6),
    PRIMARY KEY(S#)
);

CREATE TABLE Course (
    C#     char(3),
    Cname  char(12),
    Chours integer,
    Credit float(1) CONSTRAINT ctcredit CHECK (Credit >= 0.0 AND Credit <= 5.0),
    T#     char(3) REFERENCES Teacher(T#) ON DELETE CASCADE,
    PRIMARY KEY(C#),
    CONSTRAINT ctcc CHECK(Chours/Credit = 20)   -- 严格约束 20 学时一个学分
);

-- CHECK 子句中可含子查询
CREATE TABLE SC (
    S#    char(8) CHECK( S# IN (SELECT S# FROM Student)),
    C#    char(3) CHECK( C# IN (SELECT C# FROM Course)),
    Score float(1) CONSTRAINT ctscore CHECK (Score >= 0.0 AND Score <= 100.0)
);
```

> **关键点**：`CONSTRAINT 约束名 CHECK(条件)` 给约束命名，以便后续用 ALTER TABLE 撤销或追加。CHECK 条件可以是 SELECT-FROM-WHERE 中 WHERE 后的任意语句，**包含子查询**。

**追加或撤销约束**：使用 `ALTER TABLE`（不同系统语法有差异）。

#### 1.5 SQL 实现动态完整性：触发器 TRIGGER

**触发器定义（可直接背）**：触发器（Trigger）是一种**过程完整性约束**，是一段程序，该程序可以在特定时刻被**自动触发执行**（如在一次更新操作之前或之后执行）。主要用于实现**动态约束**以及**多个元组之间**的完整性约束。

**触发器三要素**：
1. 定义触发器的**表**
2. 激活触发器的**数据操作语句**（INSERT / DELETE / UPDATE）
3. 触发器要采取的**动作**

**触发器语义**：当某一事件发生时（BEFORE|AFTER），对该事件产生的结果（每一元组 FOR EACH ROW / 整个操作的所有元组），检查条件 search_condition，如果满足条件，则执行后面的程序段。

**触发时机与 corr_name_def（别名定义）**：

```
事件：BEFORE | AFTER { INSERT | DELETE | UPDATE … }

corr_name_def（引用更新前后的值）：
  OLD [ROW] [AS] old_row_corr_name    -- 更新前的旧元组命别名
  NEW [ROW] [AS] new_row_corr_name    -- 更新后的新元组命别名
  OLD TABLE [AS] old_table_corr_name  -- 更新前的旧 Table 命别名
  NEW TABLE [AS] new_table_corr_name  -- 更新后的新 Table 命别名
```

**课件两个例题（必记）**：

```sql
-- 例1：工资只能升不能降（动态约束经典例题）
CREATE TRIGGER teacher_chgsal BEFORE UPDATE OF salary
ON teacher
REFERENCING NEW x, OLD y
FOR EACH ROW WHEN (x.salary < y.salary)
BEGIN
    raise_application_error(-20003, 'invalid salary on update');
    -- 此条语句为 Oracle 的错误处理函数
END;

-- 例2：自动维护已学课程门数
-- 假设 student(S#, Sname, SumCourse)，SumCourse 初始为 0，每选修一门自动 +1
CREATE TRIGGER sumc AFTER INSERT ON sc
REFERENCING NEW ROW newi
FOR EACH ROW
BEGIN
    UPDATE student SET SumCourse = SumCourse + 1
    WHERE S# = :newi.S#;
END;
```

**触发器分类**：
- **按时机**：BEFORE 触发器 / AFTER 触发器
- **按粒度**：行级触发器（FOR EACH ROW，每行变化触发一次）/ 语句级触发器（整个 SQL 语句执行后触发一次）

### 完整性 vs 安全性 对比（重要！）

| 对比维度 | 数据库完整性 | 数据库安全性 |
|----------|-------------|-------------|
| **目标** | 防止**合法用户**误操作导致数据不合理（防"脏数据"） | 防止**非法用户**未授权访问数据（防"非法访问"） |
| **防护对象** | 不合理的数据（语义错误、不一致） | 非法用户、非授权操作 |
| **实现手段** | 完整性约束规则（CREATE TABLE + CHECK + TRIGGER） | 授权机制（GRANT/REVOKE）、视图、加密 |
| **SQL 子语言** | DDL（定义完整性约束） | DCL（数据控制语言） |
| **DBMS 组件** | 完整性检查机制 | 安全性控制模块 |

### 第二块 数据库安全性

#### 2.1 数据库安全性概念

**定义（可直接背）**：数据库安全性是指 DBMS 应该保证的数据库的一种特性（机制或手段）：**免受非法、非授权用户的使用、泄漏、更改或破坏**。

**安全性管理涉及的层面**：
- **社会法律及伦理方面**：私人信息受到保护，未授权人员访问私人信息会违法
- **公共政策/制度方面**：政府或组织的信息公开或非公开制度
- **安全策略**：集中管理和分散管理，**需者方知策略**（最少特权策略）
- **数据的安全级别**：绝密（Top Secret）、机密（Secret）、可信（Confidential）、无分类（Unclassified）
- **数据库系统 DBS 的安全级别**：物理控制、网络控制、操作系统控制、DBMS 控制

#### 2.2 DBMS 的四类安全机制

| 机制类型 | 内容 |
|----------|------|
| **自主安全性机制（DAC）** | 存取控制（Access Control）：通过权限在用户之间的传递，使用户**自主管理**数据库安全性 |
| **强制安全性机制（MAC）** | 通过对数据和用户**强制分类**，使不同类别用户能够访问不同类别的数据 |
| **推断控制机制** | 防止通过历史信息或公开信息（聚集信息）推断出不该被知道的信息（如个人隐私） |
| **数据加密存储机制** | 通过加密、解密保护数据，包含密钥、加密/解密方法与传输 |

> **课件未深入，教材补充（MAC 重点）**：
> - **主体（Subject）**：指系统中的活动实体，如用户或代表用户运行的进程
> - **客体（Object）**：指系统中的被动实体，如文件、数据、表等
> - **许可证级别**：给主体和客体都分配安全标记
> - **读写规则**：**向下读（Read Down）**——主体只能读安全级别 ≤ 自身的客体；**向上写（Write Up）**——主体只能写安全级别 ≥ 自身的客体（防止高密级信息流向低密级）

#### 2.3 自主安全性（DAC）详解

**实现方式**：通过授权机制实现。用户需由 DBA 创建账户并授予权限，可将自己的权限转授给其他用户，形成权限的传播和控制。

**自主安全性访问规则**：

```
AccessRule ::= (S, O, t, P)
  S：请求主体（用户/账户）
  O：访问对象（属性/字段、记录/元组、关系、数据库）
  t：访问权利（创建、增、删、改、查）
  P：谓词（拥有权利需满足的条件）
```

- `{AccessRule}` 存放在**数据字典**（系统目录）中，构成所有用户对 DB 的访问权利
- 用户多时，可以**按用户组**建立访问规则
- **目标粒度（Object granularity）**：属性/字段 → 记录/元组 → 关系 → 数据库

**两种控制方式**：

| 控制方式 | 实现 |
|----------|------|
| **按名控制安全性** | 存储矩阵（Access Matrix）：行为用户，列为数据对象，格子里填权限 |
| **按内容控制安全性** | 视图（View）：通过创建视图限制用户可见的数据内容 |

**视图在安全性中的作用**：
- 限制用户对关系中**某些列**（属性）的存取：`CREATE VIEW EmpV2 AS SELECT Pname, D# FROM Employee`
- 限制用户对关系中**某些行**（元组）的存取（结合谓词）：`CREATE VIEW EmpV3 AS SELECT * FROM Employee WHERE P# = :UserId`
- 用户定义视图后，视图成为新数据对象，参与存储矩阵与能力表描述

#### 2.4 权限级别体系

| 级别 | 权利类型 | 适用用户 |
|------|----------|----------|
| 级别1（低） | **Select**：读（读 DB、Table、Record、Attribute） | 关系级别（普通用户） |
| 级别2（中） | **Modify**：更新（Insert插入 / Update修改 / Delete删除） | 关系级别（普通用户） |
| 级别3（高） | **Create**：创建（Create/Alter/Drop 表空间、模式、表、索引、视图等） | 账户级别（程序员用户）、超级用户（DBA） |

> **重点**：级别高的权利**自动包含**级别低的权利。例如，拥有 Modify（更新）权利的用户，自动也拥有 Select（读）权利。

#### 2.5 授权语法：GRANT / REVOKE

**GRANT 语法（授权）**：

```sql
GRANT {ALL PRIVILEGES | privilege {, privilege …}}
ON [TABLE] tablename | viewname
TO {PUBLIC | user-id {, user-id …}}
[WITH GRANT OPTION];
```

参数说明：
- `user-id`：由 DBA 创建的合法账户
- `PUBLIC`：允许所有有效用户使用授予的权利
- `privilege`：SELECT | INSERT | UPDATE | DELETE | ALL PRIVILEGES
- `WITH GRANT OPTION`：允许被授权者**继续传播**这些权利给其他用户

**课件例题（结合员工管理数据库 Employee(P#, Pname, Page, Psex, Psalary, D#, HEAD)）**：

```sql
-- 员工管理人员 Emp2001 获得所有权利
GRANT ALL PRIVILEGES ON Employee TO Emp2001;

-- 收发员 Emp5001 只能访问姓名和部门（通过视图）
CREATE VIEW EmpV2 AS SELECT Pname, D# FROM Employee;
GRANT SELECT ON EmpV2 TO Emp5001;

-- 所有员工只能查询自己的记录（通过视图 + UserId 谓词）
CREATE VIEW EmpV3 AS SELECT * FROM Employee WHERE P# = :UserId;
GRANT SELECT ON EmpV3 TO PUBLIC;

-- 部门领导 Emp0021 能查询其部门所有人员情况
CREATE VIEW EmpV4 AS SELECT * FROM Employee WHERE HEAD = :UserId;
GRANT SELECT ON EmpV4 TO Emp0021;
```

> **注意**：授予视图访问的权利，**并不意味着**授予基本表访问的权利（两个级别：基本关系级别和视图级别）。授权者授予的权利必须是授权者**已经拥有**的权利。

**REVOKE 语法（撤权）**：

```sql
REVOKE {ALL PRIVILEGES | priv {, priv …}}
ON tablename | viewname
FROM {PUBLIC | user {, user …}};
```

**课件例题**：
```sql
REVOKE SELECT ON Employee FROM UserB;
```

#### 2.6 数据库角色 ROLE

> **课件未详细讲解，教材补充**：

#### 2.7 审计（Audit）与数据加密

> **课件未深入讲解，教材补充**：

### 第三块 数据库并发控制

#### 3.1 事务（Transaction）

**事务定义（两个角度，必背）**：

**宏观性**（应用程序员视角）：一个存取或改变数据库内容的程序的一次执行，或者说**一条或多条 SQL 语句的一次执行**被看作一个事务。事务一般由应用程序员提出，有开始和结束，结束前需要**提交（Commit）或撤消（Rollback）**。

```sql
BEGIN TRANSACTION
  EXEC SQL …
  …
  EXEC SQL …
  EXEC SQL COMMIT WORK | EXEC SQL ROLLBACK WORK
END TRANSACTION
```

**微观性**（DBMS 视角）：对数据库的一系列**基本操作（读、写）** 的一个整体性执行。

**事务与程序的关系**：
1. 程序是**静止**的，事务是**动态**的；事务是程序的执行而不是程序本身
2. 一个程序可分为多个独立部分同时执行，每一执行则是一个不同的事务，即**一个程序中可包含多个事务**

#### 3.2 事务的四大特性：ACID

| 特性 | 英文 | 内容（课件原话）|
|------|------|----------------|
| **原子性** | Atomicity | DBMS 能够保证事务的一组更新操作是原子不可分的，即对 DB 而言，**要么全做，要么全不做** |
| **一致性** | Consistency | DBMS 保证事务的操作状态是正确的，符合一致性的操作规则，不能出现三种典型的不一致性。它是进一步由隔离性来保证的 |
| **隔离性** | Isolation | DBMS 保证并发执行的多个事务之间**互相不受影响**。例如两个事务 T1 和 T2，即使并发执行，也相当于先执行了 T1 再执行 T2，或先执行 T2 再执行 T1 |
| **持久性** | Durability | DBMS 保证**已提交事务的影响是持久的**，被撤销事务的影响是可恢复的 |

> **换句话说**：具有 ACID 特性的若干数据库基本操作的组合体被称为事务。

#### 3.3 并发操作带来的问题

**经典场景（飞机订票，课件例1）**：
- 甲事务读出机票余额 A=16
- 乙事务读出机票余额 A=16
- 甲卖出一张，A←A-1=15，写回数据库
- 乙卖出一张，A←A-1=15，写回数据库
- **结果**：明明卖出两张机票，数据库中机票余额只减少 1（丢失修改）

**三类并发问题对比表**（课件提及，教材详解）：

| 问题 | 别名 | 场景描述 | 危害 |
|------|------|----------|------|
| **丢失修改** | Lost Update | T1 和 T2 先后读 A，T1 修改后 T2 再修改并写回，T1 的修改被覆盖 | 飞机票少减1 |
| **不可重复读** | Non-Repeatable Read | T1 读 A，T2 修改 A 并提交，T1 再读 A，两次读值不同 | 同一事务内读同一数据值不同 |
| **读"脏"数据** | Dirty Read | T1 修改 A，T2 读 A，T1 回滚，T2 读到了不存在的"脏"数据 | 读到了未提交后又被撤销的数据 |

#### 3.4 事务调度（Schedule）

**调度**：一组事务的基本步（读、写、其他控制操作如加锁、解锁等）的一种**执行顺序**称为对这组事务的一个调度。

**并发（并行）调度**：多个事务从宏观上看是**并行执行**的，但其微观上的基本操作（读、写）则是**交叉执行**的。

#### 3.5 封锁（Locking）机制

**封锁定义**：封锁是指事务 T 在对某个数据对象操作之前，对其**加锁**。在 T 未释放锁之前，其他事务不能更新此数据对象。

**并发控制的主要技术**：
1. **封锁（Locking）** ← 本课重点
2. 时间戳（Timestamp）
3. 乐观控制法

**基本锁规则**：
- 每一数据元素都有一唯一的锁
- 每一事务读写数据元素前，要获得锁
- 如果被其他事务持有该元素的锁，则要等待
- 事务处理完成后要释放锁

**锁的四种类型（课件明确列出）**：

| 锁类型 | 符号 | 含义 |
|--------|------|------|
| **排他锁（写锁）** | X（Exclusive） | 只有**一个**事务能读、写，其他任何事务都不能读、写 |
| **共享锁（读锁）** | S（Shared） | **所有**事务都可以读，但任何事务都不能写 |
| **更新锁** | U（Update） | 初始读，以后可**升级**为写（防止死锁的优化手段） |
| **增量锁** | I（Incremental） | 增量更新（例如 A=A+x），区分增量更新和其他类型的更新 |

**X 锁与 S 锁效果对比（课件图示文字化）**：

| 加锁类型 | 对当前事务 T | 对其他事务 Ti |
|----------|-------------|--------------|
| **X锁**（排他锁/写锁） | T 可读、可修改 A | 禁止加任何锁（禁止读，禁止写） |
| **S锁**（共享锁/读锁） | T 可读、不可改 A | 可以加 S 锁（可以读），禁止加 X 锁（不能写） |

**X 锁与 S 锁相容矩阵**（教材标准）：

|  | X | S | 无锁 |
|--|---|---|------|
| **X** | 不相容 | 不相容 | 相容 |
| **S** | 不相容 | 相容 | 相容 |

> 说明：行表示"已有的锁"，列表示"申请的锁"；相容表示可以同时持有。

#### 3.6 封锁粒度（Locking Granularity）

**封锁粒度**：封锁数据对象的大小。

**粒度单位（从细到粗）**：
```
属性值 → 元组 → 元组集合 → 整个关系 → 整个DB
某索引项 → 整个索引
```

| 粒度方向 | 并发度 | 封锁开销 |
|----------|--------|----------|
| 粒度小（从前往后，如属性值→元组） | 并发度**大** | 封锁开销**也大**（管理的锁多） |
| 粒度大（从后往前，如整个DB） | 并发度**小** | 封锁开销**小** |

> **注意**：课件此处表述："由前往后（粒度从小到大）：并发度小，封锁开销小"——对应粒度大→并发度低、开销低。请结合上下文理解：粒度越大，一个锁锁住的范围越大，能并发的事务越少（并发度低），但需要维护的锁数量少（开销小）。

#### 3.7 三级封锁协议（课件未详述，教材补充）

> **课件 PDF 未给出三级封锁协议的详细内容**，以下为教材标准内容，需结合教材背诵：

| 级别 | 加锁要求 | 解决的问题 |
|------|----------|-----------|
| **一级封锁协议** | 事务 T 在修改数据 A 之前必须先对其加 X 锁，直到事务结束才释放 | 解决**丢失修改**问题 |
| **二级封锁协议** | 在一级基础上，加上读 A 前加 S 锁，读完立即释放 S 锁 | 解决丢失修改 + **读脏数据**问题 |
| **三级封锁协议** | 在一级基础上，加上读 A 前加 S 锁，直到事务结束才释放 S 锁 | 解决丢失修改 + 读脏数据 + **不可重复读**问题 |

#### 3.8 活锁与死锁（课件未详述，教材补充）

> **课件 PDF 未深入讲解，教材补充**：
>
> **活锁**：某事务永远在等待而得不到锁。解决方法：**先来先服务**策略。
>
> **死锁**：两个或两个以上事务互相等待对方释放锁，形成循环等待。
> - **预防**：一次封锁法（一次申请所有需要的锁）、顺序封锁法（规定锁的申请顺序）
> - **诊断**：超时法（等待超过时限则认为死锁）、等待图法（检测等待图是否有环）
> - **解除**：选择一个代价最小的事务，将其撤销，释放其持有的锁

#### 3.9 两段锁协议（2PL，课件未详述，教材补充）

> **课件 PDF 未详述，教材补充**：
>
> **两段锁协议（2PL）**：指所有事务必须分两个阶段对数据加锁和解锁：
> - **第一阶段（扩展阶段/加锁阶段）**：事务可以申请获得任何数据上的任何类型的锁，但不能释放任何锁
> - **第二阶段（收缩阶段/解锁阶段）**：事务可以释放任何数据上的任何类型的锁，但不能再申请任何锁
>
> **与可串行化的关系**：若所有事务均遵守两段锁协议，则这些事务的任何并发调度策略都是**可串行化**的（是可串行化的充分条件，不是必要条件）。

### 第四块 数据库恢复

#### 4.1 恢复概念

**DBMS 存储体系**：利用**内存（主存）** 和**外存（辅存）** 进行数据库管理。内存中分为程序数据（事务数据）和系统数据。

**数据库故障恢复**：把 DB 由当前**不正确状态**恢复到已知为**正确的某一状态**。

**恢复需要保证事务的**：
- **原子性**：事务的所有操作，要么全都执行，要么全都不执行
- **持久性**：已提交的事务对数据库产生的影响是持久的，未提交的事务对数据库不应有影响

#### 4.2 三类故障对比表

| 故障类型 | 原因 | 影响范围 | 恢复方式 |
|----------|------|----------|----------|
| **事务故障** | 某一个程序（事务）自身运行错误（如运算溢出、违反完整性约束等） | 影响该程序（事务）本身 | Undo（撤消）/ Redo（重做）该事务 |
| **系统故障** | 掉电、非正常关机等 | 影响正在运行的事务以及数据库缓冲区（DB Buffer），缓冲区涉及正在运行和已经运行的事务 | 依据运行日志 Undo 未完成事务，Redo 已完成但未写回的事务 |
| **介质故障** | 介质损坏（磁盘物理损坏等） | 影响全面：既影响内存中的数据，又影响介质中存储的数据 | 用副本（Copy）替换 + 依据运行日志 Redo |

#### 4.3 恢复的两大技术

**技术一：数据转储（备份）**

**副本（Copy）**：在某一时刻，对数据库在其他介质存储上产生的另一份**等同记录**。用副本替换被损坏的数据库。

> **课件未深入，教材补充**（数据转储分类）：
> - **静态转储**：在系统中无运行事务时进行转储，转储期间不允许 DB 的存取修改，简单可靠
> - **动态转储**：转储期间允许对数据库进行存取或修改，必须结合运行日志
> - **海量转储**：每次转储全部数据库
> - **增量转储**：只转储上次转储后更新过的数据

**技术二：运行日志（System Log）**

**运行日志定义**：DBMS 维护的一个**文件**，以流水方式记录了**每一个事务**对数据库的**每一次操作**及操作顺序。

**特点**：
- 运行日志直接写入**介质存储**上，保持正确性
- **先写日志（WAL: Write-Ahead Logging）原则**：当事务对数据库进行操作时，**先写运行日志**；写成功后，再与数据库缓冲区进行信息交换

#### 4.4 Undo 与 Redo

| 操作 | 适用场景 |
|------|----------|
| **Undo（撤消）** | 事务在发生故障时**未结束（未提交）**，撤消该事务的所有操作，消除未提交事务对数据库的影响 |
| **Redo（重做）** | 事务在发生故障时**已正确结束（已提交）**，但修改内容可能未写回 DB，重做该事务保证持久性 |

**事务故障恢复**：通过 Redo（重做已提交事务）和 Undo（撤消未完成事务）来恢复。

#### 4.5 检查点（Checkpoint）技术

**检查点定义（可直接背）**：DBMS 在运行日志中**定期**设置和更新的时刻。在该时刻，DBMS **强制使内存 DB Buffer 中的内容与介质 DB 中的内容保持一致**，即将 DB Buffer 更新的所有内容写回 DB 中。

**检查点的意义**：检查点之前，内存中数据与介质中数据是保持一致的。

**系统故障恢复策略（基于检查点）**：
- **检查点之前结束的事务**：不需要恢复（已经写回 DB）
- **检查点之后结束或发生的事务**：需要依据运行日志进行恢复（不能确定是否写回 DB）
  - 故障点**前**结束的：**Redo（重做）**
  - 故障点时刻**未结束**的：**Undo（撤消）**

#### 4.6 介质故障恢复

**步骤**：
1. 用**副本（Copy）** 替换被破坏的数据库（恢复到副本时刻的状态）
2. 由于介质故障影响全面，在用副本恢复后还需要**依据运行日志**进行恢复（Redo 副本时刻之后已提交的事务）

**转储点选择原则**：
- 过频：影响系统工作效率
- 过疏：造成运行日志过大，也影响系统运行性能
- 备份转储周期与运行日志的大小密切相关

#### 4.7 总结口诀

```
三类故障：事务故障、系统故障、介质故障
三种恢复手段：事务的撤消与重做（Undo/Redo）、运行日志、备份（副本）
两个重要时刻：检查点（Checkpoint）、转储点（Copy Point）
```

## ④ 简答/论述题预测（第五讲）

### Q1：数据库完整性与安全性有何区别？

**答**（分四点）：
1. **目标不同**：完整性防止合法用户误操作导致数据语义错误；安全性防止非法用户未授权访问数据
2. **防护对象不同**：完整性针对不合理的数据（语义错误）；安全性针对非法用户和非授权操作
3. **实现手段不同**：完整性用 DDL 定义约束规则（CHECK、TRIGGER）；安全性用 DCL 实现授权（GRANT/REVOKE）
4. **DBMS 组件不同**：完整性由完整性检查机制保障；安全性由安全性控制模块保障

### Q2：触发器是什么？与普通 CHECK 约束有何不同？

**答**：
1. 触发器是一种过程完整性约束，是一段可以在特定时刻自动触发执行的程序
2. CHECK 约束只能检查**单值条件**，且只在当前行/列范围内有效；触发器可以实现跨表的、涉及多个元组的复杂约束，以及**动态约束**（如工资只能升不能降这类依赖于新旧值比较的约束）
3. 触发器可以在 BEFORE 或 AFTER 时机触发，可以执行任意 SQL 操作，表达能力远强于 CHECK

### Q3：事务的 ACID 特性各是什么意思？

**答**（逐条）：
1. **原子性**：事务的一组更新操作是原子不可分的，对 DB 而言要么全做，要么全不做
2. **一致性**：事务的操作状态是正确的，符合一致性操作规则，不出现数据不一致
3. **隔离性**：并发执行的多个事务之间互相不受影响，相当于串行执行
4. **持久性**：已提交事务的影响是持久的，被撤销事务的影响是可恢复的

### Q4：数据库有哪三类故障？各如何恢复？

**答**：
1. **事务故障**：程序自身运行错误。恢复方式：Undo（撤消）该事务
2. **系统故障**：掉电、非正常关机。恢复方式：利用运行日志，Redo 已提交事务，Undo 未完成事务
3. **介质故障**：介质物理损坏。恢复方式：先用副本替换，再利用运行日志 Redo 副本时刻之后的已提交事务

### Q5：什么是检查点？它如何加速系统故障恢复？

**答**：
1. 检查点是 DBMS 在运行日志中定期设置的时刻，在该时刻 DBMS 强制使 DB Buffer 与介质 DB 保持一致
2. 检查点之前结束的事务已经写回 DB，**不需要恢复**；只需对检查点之后的事务进行恢复（故障点前结束的 Redo，故障点时未结束的 Undo），大大缩短了需要扫描的日志范围，提升恢复效率

### Q6：GRANT 和 REVOKE 语句的作用是什么？WITH GRANT OPTION 的含义？

**答**：
1. GRANT 用于向用户授予操作数据库对象（表、视图等）的权利
2. REVOKE 用于撤销已授予用户的权利
3. WITH GRANT OPTION 允许被授权者将已获得的权利继续传播给其他用户；没有该选项则被授权者只能自己使用，不能转授

## ⑤ 易错点提醒（第五讲）

1. **完整性 ≠ 安全性**：完整性是防"脏数据"（合法用户误操作），安全性是防"非法用户"。考试常考区别，切勿混淆。

2. **CONSTRAINT 命名约束的作用**：命名的目的是后续可以用 ALTER TABLE 撤销或追加该约束。无名约束无法通过名字删除。

3. **CHECK 子句的威力**：CHECK 条件内可以含子查询，如 `CHECK(S# IN (SELECT S# FROM Student))`，这是一个常见考点。

4. **触发器中 OLD/NEW 引用**：
   - `OLD ROW` / `NEW ROW`：行级触发器中引用单行的旧值/新值
   - `OLD TABLE` / `NEW TABLE`：语句级触发器中引用操作前/后的整个临时表
   - 注意在 Oracle 中用 `:newi.属性名` 引用新行值

5. **视图授权的级别问题**：给用户授予视图的 SELECT 权限，**不等于**给用户授予基本表的 SELECT 权限，两者是独立的两个权限层级。

6. **封锁粒度的双刃剑**：粒度越大，并发度越低，但封锁管理开销越小；粒度越小，并发度越高，但封锁管理开销越大。二者是权衡关系。

7. **运行日志先写原则（WAL）**：必须先写日志，再执行数据库操作（与缓冲区交换），这样即使在操作途中宕机，日志中也有完整记录可供恢复。顺序不能颠倒。

8. **检查点与转储点区分**：
   - **检查点**（Checkpoint）：定期强制 Buffer→DB 同步，用于**系统故障恢复**的起始点
   - **转储点**（Copy Point）：定期做副本备份，用于**介质故障恢复**的起始点
   - 两者不是同一回事！

9. **ACID 中的"一致性"**：一致性是目的，原子性、隔离性、持久性是手段，一致性进一步由隔离性来保证。

10. **X 锁与 S 锁的区别**：X 锁加后其他任何人都不能加锁（既不能读也不能写）；S 锁加后其他人还可以再加 S 锁（可以读），但不能加 X 锁（不能写）。

### 第一部分 数据建模与数据库设计概述

#### 1.1 为什么要数据建模

**课件例子——折扣政策管理**：

需求：不同客户有不同折扣；不同产品有不同折扣；不同客户购买不同产品有不同折扣。

- 设计方案1、2、3各有不同的表结构，每种设计有不同问题
- 结论：**数据需求的理解与表达**是数据库设计的核心，设计方案错了会导致无法正确表达业务规则、产生空值、数据冗余等问题

#### 1.2 概念模型 vs 数据模型

| 对比维度 | 概念模型（Conceptual Model） | 数据模型（Data Model） |
|----------|-----------------------------|-----------------------|
| **表达的世界** | 信息世界（观念世界）：对现实世界的理解与抽象 | 计算机世界（数据世界） |
| **独立性** | 独立于计算机系统 | 逻辑层独立于物理系统；物理层依赖于物理系统 |
| **典型模型** | E-R 模型、O-O 模型 | 关系模型、网状模型、层次模型、O-O 模型 |
| **作用阶段** | 概念设计阶段 | 逻辑设计和物理设计阶段 |

**抽象的四步骤（课件原话）**：
1. **理解**：理解现实中的卡片、单据、表格、报表……
2. **区分**：区分表与表、表内数据项、数据项之间关系、表之间关系
3. **命名**：命名表、命名数据项、命名表之间的联系
4. **表达**：用其他人能理解的表达方法（E-R 图 / Crow's Foot / IDEF1X）

#### 1.3 数据库设计六大步骤（核心必考）

```
需求分析 → 概念结构设计 → 逻辑结构设计 → 物理结构设计 → 数据库实施 → 运行与维护
```

| 步骤 | 主要任务 | 产出物 |
|------|----------|--------|
| **需求分析** | 准确了解应用（数据需求与处理需求），调查业务流程、数据、处理规则 | 需求规范说明书（含数据字典 DD + 数据流图 DFD + 功能模块图） |
| **概念结构设计** | 形成**独立于 DBMS** 的概念模型 | 全局 E-R 图（由局部 E-R 图集成） |
| **逻辑结构设计** | 将概念模型**转换为某 DBMS 支持的数据模型**（通常是关系模型） | 关系模式集合（经过规范化优化） |
| **物理结构设计** | 选取最适合应用环境的**物理结构**（存储结构、存取方式） | 物理数据库结构方案 |
| **数据库实施** | 建立实际数据库结构，开发调试应用程序，加载数据并试运行 | 可运行的数据库系统（经功能/性能测试） |
| **运行与维护** | DBA 日常维护（转储恢复、安全、性能监督、调整、重组重构） | 持续运行的数据库系统 |

**设计过程特点**：
- 分析阶段和实现阶段交替进行，不断**修改与调整**（迭代过程）
- **规范设计法基本思想**：过程迭代和逐步求精

### 第二部分 E-R 模型（数据建模核心）

#### 2.1 E-R 模型概述

**E-R 模型**：Entity-Relationship Model，1976 年由 P.P.S.Chen 提出，用 E-R 图描述概念模型。

**E-R 模型的基本观点**：世界是由一组称作**实体**的基本对象和这些对象之间的**联系**构成的。

**E-R 模型给出的一组基本概念**：
- 实体（Entity）
- 属性（Attribute）
- 联系（Relationship）
- 关键字/码（Key）

#### 2.2 实体（Entity）

**实体定义**：客观存在并可相互区分的事物。

**实体 vs 实例**：
- **实体**（型）：一类实体，一类具有相同或相似特性的事物（如"读者"这个类）
- **实例**（值）：一个个具体的个体（如"张三"这位读者）

**实体的表示**：`实体名称(属性名1, 属性名2, ...)` 表示实体的型；`实体名称(属性值1, 属性值2, ...)` 表示实体的实例

#### 2.3 属性（Attribute）

**属性定义**：实体所具有的某一方面特性。

**属性的类型（重要，注意区分）**：

| 属性类型 | 定义 | 在关系模型中的处理 |
|----------|------|-------------------|
| **单一属性** | 原子性，不可再分 | 直接使用 |
| **复合属性** | 由若干子属性组成（如"家庭住址"含省份、详细住址） | **必须转化为单一属性**（满足1NF） |
| **单值属性** | 每个实例的该属性值只有一个 | 直接使用 |
| **多值属性** | 每个实例的该属性值有多个（如电话号码） | **必须转化为单值属性**（满足1NF），或独立成一个关系 |
| **可空值属性** | 该属性值可以是空值（NULL） | 允许 NULL |
| **非空值属性** | 该属性值不能为空 | NOT NULL 约束 |
| **导出属性** | 由其他属性计算而得（如"年龄"由"出生年份"推算） | 通常不存储，按需计算 |

#### 2.4 关键字/码（Key）

**关键字定义（可直接背）**：实体中能够用其值**唯一区分开每一实例**的属性或属性组合。

**例**：读者（借书证号，姓名，年龄，性别，家庭住址）中，借书证号是唯一的，其他属性可能有相同值，所以**借书证号是关键字**。

#### 2.5 联系（Relationship）

**联系定义**：一个实体的实例和其他实体实例之间所可能发生的联系。

**联系的度（元）**：参与发生联系的实体的数目。

| 联系类型 | 说明 | 例子 |
|----------|------|------|
| **一元联系**（自联系） | 同一实体内的实例之间的联系 | 零件的装配关系（零件A由零件B和零件C装配）、部门的上下级关系 |
| **二元联系** | 两个不同实体之间的联系 | 读者借阅图书 |
| **三元联系** | 三个实体之间的联系 | 供货商供应零件到工程项目 |

**角色（Role）**：当同一实体的不同实例参与一个联系时，为区别各实例参与联系的方式，需要显式指明其**角色**。

**例（一元联系中的角色）**：
```
零件(零件号, 零件名, …)
装配(父件号, 子件号)  -- 父件和子件都是零件，但角色不同
```

#### 2.6 联系的基数类型

| 基数类型 | 定义 | 例子 |
|----------|------|------|
| **1:1（一对一）** | 实体A的实例只能和实体B的**一个**实例发生联系，反之亦然 | 一个"经理"只管理一个"商店"，一个"商店"只有一个"经理" |
| **1:m（一对多）** | 实体A的实例能和实体B的**多个**实例发生联系，但实体B的实例只能和实体A的**一个**实例联系 | 一个"画家"可绘制多幅"作品"，一幅"作品"只由一个"画家"完成 |
| **m:n（多对多）** | 实体A的实例可和实体B的**多个**实例联系，反之亦然 | 一位同学可以选学多门课程，一门课程可由多个人选学 |

**基数的精确表示（MinCard..MaxCard）**：

- 书架参与"存放图书"联系的基数为 **(0..m)**：一个书架可以存放 0 或多本图书
- 图书参与此联系的基数为 **(1..1)**：一本图书只能存放在 1 个书架上

**完全参与 vs 部分参与**：

| 类型 | 定义 | 最小基数 | 标记 |
|------|------|----------|------|
| **完全参与联系** | 该端实例至少有一个参与到联系中 | 最小基数为 1 | (1..m) 或双线 |
| **部分参与联系** | 该端实例可以不参与联系 | 最小基数为 0 | (0..m) 或单线 |

#### 2.7 Chen 方法（E-R 图画法规范）

**图元对应表**：

| 概念 | 图形 | 特殊说明 |
|------|------|----------|
| **实体** | 矩形框（Rectange） | 实体名称写在框内 |
| **属性** | 椭圆（Ellipse） | 与实体或联系用直线连接 |
| 多值属性 | **双线椭圆** | 一个实例有多个值 |
| 导出属性 | **虚线椭圆** | 由其他属性计算得出 |
| **关键字属性** | 属性名**下加下划线** | 必须标注 |
| 复合关键字 | 标有**相同数字** | |
| **联系** | 菱形框（Diamond） | 联系名称写在框内，必须命名 |
| 实体→联系的连线 | 直线 | 连线上标注基数或箭头 |

**联系基数的 Chen 方法标注规则**：

| 联系类型 | 标注方式 |
|----------|---------|
| **1:1 联系** | 两端都画**箭头直线**（由联系指向实体），或两端标 1 |
| **1:m 联系** | 1 端为**箭头直线**，多端为**无箭头直线**；或 1 端标 1，多端标 m |
| **m:n 联系** | 两端都是**无箭头直线**；或两端标 m、n |
| **完全参与** | 实体与联系之间连接的是**双直线** |
| **部分参与** | 实体与联系之间连接的是**单直线** |

也可以用 MinCard..MaxCard 精确标注：直线旁标 1..1、0..1、1..m、0..m。

**Chen 方法建模五步骤（课件仓储管理案例）**：
1. **Step1**：理解需求，寻找实体（能用"一个个、一件件"等量词形容的，且要覆盖所有可独立管理的事物类别）
2. **Step2**：用属性刻画每一个实体（至少给出重要属性）
3. **Step3**：确定每一个实体的关键字/码（必须标记清晰）
4. **Step4**：分析实体之间的联系，标记联系基数（标记基数很重要，不同基数可能导致不同设计方案）
5. **Step5**：检查是否覆盖了所有需求（E-R 图是对需求的理解与表达，必须覆盖需求；绘制要符合 Chen 方法规范）

**图书管理 E-R 图完整示例（课件原例）**：

```
实体：读者（借书证号[下划线], 姓名, 年龄, 性别, 家庭住址）
      图书（书号[下划线], 书名, 出版日期, 出版社）
      书架（书架号[下划线], 房间号）
联系：借阅（借书证号[FK], 书号[FK], 借阅日期, 归还日期）  ← 联系也可有属性
      保存（书号[FK], 书架号[FK]）
基数：读者 0..m 借阅 0..m 图书  ← 多对多
      图书 1..1 保存 0..n 书架  ← 每本图书放在一个书架，每个书架放0到多本图书
```

#### 2.8 Crow's Foot 方法

**图元规则（与 Chen 方法对比）**：

| 元素 | Chen 方法 | Crow's Foot 方法 |
|------|-----------|-----------------|
| **实体** | 矩形框（实体名在框内） | 矩形框（实体名在**横线上面**，属性列在横线下面） |
| **属性** | 椭圆，与实体用直线连接 | 直接写在**实体框横线下方** |
| **关键字** | 属性名加下划线 | 属性名**加下划线** |
| **联系** | 菱形框 | 菱形框（可省略，直接以联系名替代） |
| **联系基数** | 直线旁标数字或箭头 | **特殊符号**（三叉线=多，竖线=1，圆圈=0，双竖线=强制1） |

**Crow's Foot 基数符号含义**：
- `||`（两条竖线）：恰好 1（强制，完全参与，最小基数=1）
- `|○`（竖线+圆圈）：0 或 1（部分参与，最小基数=0，最大基数=1）
- `>|`（三叉线+竖线）：1 或多（完全参与，最小基数=1）
- `>○`（三叉线+圆圈）：0 或多（部分参与，最小基数=0）

### 第三部分 数据库设计六阶段详解

#### 3.1 需求分析阶段

**任务**：
- 详细调查要处理的对象（组织、部门、企业等）
- 充分了解原系统（手工系统或计算机系统）工作概况
- 明确系统的各种需求，确定新系统的功能
- **调查重点**：**"数据"和"处理"**

**调查方法**：查阅文档、面谈、观察业务操作、问卷调查等

**需求分析的四项具体任务**：

**任务1：确定业务需求**（包含）：
  1. 了解业务流程
  2. 明确组成业务的数据，建立**数据字典**
  3. 明确数据处理任务
  4. 明确业务规则

**数据字典（Data Dictionary）的组成**（教务管理系统示例）：

| 组成部分 | 说明 | 例子 |
|----------|------|------|
| **数据项** | 最小数据单元，包含含义、别名、类型、长度、取值范围、与其他数据项关系等 | 学号：字符型8位，范围00000000~99999999，前两位为入学年，后六位顺序编号 |
| **数据结构** | 若干数据项的有意义组合 | 学生：学号+姓名+性别+出生日期+班号 |
| **数据流** | 数据在系统中的传输路径（在 DFD 中体现） | — |
| **数据存储** | 数据在系统中静态保存的位置 | — |
| **处理过程** | 各处理过程的说明 | — |

**任务2：确定数据需求**：
  - 通过分析制作**数据流图（DFD: Data Flow Diagram）**
  - DFD 描述数据在系统中流动和处理的过程
  - 课件示例：教务管理系统顶层 DFD（学籍管理、课程管理、选课管理、成绩管理四个处理过程，相互之间通过学生信息、课程信息、选课信息交换数据）

**任务3：确定处理需求**：
  - 用**系统功能模块图**表示
  - 课件示例：教务管理系统 → 学籍管理/课程管理/选课管理/成绩管理 → 各子模块

**任务4：编写需求分析说明书（需求规范说明书）**，内容包括：
  - 系统概况（目标、范围、背景、历史和现状）
  - 系统的原理和技术，对原系统的改善
  - 系统总体结构与子系统结构说明
  - 系统功能说明
  - 数据处理概要、工程体制和设计阶段划分
  - 系统方案及技术、经济、功能和操作上的可行性

#### 3.2 概念结构设计阶段

**方法**：通常采用**自底向上**的方法（先局部后全局）。

**步骤**：
```
需求分析（DFD + DD）
     ↓
数据抽象，设计局部 E-R 图（局部视图）
     ↓  征求用户意见
视图集成 → 全局 E-R 图
     ↓
逻辑结构设计
```

**E-R 图集成时的冲突类型（教材补充，课件提及但未展开）**：

| 冲突类型 | 说明 | 解决方法 |
|----------|------|----------|
| **属性冲突** | 同一属性在不同局部 E-R 图中有不同定义（类型、取值范围、单位等不同） | 协商统一标准 |
| **命名冲突** | 同名异义（不同实体用了相同名字）或异名同义（同一实体用了不同名字） | 统一命名 |
| **结构冲突** | 同一对象在不同局部 E-R 图中被抽象为不同结构（实体vs属性、联系类型不同等） | 统一结构定义 |

#### 3.3 逻辑结构设计阶段

**设计流程**：
```
概念结构E-R图 → 关系数据模型（初始关系模式集）→ 规范化优化 → DBMS下的数据模型
```

**步骤一：E-R 图转换为关系模式**

**基本转换规则1——实体转换**：
- E-R 图的**实体** → 一个**关系（表）**
- E-R 图的**属性** → 关系的**属性（列）**
- E-R 图的**关键字** → 关系的**关键字（主键）**

**课件例子**：
```
图书(书号, 书名, 出版日期, 出版社)
读者(借书证号, 姓名, 年龄, 性别, 家庭住址)
书架(书架号, 房间号)
```

**基本转换规则2——联系的转换**（教材标准，课件提到）：

| 联系类型 | 转换规则 |
|----------|---------|
| **1:1 联系** | 可将联系合并到任意一方实体的关系中（加入另一方的主键作为外键），不必单独建表；也可单独建一个联系表（含两端主键） |
| **1:m 联系** | 将联系的属性以及 1 端实体的主键加入到 m 端实体的关系中作为外键，不必单独建表 |
| **m:n 联系** | **必须单独建一个联系表**，该表包含两端实体的主键（合在一起构成联系表的组合主键）以及联系本身的属性 |

**步骤二：关系模式规范化（优化）**

**三个范式的定义与应用**：

**第一范式（1NF）**：
> 如果关系（表）中的每一分量都是**不可分的**（原子性），则称该关系满足 1NF。

- **不符合1NF的情况**：含复合属性或多值属性
- **处理方法**：
  - 将复合属性处理为简单属性：`Star(name, address(street, city))` → `Star(name, street, city)`
  - 将多值属性与关键字单独组成一个新的关系

**第二范式（2NF）**：
> 包含在任意一个候选关键字中的属性称为**主属性**，否则称为**非主属性**。如果一个表是 1NF 且所有非主属性**完全依赖**于某一个候选关键字（不存在部分依赖），则称该表是 2NF。

- **注意**：主关键字是**单属性**且满足1NF的表**一定是2NF**；2NF主要针对有**组合关键字**的表
- **不符合2NF**：非主属性只依赖于组合关键字的部分属性（部分依赖）

**课件练习题**：
```
学生(学号, 姓名, 班级, 课号, 课程名, 成绩, 教师, 教师职务)
  候选键：{学号, 课号}
  部分依赖：{学号,课号} →p 课程名（课程名只依赖课号）
           {学号,课号} →p 姓名  （姓名只依赖学号）
  → 不满足2NF

员工(员工码, 姓名, 出生日期, 联系电话, 最后学历, 毕业学校, 培训日期, 培训内容)
  候选键：{员工码, 培训日期}
  部分依赖：{员工码, 培训日期} →p {姓名, 出生日期}（只依赖员工码）
  → 不满足2NF

图书(书号, 书名, 出版日期, 出版社, 书架号, 房间号)
  候选键：书号（单属性）
  无部分依赖：书号 →f 每一个属性
  → 满足2NF
```

**第三范式（3NF）**：
> 如果一个表是 2NF，且表中任意非主属性都**不传递依赖**于某一个候选关键字，则该表是 3NF。

- **传递依赖**：A → B → C（A 是候选键，B 不是候选键，A 通过 B 间接决定 C）

**课件练习题**：
```
学生(学号, 系号, 系主任)
  候选键：学号
  传递依赖：学号→系号，系号→系主任 → 满足2NF但不满足3NF

员工(员工码, 姓名, 部门, 部门经理)
  候选键：员工码
  传递依赖：员工码→部门，部门→部门经理 → 满足2NF但不满足3NF
```

> **重要结论（课件原话）**：关系模式设计如满足第3范式，则**一定能满足第2范式**；反之则不然。（3NF ⊆ 2NF ⊆ 1NF）

**三个范式应用举例（课件非1NF表→1NF→2NF→3NF全流程）**：

原始非1NF表 A（学号、姓名含多值课程信息）
→ 分解为：
- 表B：学号、课程名（学生选课关系，满足1NF但含部分依赖）
- 表C：课程名、学分（课程基本信息）  
- 表D：课程名、开课教师（教师信息）
- 表E：开课教师、教师职称（教师职称，存在传递依赖）
→ 最终分解到3NF得5个关系表

#### 3.4 物理结构设计阶段

**定义**：对已确定的逻辑数据结构，利用 DBMS 所提供的方法、技术，设计出一个高效的、可实现的**物理数据库结构**。

**主要工作**：
1. DBMS 选型
2. 确定数据库的存储结构（文件类型：定长文件、不定长文件；堆文件、散列文件或 B-Tree 文件等）
3. 用 Triggers 设计完整性控制约束
4. 确定数据库的高效访问方式（索引访问、直接访问……）
5. 评估和设置磁盘空间需求

#### 3.5 数据库实施阶段

**主要工作**：
1. **应用程序的开发与调试**：定义数据库，建立实际数据库结构，开发与调试应用程序
2. **数据的加载及数据库试运行**：
   - 可先使用模拟数据（手工生成或随机数发生器生成）
   - 测试数据应尽可能覆盖需求分析阶段所需的各种数据
   - **功能测试**：实际运行应用程序，测试各种功能是否符合应用要求
   - **性能测试**：测量系统性能指标，分析是否符合设计目标

#### 3.6 数据库运行与维护阶段

数据库试运行合格后，投入正式运行。由 **DBA** 负责日常维护工作，维护内容包括：
1. 数据库的**转储和恢复**
2. 数据库的**安全性、完整性控制**
3. 数据库**性能**的监督、分析和改进
4. 数据库的**调整**
5. 数据库的**重组和重构**

| 名词 | 定义（可直接背） |

| **属性** | 实体所具有的某一方面特性 |

| **部分参与联系** | 该端实例可以不参与联系，最小基数为 0 |
| **数据字典（DD）** | 系统中各类数据描述的集合，包括数据项、数据结构、数据流、数据存储和处理过程 |
| **数据流图（DFD）** | 描述数据在系统中流动和处理过程的图形化工具 |
| **1NF（第一范式）** | 关系中的每一分量都是不可分的原子值 |
| **2NF（第二范式）** | 是1NF且所有非主属性完全依赖于某一个候选关键字（无部分依赖） |
| **3NF（第三范式）** | 是2NF且任意非主属性不传递依赖于某一个候选关键字（无传递依赖） |
| **部分依赖** | 非主属性依赖于组合候选键的真子集（而非全部） |

## ④ 简答/论述题预测（第六讲）

### Q1：数据库设计分为哪六个阶段？每个阶段的主要任务和产出物是什么？

**答**（见前文详细对比表，简答版）：
1. **需求分析**：了解数据与处理需求，产出需求规范说明书（含数据字典+DFD+功能模块图）
2. **概念结构设计**：形成独立于 DBMS 的概念模型，产出全局 E-R 图
3. **逻辑结构设计**：将 E-R 图转换为某 DBMS 支持的数据模型并规范化，产出关系模式集
4. **物理结构设计**：选取最适合应用的物理结构（存储结构、索引等），产出物理数据库结构方案
5. **数据库实施**：建库、加载数据、功能/性能测试，产出可运行的数据库系统
6. **运行与维护**：DBA 日常维护（转储恢复、安全、性能、调整、重组重构），维持数据库持续运行

### Q2：E-R 模型的三个基本概念是什么？各有何含义？

**答**：
1. **实体**：客观存在并可相互区分的事物，有类（实体型）和个体（实例）之分
2. **属性**：实体所具有的某一方面特性，分为单一/复合、单值/多值、可空/非空、导出属性等类型
3. **联系**：一个实体的实例和其他实体实例之间所可能发生的联系，按度分为一元/二元/三元，按基数分为1:1/1:m/m:n

### Q3：E-R 图向关系模式转换的基本规则是什么？

**答**（三条）：
1. **实体转换**：E-R 图的每个实体 → 一个关系（表），实体的属性 → 关系的属性，实体的关键字 → 关系的主键
2. **1:m 联系**：将 1 端实体的主键及联系属性加入 m 端实体的关系中（作为外键），无需单独建表
3. **m:n 联系**：必须单独建立一个联系表，该表的主键由两端实体的主键组合构成，同时包含联系本身的属性

### Q4：什么是1NF、2NF、3NF？举例说明。

**答**：
- **1NF**：关系中每一分量都是不可分的原子值。反例：`Star(name, address(street, city))`，address 是复合属性，需拆分为 `Star(name, street, city)`
- **2NF**：是1NF且无非主属性对候选键的部分依赖。反例：`学生(学号, 课号, 姓名, 课程名, 成绩)`，候选键{学号,课号}，但姓名只依赖学号，课程名只依赖课号，违反2NF
- **3NF**：是2NF且无非主属性对候选键的传递依赖。反例：`学生(学号, 系号, 系主任)`，候选键是学号，但学号→系号→系主任，系主任传递依赖于学号，违反3NF

### Q5：需求分析阶段的数据字典包含哪些内容？

**答**（五类）：
1. **数据项**：最小的数据单元，描述含义、类型、长度、取值范围、与其他数据项的关系等
2. **数据结构**：若干数据项的有意义的组合（如学生 = 学号+姓名+性别+出生日期+班号）
3. **数据流**：数据在系统中传输的路径（在DFD中体现）
4. **数据存储**：数据在系统中静态保存的位置
5. **处理过程**：各处理过程（功能）的说明

### Q6：Chen方法中如何区分三种基数的联系？

**答**：
- **1:1 联系**：两端都用箭头直线（由联系指向实体），或两端直线旁均标"1"
- **1:m 联系**：1 端用箭头直线（或标"1"），多端用无箭头直线（或标"m"）
- **m:n 联系**：两端都用无箭头直线（或标"m"和"n"）
- **完全参与联系**：实体与联系间用**双直线**连接
- **部分参与联系**：实体与联系间用**单直线**连接

## ⑤ 易错点提醒（第六讲）

1. **概念模型 ≠ 数据模型**：E-R 模型是概念模型，关系模型是数据模型。概念设计产生 E-R 图，逻辑设计产生关系模式。两者是不同设计阶段的产物。

2. **实体 vs 实例**：考试中容易混淆。"学生"是实体（型），"张三，男，20岁"是一个实例（值）。E-R 图中画的是实体（类），不是具体实例。

3. **联系也有属性**：联系本身可以有属性（如借阅关系有"借阅日期"和"归还日期"），不要认为只有实体才有属性。

4. **一元联系中的角色**：同一实体在一元联系中扮演不同角色时，必须在联系的属性中用不同名字区分（如"父件号"和"子件号"都是零件号，但角色不同）。

5. **完全参与 vs 部分参与**：在 Chen 方法中，双线表示完全参与（最小基数=1），单线表示部分参与（最小基数=0）。考试中容易画错。

6. **1:m 联系的转换**：1:m 联系不需要单独建表，只需将 1 端的主键加入 m 端的关系中作为外键。单独建联系表也可以，但会增加表的数量和连接开销。

7. **m:n 联系的转换**：m:n 联系**必须**单独建联系表，两端的主键合在一起构成联系表的组合主键。这是硬性规则，不可省略。

8. **3NF 包含 2NF**：满足 3NF 的关系模式一定满足 2NF，但反之不成立（满足 2NF 不一定满足 3NF）。3NF 是更严格的约束。

9. **部分依赖 vs 传递依赖**：
   - 部分依赖：非主属性依赖于**组合候选键的真子集**（违反2NF）
   - 传递依赖：A→B→C，B 不是候选键（违反3NF）
   - 单属性候选键的关系如果满足1NF，则自动满足2NF（因为不存在部分依赖），但不一定满足3NF

10. **Database Designer 不是 E-R 图工具**：课件明确指出，Database Designer（SQL Server 中的工具）建立的图称为 Database Diagram，"这种图不是 ER 图，它实际上是数据库逻辑模式的图形化"——这是易错考点。

11. **数据建模抽象的顺序**：理解 → 区分 → 命名 → 表达，不能颠倒。先理解现实世界的业务，再区分各类事物，再命名，最后用 E-R 图表达。

12. **需求分析调查的重点是"数据"和"处理"**，两个都要，缺一不可。

**简答练习**：判断以下关系模式是否满足3NF，若不满足，如何分解？
```
员工(员工码, 姓名, 部门, 部门经理)
候选键：员工码
```

**答**：不满足3NF。
- 函数依赖：员工码→部门，部门→部门经理
- 存在传递依赖：员工码→部门→部门经理
- 分解为两个关系以满足3NF：
  - `员工(员工码, 姓名, 部门)` ← 主键：员工码
  - `部门(部门, 部门经理)` ← 主键：部门
- 分解后消除了传递依赖，两个关系均满足3NF。
