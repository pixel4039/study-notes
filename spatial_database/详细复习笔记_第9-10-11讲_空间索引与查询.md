## 第一部分：空间索引（核心必考）

### 1.1 数据库物理存储基础

#### 存储层次结构

- **FAT（文件分配表）**：操作系统通过 FAT-目录（文件夹）-磁盘块/簇 组织数据
- Oracle 物理存储层次（从大到小）：
  ```
  数据库(database) → 表空间(tablespace) → 操作系统文件(datafile)
  → 表(table) → 段(segment) → 盘区(extent) → 基本数据块(data block)
  ```
- 系统表空间 SYSTEM：由 Oracle 在创建数据库时自动创建，用于数据字典管理
- 用户表空间：由用户创建

#### 记录与磁盘块的映射方式

| 方式 | 特点 | 优缺点 |
|------|------|--------|
| 非跨块存储 | 一条记录不跨越磁盘块 | 浪费部分存储空间；磁盘块间无关联可并行读取 |
| 跨块存储 | 一条记录跨越多个磁盘块，靠指针连接 | 节省存储空间；磁盘块间有关联需串行读取 |

#### 磁盘块分配方法

1. **连续分配**：数据块被分配到连续的磁盘块上（存在扩展困难问题）
2. **链接分配**：数据块中包含指向下一数据块的指针（访问速度慢）
3. **按簇分配**：按簇分配，簇是若干连续的磁盘块，簇之间靠指针连接；簇有时也称片段(Segment)或盘区(extent)
4. **索引分配**：索引块中存放指向实际数据块的指针

### 1.2 数据库索引基础

#### 索引定义（课件原话）

> 索引是定义在存储表(Table)基础之上，有助于无需检查所有记录而快速定位所需记录的一种辅助存储结构，由一系列存储在磁盘上的**索引项(index entries)**组成，每一索引项由两部分构成：
> - **索引字段**：由Table中某些列（通常是一列）中的值串接而成，类似于词典中的词条。
> - **行指针**：指向Table中包含索引字段值的记录在磁盘上的存储位置，类似于词条在词典中出现的页码。

- 存储索引项的文件称为**索引文件**；相对应，存储表又称为**主文件**。
- 索引文件是辅助存储结构，其存在与否不改变主文件的物理存储结构；但其存在可以明显提高访问速度。
- 有索引时，更新操作必须同步更新索引文件和主文件。

#### 索引文件组织方式

| 类型 | 说明 |
|------|------|
| 排序索引文件 (Ordered indices) | 按索引字段值的某一种顺序组织存储 |
| 散列索引文件 (Hash indices) | 依据索引字段值使用散列函数分配散列桶的方式存储 |

#### 稠密索引 vs 稀疏索引

| 类型 | 定义 | 特点 |
|------|------|------|
| 稠密索引 (dense index) | 对主文件中每一个记录都有一个索引项与之对应 | 查找效率高，占用空间大 |
| 稀疏索引 (sparse index / undense index) | 对主文件中部分记录有索引项与之对应 | 占用空间小，需多次I/O |

#### 主索引 vs 辅助索引

| 特性 | 主索引 | 辅助索引 |
|------|--------|--------|
| 定义 | 通常对每一存储块有一个索引项，索引项总数 = 存储块数目 | 定义在主文件的任一或多个非排序字段上 |
| 建立字段 | 主码/排序码 | 其他属性 |
| 密度 | 稀疏索引 | 稠密索引 |
| 数量 | 一个主文件只能有一个主索引 | 一个主文件可以有多个辅助索引 |
| 重组数据 | 可以利用主索引重新组织主文件数据 | 不能改变主文件数据 |
| 指针指向 | 可以仅指向块地址 | 不能仅指向块地址（需指向具体记录） |

- **块锚（block anchor）**：主文件每一存储块的第一条记录，也称锚记录(anchor record)

#### 聚簇索引 vs 非聚簇索引

- **聚簇索引**：索引中邻近的记录在主文件中也是临近存储的。主索引通常是聚簇索引。
- **非聚簇索引**：索引中邻近的记录在主文件中不一定邻近存储。辅助索引通常是非聚簇索引。
- **聚簇字段**：如果主文件的某一排序字段不是主码，则该字段上每个记录取值不唯一，此时该字段被称为聚簇字段。
- **关键区别**：主索引/聚簇索引是能够决定记录存储位置的索引；非聚簇索引只能用于查询，指出已存储记录的位置。

#### B+树索引

- **定义**：一种以树型数据结构组织索引项的多级索引
- **特点**：
  - 所有记录节点都按键值大小顺序存放在**同一层的叶子节点**上，由各叶子节点指针连接
  - 索引字段值**重复出现**于叶结点和非叶结点
  - **指向主文件的指针仅出现于叶结点**
  - 所有叶结点即可覆盖所有键值的索引（最低一层索引）
  - 索引字段值在叶结点中按顺序排列
  - 能够自动保持与主文件大小相适应的树的层次

| | B树 | B+树 |
|---|-----|------|
| 数据记录存储位置 | 内节点和叶节点均可 | 仅叶节点 |
| 叶节点链接 | 无 | 有（链表连接） |
| 范围查询 | 较麻烦 | 很高效 |

#### SQL中索引的创建与维护

```sql
-- 创建索引
CREATE [UNIQUE] INDEX indexname
ON tablename (colname [ASC | DESC] {, colname [ASC | DESC] ...});

-- 示例：在student表中创建基于Sname的索引
CREATE INDEX idxSname ON student(sname);

-- 示例：在student表中创建基于Sname和Sclass的复合索引
CREATE INDEX idxSnamcl ON student(sname, sclass);

-- 撤消索引
DROP INDEX indexname;
```

**注意**：
- 当定义Table并定义了主键，系统将自动创建主索引
- 索引创建后，DBMS自动维护所有索引与Table的一致性
- 当Table被删除(DROP TABLE)后，该Table上的所有索引自动被撤消

**建立索引需考虑**：访问时间、插入时间、删除时间与空间负载。对经常出现在检索条件、连接条件、分组计算条件（SELECT...WHERE...GROUP BY）中的属性可建立索引。

### 1.3 空间索引概念与分类

#### 空间索引定义（课件原话）

> 空间索引就是指依据空间对象的位置和形状或空间对象之间的某种空间关系，按一定的顺序排列的一种数据结构，其中包含空间对象的概要信息，如对象的**标识**、**外接矩形**及指向**空间对象实体的指针**。

- 空间索引也称为**空间访问方法（Spatial Access Method，SAM）**
- 空间数据库索引技术是对存储在介质上的数据位置信息的描述，是为了**快速访问**一条特定查询所请求的数据，而无需遍历整个数据库。

#### 为什么需要空间索引

- 空间数据量庞大，全表扫描代价极高
- 空间查询（相交、包含、邻近等）无法直接用B+树处理（空间对象多维度，不能简单排序）
- 需要高效的**近似过滤**机制，先快速排除不可能命中的对象

#### MBR（最小外接矩形）

- **定义**：Minimum Bounding Rectangle，是能够完全包围一个空间对象的最小坐标对齐矩形
- **作用**：作为空间对象的近似描述，用于空间索引的过滤步骤
- **表示**：通常以 (x1, y1, x2, y2) 形式存储（左下角和右上角坐标）
- **特性**：MBR 的运算（相交、包含等判断）远比精确几何运算快

#### 过滤—求精两步查询框架（重要）

```
查询请求
    ↓
【过滤步（Filter Step）】
  访问空间索引（存储MBR近似描述）
  对MBR做相关操作，排除不可能满足查询条件的对象
  得到"候选集"（可能满足查询的对象）
    ↓
【求精步（Refinement Step）】
  对候选集对象的实际数据进行精确测试
  采用复杂的计算几何算法
  测试候选对象是否确实满足查询条件
    ↓
最终结果输出
```

**Oracle Spatial的三步扩展**：在过滤步和求精步之间插入中间步骤，判断对象的MBR与查询窗口内部表示(Interior(Q))的关系，若MBR(G)完全包含在Interior(Q)内则直接作为结果输出，大大减少计算量。Oracle Spatial使用**双层查询模型**（一级过滤 + 二级过滤）。

#### 常见空间索引类型（课件列举）

1. 对象范围索引
2. **格网索引（Grid Index）**
3. **四叉树索引（Quadtree）**
4. BSP树索引
5. **R树和R+树索引**（课件重点讲解）

### 1.4 R树索引（最重要）

#### R树起源

- R树算法由美国加州大学的 **Antomm Guttman 教授在1984年**提出
- 是一种空间数据库**动态索引**算法，现已成为空间数据库索引中应用最广泛的算法之一
- R树是**处理空间数据的B+树的改进**，像B+树一样，是一个**高度平衡的数据结构**
- R树较好地解决了许多传统算法未解决的空间数据动态索引问题

#### R树的核心特性

- **每个节点存储一个MBR**，该MBR是其所有子节点MBR的最小外接矩形
- **MBR空间范围可以相互重叠（相交）**，但所有对象仅属于一个MBR（每个对象只存储在一个叶节点中）
- 树结构：高度平衡，叶节点位于同一层次

#### R树节点结构

- **叶节点**存储格式：`ID | x1 | y1 | x2 | y2 | 指向实际对象的指针`
- **非叶节点**存储格式：`ID | x1 | y1 | x2 | y2 | 指向子节点的指针`

#### R树查询过程

1. 从根节点开始，检查查询窗口与各子节点MBR是否相交
2. 对相交的子节点，递归进行同样的检查
3. 到达叶节点时，对该叶节点中的实际对象进行精确的几何测试
4. 输出满足条件的结果

#### R树空间连接（Spatial Join）

- R树的匹配算法：从匹配两个R树的**根节点**开始，寻找是否有重叠的情况，然后**深度优先递归搜索**匹配的子节点，当达到叶节点时，得出匹配结果并输出
- 要求每个参与空间连接的对象集合均预先建立了R树索引

#### R树的优缺点

| | 内容 |
|---|------|
| 优点 | 动态索引，支持插入/删除；高度平衡；支持范围查询/点查询/空间连接；PostGIS等主流系统广泛采用 |
| 缺点 | 节点MBR可能重叠，导致查询时可能访问多条路径；MBR近似质量有限，候选集中可能包含大量"假命中" |

### 1.5 R+树

#### R+树的核心差异

- **R树**：MBR空间范围可以相交，但所有对象仅属于一个MBR
- **R+树**：**所有MBR不相交**，但**对象可以属于多个MBR**（即同一对象可能被存储在多个叶节点中）

#### R+树的节点结构

- 叶节点：`ID1 | x1 | y1 | x2 | y2 | ID2`（对象跨越多个MBR时，同一对象ID出现在多个节点）

#### R+树 vs R树对比

| 特性 | R树 | R+树 |
|------|-----|------|
| 节点MBR是否可重叠 | 可以重叠 | 不重叠 |
| 一个对象属于几个节点 | 仅属于一个叶节点 | 可属于多个叶节点（冗余存储） |
| 查询路径数 | 可能多条（因为MBR重叠） | 通常更少（MBR不重叠，路径唯一） |
| 空间利用率 | 较高 | 因冗余存储，利用率较低 |
| 更新代价 | 较小 | 较大（更新一个对象需更新多处） |

#### R*树（大纲提及，课件未详述）

课件未详述R*树具体算法，仅在大纲中提及。基本概念：R*树是R树的改进版，优化了节点分裂策略（最小化节点MBR的面积、周长和重叠），提高了查询性能。

### 1.6 其他空间索引

**课件说明**：课件（第九讲）对格网索引、四叉树、KD树、BSP树只列出名称，未进行详细原理讲解。以下内容为补充说明，不作为主要考点，按课件实际内容为准。

#### 格网索引（Grid Index）

- 将空间区域划分为均匀的格网单元
- 每个格网单元记录与其相交的空间对象
- 优点：结构简单，查询快速
- 缺点：空间分布不均匀时效率低；多级格网解决此问题

#### 四叉树（Quadtree）

- 递归地将二维空间划分为四个象限
- 分类：点四叉树、区域四叉树、MX四叉树、PR四叉树
- 优点：自适应空间分布；节省内存
- 缺点：树深度不平衡；不适合动态更新

#### KD树（K-D Tree）

- 是一种对k维空间中的点进行分类的数据结构
- 每次选取一个维度进行二分，轮流切换维度
- 适合点数据的最近邻查询
- 缺点：不适合存储扩展空间对象（如多边形）

### 1.7 PostGIS空间索引实践

#### PostgreSQL提供的索引类型

| 索引类型 | 适用场景 |
|--------|--------|
| B-tree | 能按顺序存储的数据，适用于 <、<=、=、>=、> 操作符 |
| Hash | 只处理简单的等于(=)比较 |
| GiST | 一种架构（通用搜索树），不同的索引策略实现不同功能；**PostGIS空间数据用此类型** |
| GIN | 反转索引，处理包含多个键的值（如数组） |

#### PostGIS的GiST空间索引

- PostGIS不通过B-tree索引加速空间查询，而是通过**GiST索引**
- GiST索引通过**R树结构**来实现对空间类型数据的索引查询，结构类似于B-tree
- GiST索引不是对几何要素本身建索引，而是对几何要素的**边界框（Bounding Box，即MBR）**建索引

#### 创建GiST空间索引（课件例题）

```sql
-- 创建空间索引
CREATE INDEX idx_t_pos_1 ON t_pos USING GIST(pos);

-- 使用EXPLAIN分析查询计划（未建索引时：全表扫描）
-- 执行时间约0.612ms
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT * FROM t_pos WHERE pos && st_astext('POINT(73.689759 3.880185)');
-- 结果：Seq Scan on t_pos ... Filter: ... Rows Removed by Filter: 999

-- 建立索引后：使用索引扫描
-- 执行时间约0.074ms（提升约8倍）
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT * FROM t_pos WHERE pos && st_astext('POINT(73.689759 3.880185)');
-- 结果：Index Scan using idx_t_pos_1 on t_pos ...
```

- **&&操作符**：检测两个几何对象的边界框是否重叠（不是精确几何相交）
- 执行时间对比：无索引 0.612ms vs 有索引 0.074ms（数据量1000条时提升明显）

#### 栅格数据的空间索引

```sql
-- 栅格数据使用凸包(ConvexHull)建立GiST索引
CREATE INDEX myrasters_rast_st_convexhull_idx
ON myrasters USING gist(ST_ConvexHull(rast));
-- 注意：大多数栅格操作符基于栅格的凸包
```

### 1.8 空间索引方法对比表

| 索引方法 | 结构 | MBR重叠 | 对象冗余 | 适用数据 | 优点 | 缺点 |
|--------|------|---------|---------|---------|------|------|
| R树 | 高度平衡树 | 允许 | 无（一个对象一个节点） | 各类空间对象（点、线、面） | 动态，支持插入删除；应用最广 | MBR重叠导致多路径查询 |
| R+树 | 高度平衡树 | 不允许 | 有（对象可属于多个节点） | 各类空间对象 | 查询路径确定，减少无效访问 | 冗余存储，更新代价高 |
| 格网 | 均匀格网 | N/A | 有 | 均匀分布的点/面 | 结构简单 | 分布不均时效率低 |
| 四叉树 | 递归四分树 | N/A | 有 | 二维点/面 | 自适应空间分布 | 不平衡，不适合动态更新 |
| KD树 | 二叉树 | N/A | 无 | 点数据 | 最近邻查询效率高 | 不适合扩展空间对象 |
| B+树（普通） | 平衡树 | N/A | 无 | 一维有序数据 | 范围查询高效 | 无法直接处理多维空间数据 |

## 第二部分：空间查询（一）

### 2.1 空间查询定义与分类

#### 空间查询定义（课件原话）

> 空间查询是**从空间数据库中查找出满足某一条件的空间目标的过程**。

- 在GIS中，空间查询无处不在，是其他操作（增加、删除、更新等）的基础
- 空间查询性能是GIS应用系统成功的关键

#### 空间查询方式（课件图示，文字概括）

- 属性查询：通过属性条件（如名称、人口、GDP等）查找空间对象
- 空间关系查询：通过空间拓扑关系（相交、包含、相邻等）查找空间对象
- 组合查询：同时使用属性条件和空间条件

---

### 2.2 标准SQL的局限

#### 标准SQL不足（课件原话）

1. **数据类型太简单**：只有整型、日期型、字符串型等，不能支持点、线、面等复杂数据类型（没有相应的操作原语）
2. **数据类型固定**：所有数据类型都是固定的，无法增加新的数据类型

#### 解决方法（三种）

| 方法 | 说明 | 缺点 |
|------|------|------|
| BLOB字段存储 | 利用BLOB类型字段存储空间信息，依赖于宿主语言的应用程序代码 | 不能利用数据库服务（查询语言、索引等） |
| 混合系统 | 空间数据以文件形式保存在数据库外部（如MapInfo） | 无法利用数据库的并发控制、索引支持等服务 |
| OR-DBMS自定义ADT | 利用对象关系数据库管理系统自定义抽象数据类型，如自定义polygon数据类型和adjacent函数 | 实现复杂 |

### 2.3 扩展SQL查询语言

#### 三种扩展类型

**1. 查询谓词的扩展**（在WHERE子句中增加空间操作算子）

```sql
-- Mapinfo示例：美国"I 10"号高速公路经过哪几个州？
SELECT * FROM States
WHERE States.obj CONTAINS Us_Hiway.obj AND
(States.obj INTERSECTS (SELECT obj FROM Us_Hiway
WHERE Us_Hiway.highway = "I 10"))

-- 三峡地区长江流域人口大于50万的县或市
SELECT *
FROM 县或市
WHERE 县或市·人口 > 50万
AND CROSS(河流·名称 = "长江")

-- 河南省示例1：选择河南省所有城市和人口
SELECT 城市名, 人口 FROM 城市
WHERE CENTER(城市地图) INSIDE 河南;

-- 河南省示例2：流经河南省的所有河流名称和河南境内长度
SELECT 河流名, LENGTH(INTERSECTS(ROUTE(河流流域图), 河南))
FROM 河流
WHERE ROUTE(河流流域图) INTERSECTS 河南;
```

**2. 面向对象的扩展**（OGIS协会提出的空间扩展标准）

- OGIS（Open GIS）由主要软件供应商组成的联盟，制定GIS互操作相关行业标准
- 把二维地理空间ADT（抽象数据类型）整合到SQL之中
- 包括指定拓扑的操作和空间分析操作

**3. 模糊扩展**（支持模糊查询条件）

```sql
-- 示例：查询面积较小、噪声小、住宅低价偏低的宗地号

WHERE (面积='较小' AND 噪声='小' AND 住宅低价='偏低')
```

### 2.4.1 OGIS课件例题（World数据库，9题全部）

**数据库模式**：

```sql
City (Name: varchar(35), Country: varchar(35), Pop: Integer,
      Capital: char(1), Shape: Pointid)

Country (Name: varchar(35), Cont: varchar(35), Pop: Integer,
         GDP: Numeric, Life-Exp: Numeric, Shape: Polygonid)

River (Name: varchar(35), Origin-Country: varchar(35),
       Length: Numeric, Shape: Lineid)
```

**部分数据**：

| Country | Cont | Pop(m) | GDP(b) | Life-Exp | Shape |
|---------|------|--------|--------|---------|-------|
| Canada | NAM | 30.1 | 658.0 | 77.08 | Polygonid-1 |
| Mexico | NAM | 107.5 | 694.3 | 69.36 | Polygonid-2 |
| USA | NAM | 270.0 | 8003.0 | 75.75 | Polygonid-5 |

---

**查询1：列出Country表中所有与美国相邻的国家**

```sql
SELECT C1.Name AS "Neighbors of USA"
FROM Country C1, Country C2
WHERE Touch(C1.Shape, C2.Shape) = 1
AND C2.Name = "USA"
```

> Touch：检测两个几何对象是否彼此相邻而又不交叠，是OGIS中定义的8个拓扑谓词之一。

---

**查询2：找出河流流经的国家**

```sql
SELECT R.Name, C.Name
FROM River R, Country C
WHERE Cross(R.Shape, C.Shape)
```

> Cross：拓扑谓词，常用于线对象与多边形对象的穿过判断。

---

**查询3：找出距离每条河流最近的城市**

```sql
SELECT C1.Name, R1.Name
FROM City C1, River R1
WHERE Distance(C1.Shape, R1.Shape) <
    ALL (SELECT Distance(C2.Shape, R1.Shape)
         FROM City C2
         WHERE C1.Name <> C2.Name)
```

> Distance：二元运算，返回两个几何对象之间最短距离的实数值。

---

**查询4：圣劳伦斯河方圆300公里内能获供水的城市**

```sql
SELECT Ci.Name
FROM City Ci, River R
WHERE Overlap(Ci.Shape, Buffer(R.Shape, 300)) = 1
AND R.Name = "St.Lawrence"
```

> Buffer：返回围绕几何对象给定距离范围内的多边形区域。

---

**查询5：列出每个国家的名字、人口和国土面积**

```sql
SELECT C.Name, C.Pop, Area(C.Shape) AS "Area"
FROM Country C
```

> Area：面积函数，只适用于Polygon和MultiPolygon类型。

---

**查询6：求出河流在流经各国境内的长度**

```sql
SELECT R.Name, C.Name, Length(Intersection(R.Shape, C.Shape)) AS "Length"
FROM River R, Country C
WHERE Cross(R.Shape, C.Shape) = 1
```

> Intersection()：函数计算两个几何对象的交部分（返回几何对象）。注意与Intersects（拓扑谓词）的区别。

---

**查询7：列出每个国家的GDP及其首都到赤道的距离**

```sql
SELECT Co.GDP, Distance(Point(0, Ci.Shape.x), Ci.Shape) AS "Distance"
FROM Country Co, City Ci
WHERE Co.Name = Ci.Country AND Ci.Capital = "Y"
```

---

**查询8：按邻国数目多少列出所有国家**

```sql
SELECT Co.Name, Count(CO1.Name)
FROM Country Co, Country Co1
WHERE Touch(Co.Shape, Co1.Shape)
GROUP BY Co.Name
ORDER BY Count(Co1.Name)
```

---

**查询9：哪个国家的邻国最多**

```sql
CREATE VIEW Neighbor AS
SELECT Co.Name, Count(Co1.Name) AS num_neighbors
FROM Country Co, Country Co1
WHERE Touch(Co.Shape, Co1.Shape)
GROUP BY Co.Name;

SELECT Co.Name, num_neighbors
FROM Neighbor
WHERE num_neighbors = (SELECT Max(num_neighbors) FROM Neighbor)
```

### 2.5 对象关系SQL（SQL3 / OR-DBMS）

#### 背景

- 针对OR-DBMS（对象关系数据库管理系统）提出的SQL标准**SQL3/SQL99**
- Oracle 9i 是Oracle公司开发的OR-DBMS，OR-DBMS实现了SQL3标准的一部分，ADT在这种系统中称作"**对象类型**"（Object Type）

#### SQL3两大扩展

**1. ADT（抽象数据类型）**

```sql
-- 定义点ADT
CREATE TYPE Point (
    x NUMBER,
    y NUMBER,
    FUNCTION Distance(:u Point, :v Point) RETURNS NUMBER
)
-- u和v前的冒号表示局部变量
```

**2. 行类型（Row Type）**

```sql
-- 定义行类型
CREATE ROW TYPE Point (x NUMBER, y NUMBER);

-- 创建以该行类型为模板的表
CREATE TABLE PointTable OF TYPE Point;
```

注意：用ADT作为列类型能很自然地与OR-DBMS保持一致。

#### Oracle中创建空间对象

```sql
-- 建立点对象类型
CREATE TYPE Point AS OBJECT (
    x NUMBER,
    y NUMBER,
    MEMBER FUNCTION Distance(P2 IN Point) RETURN NUMBER
    PRAGMA RESTRICT_REFERENCES(Distance, WNDS)
);

-- City表（使用Point类型）
CREATE TABLE City (
    Name varchar(30),
    Country varchar(30),
    Pop Number,
    Shape Point
);

-- 创建中间线类型
CREATE TYPE LineType AS VARRAY(500) OF Point;

-- LineString类型
CREATE TYPE LineString AS OBJECT (
    Num_of_Points INT,
    Geometry LineType,
    MEMBER FUNCTION Length(SELF IN) RETURN NUMBER,
    PRAGMA RESTRICT_REFERENCES(LENGTH, WNDS)
);

-- 插入河流数据
INSERT INTO RIVER ("MISSISSIPPI", "USA", 6000,
    LineString(3, LineType(Point(1,1), Point(1,2), Point(2,3))));

-- 多边形类型
CREATE TYPE PolyType AS VARRAY(1000) OF Point;
CREATE TYPE Polygon AS OBJECT (
    Num_of_Points INT,
    Geometry PolyType,
    MEMBER FUNCTION Area(SELF IN) RETURN NUMBER,
    PRAGMA RESTRICT_REFERENCES(Area, WNDS)
);

-- 国家表（使用Polygon类型）
CREATE TABLE Country (
    Name Varchar(30), Cont Varchar(30), Pop int,
    GDP number, Life-Exp number, Shape Polygon
);
INSERT INTO Country ("Mexico", "NAM", 107.5, 694.3, 69.36,
    Polygon(23, Polytype(Point(1,1), ..., Point(1,1))));
```

#### OR-DBMS查询示例

```sql
-- 找出City表所有城市两两之间的距离（使用ADT方法）
SELECT C1.Name, C1.Distance(C2.Shape) AS "Distance"
FROM City C1, City C2
WHERE C1.Name <> C2.Name;

-- 列出与美国接壤的国家的名字、人口和国土面积
SELECT C2.Name, C2.Pop, C2.Area() AS "Area"   -- ADT函数
FROM Country C1, Country C2
WHERE C1.Name = "USA"
AND C1.Touch(C2.Shape) = 1;                    -- 拓扑谓词
```

### 3.3 空间查询优化实战

#### 问题一：SQL不走空间索引（重要）

**原因分析**：WHERE后对geom字段使用表达式函数会导致索引失效。

```sql
-- 错误写法：对geom字段使用ST_Buffer，导致全表扫描
EXPLAIN SELECT a.* FROM poi a
WHERE ST_Intersects(ST_Buffer(a.geom, 0.01),
      ST_SetSrid(ST_MakePoint(118.3749, 32.0987), 4326));
-- 结果：Parallel Seq Scan（顺序扫描）

-- 正确写法：Buffer应用于查询点（常量），而非geom字段
EXPLAIN SELECT a.* FROM poi a
WHERE ST_Intersects(a.geom,
      ST_Buffer(ST_SetSrid(ST_MakePoint(118.3749, 32.0987), 4326), 0.01));
-- 结果：Index Scan using poi_geom_idx（索引扫描）
```

**ST_Distance vs ST_DWithin**：

```sql
-- 错误写法：ST_Distance在WHERE后对geom做计算，不走索引
SELECT a.* FROM poi a
WHERE ST_Distance(a.geom, ST_SetSrid(ST_MakePoint(118.3749, 32.0987), 4326)) < 0.01;
-- 结果：Parallel Seq Scan（全表扫描，耗时307ms~563ms）

-- 正确写法：使用ST_DWithin，走空间索引
EXPLAIN(ANALYSE) SELECT a.* FROM poi a
WHERE ST_DWithin(a.geom,
      ST_SetSrid(ST_MakePoint(118.3749, 32.0987), 4326), 0.01);
-- 结果：Index Scan（耗时约0.09ms）
```

**特殊类型转换导致索引失效**：

```sql
-- 类型转换(geom::geography)导致索引失效
EXPLAIN(ANALYSE) SELECT a.* FROM poi a
WHERE ST_DWithin(a.geom::geography,
      ST_SetSrid(ST_MakePoint(118.3749, 32.0987), 4326)::geography, 1000);
-- 结果：Seq Scan（顺序扫描）

-- 解决方案：建立表达式索引
CREATE INDEX poi_geogra_idx ON poi USING gist((geom::geography));
-- 之后相同查询：可以使用Bitmap Index Scan
```

**核心结论（课件总结）**：
1. PostGIS解决同一问题的写法有多种，需要仔细审核与对比
2. **不在WHERE后对geom字段进行函数计算或类型转换操作**
3. 特殊情况可建立表达式索引

#### 问题二：ST_Disjoint不走索引（重要）

**场景**：居民点house表25000条，交通点station表6000条，统计所有交通点200米以外的居民点数量。

```sql
-- 错误写法：使用ST_Disjoint查询（耗时36秒）
EXPLAIN(ANALYSE) SELECT count(a.*) FROM house a
WHERE gid NOT IN (
    SELECT DISTINCT(a.gid) FROM house a,
    (SELECT st_union(st_buffer(geom::geography, 200)::geometry) geom FROM station) b
    WHERE ST_Intersects(a.geom, b.geom)
);
-- 索引耗时32s/总耗时36s，索引效果极差

-- 正确思路：用ST_Intersects取反替代ST_Disjoint
SELECT count(a.*) FROM house a
WHERE gid NOT IN (
    SELECT DISTINCT(a.gid) FROM house a,
    (SELECT st_buffer(geom::geography, 200)::geometry geom FROM station) b
    WHERE ST_Intersects(a.geom, b.geom)
);  -- 耗时0.73s
```

**核心结论（课件总结）**：
1. **ST_Disjoint（相离）函数不符合R树索引的查询机制，绝对不会走索引**
2. ST_Disjoint仅适用于判定空间关系的结果，不可用于WHERE后根据空间关系查询的条件
3. **ST_Disjoint应采用"ST_Intersects（相交）取反"方案优化**

**ST_Union在子查询中使用的问题**：

```sql
-- 错误写法：ST_Union在子查询where条件里，导致索引扫描范围过大（36s）
-- 图形太复杂（合并后的大多边形），bbox大，索引效率差

-- 正确写法：不合并缓冲区，直接逐个buffer与house相交（0.73s）
SELECT count(a.*) FROM house a
WHERE gid NOT IN (
    SELECT DISTINCT(a.gid) FROM house a,
    (SELECT st_buffer(geom::geography, 200)::geometry geom FROM station) b
    WHERE ST_Intersects(a.geom, b.geom)
);
```

**结论**：Union等图形处理函数应该只在输出结果中使用，不能在查询条件里使用。

#### 问题三：空间索引效率低

**原因**：geom对象本身太复杂（如MultiPolygon），bbox大，会导致索引扫描范围放大，效率变差。

**解决方案**：

```sql
-- 查询几何对象数量
ST_NumGeometries(geom)
-- 提取第N个子几何对象
ST_GeometryN(geom, N)
-- 将复杂多边形细分为小多边形集合（ST_SubDivide）
ST_SubDivide(geom, n)
```

### 3.5 空间查询执行过程

#### 一般SQL查询执行的三个阶段

**一、连接**
1. 客户端发起Query请求，连接管理模块接收
2. 转发至连接进/线程模块
3. 调用用户模块进行授权检查
4. 从线程连接池取出空闲连接线程与客户端对接

**二、处理（语句解析）**
1. 查询高速缓存（library cache）
2. 语句合法性检查（data dict cache）
3. 语言含义检查（data dict cache）
4. 获得对象解析锁（control structure）
5. 数据访问权限核对（data dict cache）
6. 确定最佳执行计划

**三、提取结果**
1. Query完成后，将结果集返回给连接进/线程模块
2. 返回状态标识（成功或失败）
3. 进行清理工作

#### 数据库查询处理过程（完整流程）

```
SQL查询语句
    ↓
【查询解析器】→ 词法分析 + 语法分析 → 语法分析树
    ↓
【查询预处理器】→ 语义检查（有效性检查）→ 关系代数查询树
    ↓
【查询优化器】→ 代数优化 + 物理优化 → 关系代数优化查询树
    ↓
【查询代码生成器】→ 生成查询计划的执行代码
    ↓
【执行引擎】→ 执行查询计划
    ↓
执行结果
```

**示例**（课件例题：查询选修了c02课程的学生姓名）：

```sql
-- Q1: 连接查询
SELECT SN FROM S, SC WHERE S.Sno = SC.Sno AND SC.Cno = 'c02';
-- 关系代数：π_SN(σ_{S.Sno=SC.Sno ∧ SC.Cno='c02'}(S × SC))

-- Q2: 嵌套查询
SELECT SN FROM S WHERE Sno IN (SELECT Sno FROM SC WHERE Cno = 'c02');
-- 关系代数：π_SN(S ⋈ π_Sno(σ_{SC.Cno='c02'}(SC)))
```

#### SARG（搜索参数）

- **定义**：用于限制搜索的一个操作，通常是指特定匹配、范围内匹配或多个AND条件连接。
- 形式：`列名 操作符 <常数或变量>` 或 `<常数或变量> 操作符 列名`
- 示例：`Name='张三'`、`价格>5000`、`Name='张三' AND 价格>5000`
- 可以使用索引的条件称为**可优化的（SARG形式）**

#### SQL Server查询扫描方式（从慢到快）

| 方式 | 速度 | 说明 |
|------|------|------|
| Table Scan（表扫描） | 最慢 | 对表记录逐行检查 |
| Clustered Index Scan（聚集索引扫描） | 较慢 | 按聚集索引对记录逐行检查 |
| Index Scan（索引扫描） | 普通 | 根据索引滤出部分数据再逐行检查 |
| Index Seek（索引查找） | 较快 | 根据索引定位记录所在位置再取出记录 |
| Clustered Index Seek（聚集索引查找） | 最快 | 直接根据聚集索引获取记录 |

#### 空间查询执行过程（两步 + 扩展）

**基本两步**（课件原话）：

> 空间查询处理的过程一般分为两个步骤——**过滤（Filter）**和**求精（Refinement）**。
>
> 当执行空间查询时，查询处理器首先访问空间索引。空间索引中存储的是空间对象的近似描述，对这些近似描述做相关的操作，排除掉不可能满足查询条件的对象，余下的对象可能满足查询条件，这些对象构成了**候选集**。这就是**过滤步骤**。
>
> 将候选集对象的实际数据输入**求精步**进行下一步的处理。在求精步，需要测试候选对象是否确实满足查询条件（由空间谓词描述），采用复杂的计算几何算法，这个测试由不同的阶段组成。

**Oracle Spatial扩展（中间步骤）**：

> 在过滤步和求精步之间插入一个中间步骤，对对象的MBR与查询窗口的内部表示（Interior(Q)）的关系进行判断：
> - 若MBR(G)完全包含在Interior(Q)内 → 直接作为结果输出（无需精确计算）
> - 否则 → 继续进行精确计算，但计算量已大大减少

**Oracle Spatial双层查询模型**：一级过滤（Primary Filter）+ 二级过滤（Secondary Filter）= 精确结果集

### 3.6 关系数据库查询优化

#### 查询优化的必要性

- 数据的物理独立性使查询效率不如非关系数据库
- 查询优化极大影响RDBMS性能
- 查询优化不只是DBMS的任务，用户的查询计划质量也直接影响结果

#### DBMS进行查询优化的好处

- 优化器可从数据字典获取统计信息，用户程序难以获得
- 物理统计信息改变时，系统可自动重新优化查询
- 优化器可考虑数百种执行计划，程序员只能考虑有限几种
- 优化器包含复杂的优化技术

#### 查询优化步骤（四步）

1. 将查询转换成某种内部表示（语法树）
2. 根据等价变换规则把语法树转换成标准形式（代数优化）
3. 选择低层的操作算法（物理优化）
4. 生成查询计划（选择代价最小的方案）

#### 查询调优四个层级

| 层级 | 说明 |
|------|------|
| 语法级 | 查询语言层的优化，基于语法进行优化 |
| 代数级 | 使用形式逻辑优化，运用关系代数原理 |
| 语义级 | 根据完整性约束，对查询语句进行语义理解，推知可优化操作 |
| 物理级 | 物理优化技术，基于代价估算模型，选择代价最小的路径 |

#### 优化思路（四条）

1. 将过程性查询转换为描述性查询（视图重写技术）
2. 将复杂查询（嵌套子查询、外连接、嵌套连接）尽可能转换为多表连接查询
3. 将效率低的谓词转换为等价的效率高的谓词（等价谓词重写）
4. 利用等式和不等式，简化WHERE、HAVING和ON条件

#### 常用优化技术（课件总结）

```sql
-- 1. 使用索引优化查询（首先在WHERE/ORDER BY/GROUP BY列上建索引）
-- 2. 使用EXPLAIN分析执行计划
EXPLAIN SELECT * FROM news;

-- 3. 优化SQL语句
-- 避免：SELECT * FROM table
-- 推荐：SELECT 具体列名 FROM table
-- 避免：WHERE子句中对索引字段进行计算
-- 避免：WHERE子句中使用IN、NOT IN、OR或HAVING
-- 避免：DISTINCT、UNION、MINUS、INTERSECT、ORDER BY（耗费排序资源）

-- 4. 优化数据库对象
-- 3.1 优化表的数据类型（使用能正确表示数据的最短类型）
SELECT * FROM 表名 PROCEDURE ANALYSE();

-- 3.2 表拆分
-- 垂直拆分：把常用列和不常用列分开到不同表
-- 水平拆分：按数据行分到不同表

-- 3.3 使用中间表提高查询速度
-- 4. 硬件优化：多核/高频CPU；大内存；RAID 0+1磁盘阵列；合适磁盘调度算法
```

### 3.7 空间查询优化理论

#### 空间查询优化四个阶段（课件原话）

1. 将查询转换为某种内部表示（通常是语法树形式）
2. 进行某些提高效率的表达式变换（如先进行选择再连接，先进行投影再连接，变换规则通常有正确性保证）
3. 谓词代价计算
4. 选择代价最小的执行计划

#### 空间查询优化三个方向

| 方向 | 说明 |
|------|------|
| ① 空间索引 | 从数据存取角度优化，有效读写海量数据；空间数据量庞大是解决数据库瓶颈的重要方法 |
| ② 查询处理算法 | 从提高具体查询执行效率角度优化；目标是用尽可能少的复杂几何计算得到正确结果 |
| ③ 代价模型 | 从指导查询过程角度优化（采用哪种查询计划）；在处理含多个空间查询条件时作用重要 |

#### 基于R树的空间连接优化

> R树索引结构简单，能对空间对象进行有效处理，故可提出基于R树的算法来优化空间连接这一复杂耗时的空间查询操作。这些算法要求每个参与空间连接的对象集合均预先建立了R树索引。
>
> R树的匹配算法是直接的：从匹配两个R树的**根节点**开始，寻找是否有重叠的情况，然后**深度优先递归搜索**匹配的子节点，当达到叶节点时，得出匹配结果并输出。

#### 近似质量与过滤效果

> 在过滤步骤中的对象近似应当是简单的，这样才能实现快速的过滤。另一方面，对空间对象的近似程度越高，滤除未命中对象的效果越好。
>
> 在过滤步骤中通常采用**MBR近似**，其近似效果有限。
>
> 与MBR相比，其他近似描述的SAM需要访问更多的数据页，但由于近似质量高，可以减少未命中对象，从而减少后继处理中的几何计算开销。**近似使用的参数越多，近似质量越高**。

## 名词解释清单

1. **空间索引（Spatial Index）**：依据空间对象的位置和形状或空间对象之间的某种空间关系，按一定的顺序排列的一种数据结构，其中包含空间对象的概要信息，如对象的标识、外包络矩形及指向空间对象实体的指针。也称为空间访问方法（SAM）。

2. **MBR（Minimum Bounding Rectangle，最小外接矩形）**：能够完全包围一个空间对象的最小坐标轴对齐矩形。用 (x1,y1,x2,y2) 四个坐标值表示，是空间索引中最常用的近似表示方式。

3. **R树（R-Tree）**：一种用于处理空间数据的高度平衡树状索引结构，是B+树在多维空间数据上的改进，由Guttman于1984年提出。每个节点存储一个MBR，MBR之间可以重叠，但每个空间对象只属于一个叶节点。

4. **R+树（R+ Tree）**：R树的变体，要求所有节点的MBR不相交，但允许同一空间对象被存储在多个叶节点（冗余存储），从而避免查询时的多路径问题。

5. **GiST（Generalized Search Tree，通用搜索树）**：PostgreSQL中的一种索引架构，可在此架构上实现不同的索引策略。PostGIS使用GiST索引来支持空间数据查询，底层通过R树结构实现。

6. **稠密索引（Dense Index）**：对主文件中每一个记录都有一个索引项与之对应的索引。

7. **稀疏索引（Sparse Index）**：对主文件中部分记录有索引项与之对应的索引，通常每一存储块只有一个索引项。主索引是稀疏索引。

8. **聚簇索引（Clustered Index）**：索引中邻近的记录在主文件中也是临近存储的索引，是能够决定记录存储位置的索引。

9. **过滤步（Filter Step）**：空间查询的第一步，利用空间索引对MBR进行快速相交/包含判断，快速排除不可能命中的对象，得到"候选集"。

10. **求精步（Refinement Step）**：空间查询的第二步，对候选集中的每个对象进行精确的几何计算，确认是否真正满足查询条件。

11. **OGIS（Open GIS）**：由主要软件供应商组成的联盟，制定GIS互操作相关行业标准，提出了将二维地理空间ADT整合到SQL中的规范，包括拓扑操作和空间分析操作。

12. **ADT（Abstract Data Type，抽象数据类型）**：用于定义和封装数据结构及其操作的概念，SQL3中可用CREATE TYPE语句定义，包含属性和成员函数。

13. **OR-DBMS（Object-Relational DBMS，对象关系数据库管理系统）**：扩展了关系数据库的面向对象特性，支持用户自定义数据类型（ADT），如Oracle 9i、PostgreSQL等。

14. **DE-9IM（Dimensionally Extended 9-Intersection Model，九交模型）**：一种精确描述两个几何对象拓扑关系的模型，通过比较两个对象的内部(I)、边界(B)、外部(E)之间9个相交矩阵元素来确定拓扑关系。PostGIS中用ST_Relate函数实现。

15. **SARG（Search ARGument，搜索参数）**：用于限制搜索的操作，形式为"列名 操作符 常数/变量"，可以利用索引快速定位数据。

16. **候选集（Candidate Set）**：空间查询过滤步后得到的结果集合，其中的对象"可能"满足查询条件，但不一定真正满足，需要经过求精步的精确计算才能确认。

17. **空间查询（Spatial Query）**：从空间数据库中查找出满足某一条件（空间关系或属性）的空间目标的过程。

18. **raster（栅格）数据类型**：PostGIS中用于存储多波段栅格图像的数据类型，不仅存储像素值，还存储空间参考、像素大小、偏移量和旋转等元数据信息。

## 简答/论述题预测

### 题1：什么是空间索引？为什么空间数据库需要专门的空间索引？

**参考答案要点**：

空间索引的定义（课件原话）见名词解释部分。

需要专门空间索引的原因：
1. **数据量庞大**：空间数据体量通常远大于普通数据库
2. **普通索引失效**：空间对象（点线面）是多维度的，B+树等一维索引无法直接用于多维空间对象的范围查询和相交查询
3. **查询代价高**：空间查询（相交、包含、邻近等）涉及复杂几何运算，如果逐一精确计算所有对象代价极高
4. **需要近似过滤**：通过MBR的快速过滤，先大幅减少候选集，再进行精确几何计算，实现高效查询

### 题2：请描述空间查询的"过滤—求精"两步查询框架。

**参考答案要点**：

**过滤步（Filter Step）**：
- 利用空间索引（如R树）存储空间对象的近似描述（MBR）
- 对MBR进行快速的相交/包含判断，排除不可能满足查询条件的对象
- 余下对象构成"候选集"

**求精步（Refinement Step）**：
- 对候选集中的每个对象，获取其真实的几何数据
- 采用精确的计算几何算法，测试是否真正满足查询条件（由空间谓词描述）
- 输出最终结果

**Oracle Spatial扩展**（三步）：在过滤步和求精步之间增加中间步骤，判断MBR是否完全包含于查询窗口内部，若是则直接输出结果，进一步减少求精步的计算量。Oracle Spatial使用双层查询模型（一级过滤+二级过滤）。

### 题3：比较R树与R+树的异同，各有什么优缺点？

**参考答案**（见1.4、1.5节及对比表）：

相同点：都是高度平衡树，每个节点存储MBR，用于空间数据索引。

不同点：
- R树：MBR可以重叠，但每个对象只属于一个叶节点（无冗余）
- R+树：MBR不允许重叠，但同一对象可能出现在多个叶节点（有冗余）

优缺点分析：
- R树：更新代价小，空间利用率高；缺点是MBR重叠导致查询时需访问多条路径
- R+树：查询时路径确定，效率更高；缺点是冗余存储导致更新代价高，空间占用大

### 题4：PostGIS中为什么使用GiST索引而不是B树索引？请解释GiST索引的工作原理。

**参考答案要点**：

- B树只能处理能按线性顺序排列的数据，不能直接处理多维空间几何对象
- GiST（通用搜索树）是一种架构，PostGIS通过其R树策略实现空间索引
- GiST索引不是对几何要素本身建索引，而是对**边界框（MBR）**建索引
- 查询过程：先用MBR判断是否相交（过滤步），再对候选集进行精确几何计算（求精步）
- 示例：查询与某几何体相交的对象，GiST先找出MBR重叠的对象，再精确判断是否真正相交

### 题5：PostGIS中常见的导致查询不走空间索引的情况有哪些？如何解决？

**参考答案要点**：

情况一：WHERE后对geom字段使用表达式函数
```sql
-- 错误：ST_Buffer(a.geom, 0.01)包装了geom字段
WHERE ST_Intersects(ST_Buffer(a.geom, 0.01), query_point)
-- 正确：Buffer应用于查询点（常量）
WHERE ST_Intersects(a.geom, ST_Buffer(query_point, 0.01))
```

情况二：使用ST_Distance代替ST_DWithin
```sql
-- 错误：不走索引
WHERE ST_Distance(a.geom, query_point) < 0.01
-- 正确：走索引
WHERE ST_DWithin(a.geom, query_point, 0.01)
```

情况三：特殊函数导致geom类型改变（如::geography转换）
- 解决：建立表达式索引 `CREATE INDEX ON table USING gist((geom::geography))`

情况四：使用ST_Disjoint（永远不走索引）
- 解决：用 NOT IN + ST_Intersects 替代

## 易错点总结

### 索引相关易错点

1. **R树 vs R+树混淆**
   - 易错：以为R树和R+树都不允许MBR重叠
   - 正确：**R树允许MBR重叠**，但每个对象只属于一个节点；**R+树不允许MBR重叠**，但对象可属于多个节点

2. **主索引是稠密索引还是稀疏索引**
   - 易错：认为主索引是稠密索引
   - 正确：**主索引是稀疏索引**（每个存储块一个索引项），**辅助索引是稠密索引**

3. **B树与B+树的区别**
   - 易错：认为两者叶节点相同
   - 正确：B+树所有数据指针仅在叶节点，且叶节点用链表连接；B树内部节点也可存数据指针

4. **GiST索引对什么建索引**
   - 易错：认为GiST对几何形状本身建索引
   - 正确：**GiST对几何要素的边界框（MBR）建索引**，不是对几何本身

5. **PostGIS中&&操作符含义**
   - 易错：认为&&是精确相交
   - 正确：&&是**边界框（MBR）重叠**判断，不是精确几何相交

### 查询函数易错点

6. **ST_Intersects vs ST_Intersection混淆**
   - ST_Intersects：**拓扑谓词**，返回布尔值（是否相交）
   - ST_Intersection：**处理函数**，返回几何对象（两对象相交的部分）

7. **ST_Within vs ST_Contains方向混淆**
   - `ST_Within(A, B)`：A被B包含（A在B的内部）
   - `ST_Contains(A, B)`：A包含B（B在A的内部）
   - 记忆：Contains = 包含（主语包含宾语），Within = 在...内（主语在宾语内）

8. **ST_Distance不走索引**
   - 易错：用ST_Distance在WHERE后做判断
   - 正确：应使用**ST_DWithin**替代，ST_DWithin走索引，ST_Distance不走索引

9. **ST_Disjoint永远不走索引**
   - 必须用"ST_Intersects取反"方案替代

10. **WHERE后对geom做函数计算导致索引失效**
    - 凡是对geom列做任何函数计算（Buffer、类型转换等），都会使索引失效
    - 函数计算应施加在查询参数（常量）上，而非geom字段

### SQL语言易错点

11. **Intersection() 函数 vs Intersects 谓词**
    - 在OGIS和Oracle SQL中：`Intersection()` 返回几何对象，`Intersects` 返回布尔值
    - 两者用途完全不同

12. **Touch谓词的含义**
    - Touch：两个几何对象彼此**相邻而又不交叠**（边界相接，内部不相交）
    - 不是"接触"（不等于相交），而是"共享边界但不共享内部"

13. **OGIS标准的局限性**
    - 不支持场模型（只支持对象模型）
    - 不支持方向谓词（东南西北）
    - GROUP BY、HAVING存在问题

14. **过滤步的输出是候选集，不是最终结果**
    - 候选集中的对象仅是"可能"满足条件，还需经过求精步精确验证

### 其他易错点

15. **Oracle Spatial是双层查询**，不是单层；两层过滤才得精确结果

16. **栅格空间索引基于凸包**（ST_ConvexHull），不是MBR；大多数栅格操作符基于栅格的凸包

17. **raster2pgsql的-I参数**：在栅格列上创建GiST索引（不是普通索引）

18. **ST_Union在WHERE条件中使用会严重影响性能**，应只在输出结果中使用

19. **SQL执行顺序**（易混淆）：FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
