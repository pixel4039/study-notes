## ② 知识点逐条精讲

### 1. 数据库基本概念

#### 1.1 数据（Data）

数据是描述现实世界事物的符号记录，是数据库中存储的基本单位。数据库中数据的**最小单位**是**数据项**或**记录**，记录由数据项组成。

#### 1.2 数据库（Database，DB）

**定义（课件原话）**：数据库（Database，简记为 DB），是**长期存储在计算机内、有组织的、可共享的大量相关数据的集合**。

**关键词拆解**：

| 关键词 | 含义 |
|--------|------|
| 长期存储 | 数据持久化存放，不随程序结束而消失 |
| 有组织 | 按照数据模型组织，有结构规律 |
| 可共享 | 多个用户、多个应用程序可同时使用 |
| 大量相关数据 | 面向某一应用领域，数据之间存在联系 |

**数据库的五大特征**：
1. 数据按一定的数据模型组织、描述和储存
2. 可为各种用户共享
3. 冗余度较小
4. 数据独立性较高
5. 易扩展

#### 1.3 数据库管理系统（DBMS）

**定义**：数据库管理系统（Database Management System，简称 DBMS）是**位于用户与操作系统之间**的一层**数据管理软件**。

**DBMS 的用途**：组织和存储好大量的数据，并提供方便、高效地检索数据和维护数据的手段。

**DBMS 的三大核心功能**（对应三种语言）：

| 功能 | 语言名称 | 英文全称 | 具体内容 |
|------|----------|----------|----------|
| 数据库**定义**功能 | DDL | Data Definition Language | 创建数据库、定义表名称、建立表结构等 |
| 数据库**操纵**功能 | DML | Data Manipulation Language | 向表中增加/删除/更新数据以及查询、检索、统计等 |
| 数据库**控制**功能 | DCL | Data Control Language | 数据安全性控制、完整性控制、并发控制和数据库恢复等 |

> 这三大功能统一用 **SQL（Structured Query Language，结构化查询语言）** 来实现。

**DBMS 提供的四项数据控制功能**：
1. **数据安全性（Security）保护**：保护数据，以防止不合法的使用造成的数据的泄密和破坏。
2. **数据完整性（Integrity）检查**：将数据控制在有效的范围内，或保证数据之间满足一定的关系。
3. **并发（Concurrency）控制**：对多用户的并发操作加以控制和协调，防止相互干扰而得到错误的结果。
4. **数据库恢复（Recovery）**：将数据库从错误状态恢复到某一已知的正确状态。

#### 1.4 数据库系统（DBS）

**定义**：数据库系统（Database System，简称 DBS）是指**带有数据库并利用数据库技术进行数据管理的计算机系统**。

**DBS 的构成（四要素）**：
1. **数据库（DB）**——存储的数据本身
2. **数据库管理系统（DBMS）及其开发工具**——管理数据的软件
3. **应用系统（应用程序）**——具体的业务软件
4. **数据库管理员（DBA）**——负责管理维护的人员

**三者包含关系**：**DBS 包含 DB 和 DBMS**（DBS ⊃ DBMS ⊃ / DB）。

> 易错：DB、DBS、DBMS 不是同一个概念！DBS 是最大的整体，包含了 DB（数据）和 DBMS（软件）以及应用和人员。

#### 1.5 数据库管理员（DBA）的职责

DBA 是专门负责管理和维护数据库系统的专业人员，其职责包括：
- 决定数据库中的信息内容和结构
- 决定数据库的存储结构和存取策略
- 定义数据的安全性要求和完整性约束条件
- 监控数据库的使用和运行
- 负责数据库的改进和重组、重构

### 2. 数据管理技术的发展

#### 2.1 什么是数据管理

数据管理是指对数据进行**分类、组织、编码、存储、检索和维护**，是数据处理的中心问题。

#### 2.2 三阶段对比表

| 比较维度 | 人工管理阶段 | 文件系统阶段 | 数据库系统阶段 |
|----------|-------------|-------------|---------------|
| **时期** | 20世纪40年代中期—50年代中期 | 20世纪50年代末—60年代中期 | 20世纪60年代末至今 |
| **应用背景** | 科学计算 | 科学计算、管理 | 大规模数据管理 |
| **硬件水平** | 无直接存取存储设备 | 有磁盘、磁鼓 | 大容量磁盘、磁盘阵列 |
| **软件水平** | 没有操作系统 | 有文件系统（专门数据管理软件） | 有数据库管理系统（DBMS） |
| **处理方式** | 批处理 | 联机实时处理、批处理 | 联机实时处理、分布处理、批处理 |
| **数据保存** | 不保存（随程序结束消失） | 可长期保存 | 可长期保存 |
| **数据管理者** | 应用程序员自己管理 | 文件系统管理 | DBMS 统一管理 |
| **数据共享性** | 无共享，冗余度极大 | 共享性差，冗余度大 | 共享性高，冗余度小 |
| **数据独立性** | 完全不独立，依赖程序 | 独立性差，逻辑结构改变必须修改程序 | 有较高的数据独立性 |
| **数据结构化** | 无结构化 | 文件内部有结构，整体无结构 | 整体结构化，采用数据模型 |
| **数据控制能力** | 应用程序自己控制 | 应用程序自己控制 | DBMS 提供安全、完整、并发、恢复控制 |

**程序与数据的对应关系**：
- 人工管理：每个应用程序对应自己的数据集，一一对应，完全依赖
- 文件系统：多个应用程序共用文件，但每个程序使用特定文件
- 数据库系统：多个应用程序共用同一数据库，DBMS 统一调度

> 记忆要点：从人工→文件→数据库，共享性越来越高，冗余度越来越低，独立性越来越高，结构化程度越来越强。

#### 2.3 数据库系统阶段的四大特点（重点简答）

1. **数据结构化**：在描述数据时不仅要描述数据本身，还要描述数据之间的联系。在同一数据库中，各个数据文件存在联系，且在整体上服从一定的结构形式。
2. **数据的共享性高、冗余度小、易扩充**：数据面向整个系统，可以被多个用户、多个应用共享使用。减少数据冗余，节约存储空间，避免数据之间的不相容性与不一致性。
3. **有较高的数据独立性**：物理独立性和逻辑独立性（见三级模式部分详解）。
4. **数据由 DBMS 统一管理和控制**：DBMS 提供安全性、完整性、并发控制和数据库恢复四项控制功能。

### 3. 数据库系统的演变与发展（DBMS 三代）

| 代数 | 代表系统 | 数据模型 | 时间 | 特点 |
|------|----------|----------|------|------|
| **第一代** | IDS（1961，通用电气，Charles Bachman）；IMS（1968，IBM） | 网状模型（图）+ 层次模型（树） | 1960年代 | 世界上第一个 NDBMS（IDS）和第一个 HDBMS（IMS） |
| **第二代** | System R；Oracle；MySQL 等 | **关系模型**（表） | 1970年代起 | E.F.Codd 1970年提出关系模型；1974年 Boyce&Chamberlin 提出 SQL；1987年 ISO 颁布 SQL 国际标准 |
| **第三代** | 面向对象数据库、对象关系数据库 | 面向对象模型 | 1990年代起 | 1990年《第三代数据库系统宣言》：支持数据/对象/知识管理；继承第二代技术；对其他系统开放 |

**当前主流数据库分类**：
- **关系型数据库**：
  - 商业：Oracle、Microsoft SQL Server、IBM DB2
  - 开源：MySQL、PostgreSQL、SQLite
- **NOSQL（非关系型数据库）**：
  - 文档型：MongoDB
  - 列存储：Cassandra
  - 键值存储：Redis
  - 图数据库：Neo4J

### 4. 数据库系统的标准结构（三级模式两级映像）

#### 4.1 模式与视图的基本概念

- **模式（Schema）**：对数据库中数据所进行的一种**结构性的描述**，即观察到数据的结构信息（描述数据的结构/格式）。
- **视图（View）/ 数据（Data）**：某一种表现形式下表现出来的数据库中的**实际数据**（即根据模式存储或展示的数据）。

> 区分：模式 = 数据的结构描述（如表定义）；视图/数据 = 实际展现的数据（如表中的记录）。

#### 4.2 DBMS 管理数据的三个层次

| 层次 | 英文名称 | 别名 | 含义 |
|------|----------|------|------|
| 外部层 | External Level | User Level（用户层） | 某一用户能够看到与处理的数据，全局数据中的某一部分 |
| 概念层 | Conceptual Level | Logic Level（逻辑层） | 从全局角度理解/管理的数据，含相应的关联约束 |
| 内部层 | Internal Level | Physical Level（物理层） | 存储在介质上的数据，含存储路径、存储方式、索引方式等 |

#### 4.3 三级模式结构（核心重点）

```
用户/应用程序
      ↕（使用）
  外模式（External Schema）= 外部视图（External View）
      ↕（E-C 映像）
  概念模式（Conceptual Schema）= 概念视图（Conceptual View）
      ↕（C-I 映像）
  内模式（Internal Schema）= 内部视图（Internal View）
      ↕（物理访问）
    物理存储（磁盘等）
```

**三级模式各自的定义与作用**：

1. **外模式（External Schema）**
   - 又称**子模式**或**用户模式**
   - 定义：某一用户能够看到与处理的数据的结构描述，是数据库用户（包括应用程序员和最终用户）使用的局部数据的逻辑结构和特征的描述。
   - 作用：每个用户只能看到和访问所对应的外模式中的数据，保护了数据安全性。
   - 一个数据库可以有**多个**外模式（每个用户或应用可以有自己的外模式）。
   - 在 SQL 中，外模式对应**视图（View）**。

2. **概念模式（Conceptual Schema）**
   - 又称**模式**或**逻辑模式**
   - 定义：从全局角度理解/管理的数据的结构描述，包含相应的关联约束，体现数据之间的内在本质联系。
   - 作用：是数据库的核心，描述了数据库中**全部数据**的逻辑结构和特征，以及数据间的联系。
   - 一个数据库**只有一个**概念模式。
   - 在 SQL 中，概念模式对应**基本表（Base Table）**。

3. **内模式（Internal Schema）**
   - 又称**存储模式**或**物理模式**
   - 定义：存储在介质上的数据的结构描述，包含存储路径、存储方式、索引方式等。
   - 作用：描述数据在存储介质上的物理存储方式和物理结构。
   - 一个数据库**只有一个**内模式。

#### 4.4 两级映像

| 映像名称 | 英文简写 | 功能 | 位置 |
|----------|----------|------|------|
| **外模式—概念模式映像** | E-C Mapping | 将外模式映射为概念模式，支持实现数据概念视图向外部视图的转换，便于用户观察和使用 | 外模式与概念模式之间 |
| **概念模式—内模式映像** | C-I Mapping | 将概念模式映射为内模式，支持实现数据概念视图向内部视图的转换，便于计算机进行存储和处理 | 概念模式与内模式之间 |

#### 4.5 数据独立性（重点！务必搞清对应关系）

| 独立性类型 | 定义（课件原话） | 由哪级映像保证 | 改变时的处理方式 |
|------------|-----------------|---------------|-----------------|
| **逻辑数据独立性** | 用户的应用程序与数据库的逻辑结构是相互独立的。数据的逻辑结构改变了，用户程序也可以不变。 | **E-C 映像**（外模式—概念模式映像） | 当概念模式变化时，只需改变 E-C Mapping，无需改变外部模式，从而无需改变应用程序 |
| **物理数据独立性** | 用户的应用程序与存储在磁盘上的数据库中数据是相互独立的。当数据的物理存储改变了，应用程序不用改变。 | **C-I 映像**（概念模式—内模式映像） | 当内部模式变化时，只需改变 C-I Mapping，无需改变概念模式，从而不改变外部模式 |

> 记忆口诀：**逻辑独立靠 E-C，物理独立靠 C-I**。逻辑→概念层变化→E-C 映像调整保证应用不变；物理→内部层变化→C-I 映像调整保证逻辑不变。

### 5. 数据库的数据模型

#### 5.1 数据模型的定义

**数据模型**：规定模式统一描述方式的模型，包括数据结构、操作和约束。数据模型是对模式本身结构的抽象，模式是对数据本身结构形式的抽象。

#### 5.2 数据模型的三要素

| 要素 | 描述 | 特征 |
|------|------|------|
| **数据结构** | 数据的表示方法（数据的组织形式） | 静态特征的描述 |
| **数据操作** | 对数据库的查询、修改、删除和插入等操作 | 动态特征的描述 |
| **完整性约束** | 数据及数据间联系应具有的制约和依赖规则 | 如：一个系可有多个学生，一个学生只能属于一个系 |

#### 5.3 三种经典逻辑模型对比表

| 比较维度 | 层次模型（Hierarchical） | 网状模型（Network） | 关系模型（Relational） |
|----------|------------------------|---------------------|----------------------|
| **数据结构** | 有序**树**（森林） | **图** | **二维表**（关系/表） |
| **节点联系** | 有且仅有一个根节点；其他节点有且仅有一个父节点 | 允许一个以上节点无父节点；一个节点可以有多于一个父节点 | 行（元组）之间通过外键关联 |
| **代表系统** | IMS（IBM，1968年） | IDS（通用电气，1961年） | Oracle、MySQL 等（第二代） |
| **能否直接表示 m:n 关系** | 不能（只能表示1:1和1:m） | 能 | 能（通过外键/中间表） |
| **优点** | 结构简单，查询效率高 | 能反映现实中复杂的联系 | 结构简单，理论基础扎实，SQL支持 |
| **缺点** | 不能直接表示多对多关系 | 结构复杂，用户使用不方便 | 早期查询效率较低（现已改善） |

> 选择题常考：层次模型→树；网状模型→图；关系模型→表。层次模型不能直接表示 m:n 关系！

## ④ 简答 / 论述题预测

### 【简答1】数据库系统阶段的主要特点是什么？

答：数据库系统阶段具有以下四个主要特点：
1. **数据结构化**：在描述数据时不仅描述数据本身，还描述数据之间的联系，整体数据在整体上服从一定的结构形式。
2. **数据共享性高、冗余度小、易扩充**：数据面向整个系统，可被多个用户、多个应用共享，从而减少冗余，避免数据不相容和不一致。
3. **有较高的数据独立性**：具备逻辑独立性和物理独立性，由 DBMS 的二级映像功能来保证。
4. **数据由 DBMS 统一管理和控制**：DBMS 提供安全性保护、完整性检查、并发控制和数据库恢复四项控制功能。

### 【简答2】什么是三级模式结构？有什么意义？

答：三级模式结构是指数据库系统采用外模式、概念模式、内模式三个层次来组织和管理数据。
- **外模式**（子模式）：用户使用的局部数据的逻辑结构描述，对应视图，一个数据库可以有多个外模式。
- **概念模式**（逻辑模式）：全局数据的逻辑结构描述，对应基本表，一个数据库只有一个。
- **内模式**（存储模式）：数据在存储介质上的物理结构描述，一个数据库只有一个。

意义：通过两级映像（E-C 映像和 C-I 映像）实现了数据的逻辑独立性和物理独立性，有利于保持数据库的数据独立性，便于维护和扩展。

### 【简答3】逻辑独立性和物理独立性分别由什么保证？

答：
- **物理数据独立性**：由**概念模式—内模式映像（C-I 映像）**保证。当内部模式（物理存储结构）变化时，只需修改 C-I 映像，概念模式无需改变，从而外模式和应用程序也无需改变。
- **逻辑数据独立性**：由**外模式—概念模式映像（E-C 映像）**保证。当概念模式变化时，只需修改 E-C 映像，外模式无需改变，从而应用程序也无需改变。

### 【简答4】DBMS 的主要功能是什么？

答：DBMS 的主要功能分为三大类：
1. **数据库定义功能（DDL）**：创建数据库、定义表名称、建立表结构及各属性的约束条件等。
2. **数据库操纵功能（DML）**：向数据库的表中增加/删除/更新数据以及进行查询、检索、统计等操作。
3. **数据库控制功能（DCL）**：包括数据安全性控制、数据完整性控制、多用户环境下的并发控制和数据库恢复。

---

## ⑤ 易错点提醒

1. **DB ≠ DBMS ≠ DBS**：三者不能混用。DBS 是包含 DB、DBMS、应用系统和 DBA 的整体系统；DBMS 是管理软件；DB 是数据本身。选择题"DB 就是 DBS"是错的。
2. **数据库不保存的是"人工管理阶段"**：文件系统阶段数据可以长期保存，人工管理阶段数据不保存。
3. **模式（全局逻辑）vs 外模式（局部逻辑）vs 内模式（物理）**：选择题"用户看到的那部分局部逻辑结构"是**外模式/子模式**，不是"模式"。
4. **逻辑独立性 vs 物理独立性与映像对应**：记住"保证逻辑独立性，修改的是**外模式—概念模式之间的映射**"；"保证物理独立性，修改的是**概念模式—内模式之间的映射**"。
5. **层次模型不能直接表示 m:n 关系**，只能表示 1:1 和 1:m 关系。
6. **"数据库技术的主要特点"不包括"程序的标准化"**：标准化是程序设计原则，不是数据库特点。

## ② 知识点逐条精讲

### 1. SQL 概述

#### 1.1 SQL 的历史

- SQL（Structured Query Language，**结构化查询语言**）由 IBM 的 **Boyce 和 Chamberlin** 于 **1974 年**提出。
- 最初在 IBM 公司的关系数据库系统 **System R** 上实现。
- **1986 年**，ANSI 数据库委员会批准 SQL 作为数据库语言的美国标准；ISO 随后也提出了同样决定。
- **1987 年**，国际标准化组织（ISO）颁布 SQL 正式国际标准。

#### 1.2 SQL 标准演进

| 标准版本 | 主要特性 |
|---------|---------|
| SQL-86 | 数据库语言 SQL 的第一个标准 |
| SQL-89 | 增加了对完整性约束的支持（"具有完整性增强的数据库语言SQL"） |
| SQL-92 | SQL-89 的超集，增加了新的数据类型、更丰富的数据操作、更强的完整性和安全性支持 |
| SQL-99 | 增加了面向对象特性等 |
| SQL-2003 | 进一步扩展 |

> 注：不同 DBMS 产品支持 SQL 的程度有所不同（就像不同方言）。

#### 1.3 SQL 分类

**按使用方式**：交互式 SQL → 嵌入式 SQL → 动态 SQL

**按功能分类（重点）**：

| 分类 | 英文全称 | 中文名 | 主要语句 |
|------|----------|--------|----------|
| **DDL** | Data Definition Language | 数据定义语言 | CREATE、DROP、ALTER |
| **DML** | Data Manipulation Language | 数据操纵语言 | INSERT、UPDATE、DELETE |
| **DCL** | Data Control Language | 数据控制语言 | GRANT、REVOKE |
| **DQL** | Data Query Language | 数据查询语言 | SELECT |
| **TCL** | Transaction Control Language | 事务控制语言 | COMMIT、ROLLBACK |

#### 1.4 SQL 的特点

1. **综合统一**：集数据定义、操纵、查询、控制于一体
2. **非过程化**：用户只需说明"做什么"，无需说明"怎么做"
3. **面向集合**：操作对象和结果都是记录的集合
4. **功能丰富**：使用方便、灵活、语言简洁易学
5. **应用系统范围广**：统一标准，所有主流 DBMS 均支持

### 2. 数据定义（DDL）

#### 2.1 创建与删除数据库

```sql
-- 创建数据库
CREATE DATABASE 数据库名;

-- 示例：创建课程学习数据库 SCT
CREATE DATABASE SCT;

-- 删除数据库
DROP DATABASE [IF EXISTS] 数据库名;

-- 示例：删除 SCT
DROP DATABASE SCT;
```

> DDL 通常由 DBA 使用，也有经 DBA 授权后由应用程序员使用。

#### 2.2 基本表的定义（CREATE TABLE）

**语法**：
```sql
CREATE TABLE 表名 (
    列名  数据类型  [PRIMARY KEY | UNIQUE]  [NOT NULL]
    [, 列名  数据类型  [NOT NULL], ...]
);
```

**三种约束说明**：

| 约束 | 关键字 | 说明 |
|------|--------|------|
| 主键约束 | `PRIMARY KEY` | 每个表**只能创建一个**主键约束，主键列值唯一且不能为 NULL |
| 唯一性约束 | `UNIQUE` | 即候选键，可以有**多个**唯一性约束，允许 NULL |
| 非空约束 | `NOT NULL` | 该列不允许有空值出现 |

**示例**：
```sql
-- 定义学生表 Student
CREATE TABLE Student (
    S#      char(8)    NOT NULL,
    Sname   char(10),
    Ssex    char(2),
    Sage    integer,
    D#      char(2),
    Sclass  char(6)
);

-- 定义课程表 Course
CREATE TABLE Course (
    C#      char(3),
    Cname   char(12),
    Chours  integer,
    Credit  float(1),
    T#      char(3)
);
```

#### 2.3 常用数据类型

| 数据类型 | 说明 | 备注 |
|---------|------|------|
| `char(n)` | 固定长度字符串，长度为 n | 长度固定，不足时补空格 |
| `varchar(n)` | 可变长字符串，最大长度为 n | 实际占用按实际内容 |
| `int` / `integer` | 整数 | 两种写法均可 |
| `numeric(p, q)` | 固定精度数字，小数点左边 p 位，右边 p-q 位 | |
| `real` / `float(n)` | 浮点精度数字，小数点后保留 n 位 | 两种写法均可 |
| `date` | 日期（如 2003-09-12） | |
| `time` | 时间（如 23:15:00） | |

> **char vs varchar 区别**：char 是定长，分配固定空间；varchar 是变长，按实际内容占用空间。

#### 2.4 删除基本表（DROP TABLE）

```sql
DROP TABLE 表名;

-- 示例
DROP TABLE Student;
DROP TABLE Teacher;
```

> 注意：`DELETE` 语句只删除表中的**元组（数据）**，而 `DROP TABLE` 则撤销包括表格式、表中所有元组、由该表导出的视图等所有相关内容，使用时要特别注意！

#### 2.5 修改基本表（ALTER TABLE）

```sql
ALTER TABLE 表名
    [ADD {列名 数据类型, ...}]       -- 增加新列
    [DROP {完整性约束名}]            -- 删除完整性约束
    [MODIFY {列名 数据类型, ...}];   -- 修改列定义
```

**示例**：
```sql
-- 在 Student 表中增加两列（住址和身份证号）
ALTER TABLE Student ADD Saddr char(40), PID char(18);

-- 将 Sname 列的数据类型修改为12个字符长度
ALTER TABLE Student MODIFY Sname char(12);

-- 删除 Sname 列的唯一性约束
ALTER TABLE Student DROP UNIQUE(Sname);
```

### 3. 数据查询（SELECT，核心）

#### 3.1 SELECT 的基本语法结构

```sql
SELECT 列名 [[, 列名] ...]
FROM 表名
[WHERE 检索条件];
```

**语义**：从 FROM 子句指定的表中，查询满足 WHERE 检索条件的元组，并按 SELECT 给定的列名及顺序进行投影显示。

#### 3.2 基本查询示例

```sql
-- 检索所有学生信息
SELECT S#, Sname, Ssex, Sage, Sclass, D# FROM Student;
SELECT * FROM Student;   -- * 表示所有列

-- 检索所有学生的姓名及年龄
SELECT Sname, Sage FROM Student;

-- 检索所有年龄小于等于19岁的学生的年龄及姓名
SELECT Sage, Sname FROM Student WHERE Sage <= 19;
```

#### 3.3 WHERE 子句常用检索条件

| 条件类型 | 语法 | 说明 | 示例 |
|---------|------|------|------|
| **比较运算** | `=, <>, <, >, <=, >=` | 数值/字符比较 | `WHERE Sage <= 19` |
| **逻辑运算** | `AND, OR, NOT` | 组合条件 | `WHERE Salary < 1500 OR Salary > 2000` |
| **范围** | `BETWEEN...AND` | 包含端点，等价于 `>= AND <=` | `WHERE Sage BETWEEN 18 AND 20` |
| **集合** | `IN (值1, 值2, ...)` 或 `NOT IN` | 判断是否在集合内 | `WHERE Sname IN ('张三', '王三')` |
| **模糊匹配** | `LIKE '字符串'` 或 `NOT LIKE` | 使用通配符 | `WHERE Sname LIKE '张%'` |
| **空值判断** | `IS NULL` 或 `IS NOT NULL` | 注意：不能用 `= NULL` | `WHERE Score IS NULL` |

> 运算符优先级（从高到低）：算术运算符 > 比较运算符 > NOT > AND > OR

#### 3.4 LIKE 模糊查询（重点）

**语法**：
```sql
列名 [NOT] LIKE '字符串'
```

**通配符**：

| 通配符 | 含义 | 示例 |
|--------|------|------|
| `%` | 匹配**零个或多个**任意字符 | `'张%'` 匹配所有姓张的 |
| `_`（下划线） | 匹配**任意单个**字符 | `'张_ _'` 匹配张某某（两个字） |
| `\`（反斜杠） | 转义字符，使特殊字符被作为普通字符 | `'\%'` 匹配字符 `%`；`'\_'` 匹配字符 `_` |

**示例**：
```sql
-- 检索所有姓张的学生学号及姓名
SELECT S#, Sname FROM Student WHERE Sname LIKE '张%';

-- 检索名字为张某某的所有同学（三个字，姓张）
SELECT Sname FROM Student WHERE Sname LIKE '张_ _';

-- 检索名字不姓张的所有同学
SELECT Sname FROM Student WHERE Sname NOT LIKE '张%';
```

#### 3.5 DISTINCT 去除重复元组

```sql
-- 有重复（默认）
SELECT S# FROM SC WHERE Score > 80;

-- 去掉重复（只保留一份）
SELECT DISTINCT S# FROM SC WHERE Score > 80;
```

> 说明：缺省为保留重复元组（ALL），加 `DISTINCT` 或 `UNIQUE` 关键字可去掉重复元组。

#### 3.6 ORDER BY 排序

```sql
ORDER BY 列名 [ASC | DESC]
```

- `ASC`：升序（默认，从小到大）
- `DESC`：降序（从大到小）
- ORDER BY 必须在**所有其他子句之后**作为最后一个子句出现

**示例**：
```sql
-- 按学号由小到大显示所有学生的学号及姓名
SELECT S#, Sname FROM Student ORDER BY S# ASC;

-- 检索002号课大于80分的同学学号，按成绩由高到低排序
SELECT S# FROM SC WHERE C# = '002' AND Score > 80 ORDER BY Score DESC;
```

### 4. 连接查询

#### 4.1 多表联合查询（等值连接）

```sql
SELECT 列名 [[, 列名] ...]
FROM 表名1, 表名2, ...
WHERE 检索条件;  -- 检索条件中要包含连接条件
```

**等值连接示例**：
```sql
-- 按"001"号课成绩由高到低顺序显示所有学生的姓名（二表连接）
SELECT Sname FROM Student, SC
WHERE Student.S# = SC.S# AND SC.C# = '001'
ORDER BY Score DESC;

-- 按"数据库"课成绩由高到低顺序显示所有同学姓名（三表连接）
SELECT Sname FROM Student, SC, Course
WHERE Student.S# = SC.S# AND SC.C# = Course.C#
AND Cname = '数据库'
ORDER BY Score DESC;
```

> 多表连接时，如果两个表有相同属性名，必须采用**表名.属性名**的方式限定。

#### 4.2 表别名

```sql
SELECT 列名 AS 列别名 [[, 列名 AS 列别名] ...]
FROM 表名1 AS 表别名1, 表名2 AS 表别名2, ...
WHERE 检索条件;
```

> `AS` 可以省略。定义别名后可在检索条件中使用别名限定属性。

**示例**：
```sql
-- 求年龄有差异的任意两位同学的姓名（同一张表自连接，必须用别名）
SELECT S1.Sname AS Stud1, S2.Sname AS Stud2
FROM Student S1, Student S2
WHERE S1.Sage > S2.Sage;
```

#### 4.3 多表联合查询练习

```sql
-- 求既学过"001"号课又学过"002"号课的所有学生的学号
SELECT S1.S# FROM SC S1, SC S2
WHERE S1.S# = S2.S# AND S1.C# = '001' AND S2.C# = '002';

-- 求"001"号课成绩比"002"号课成绩高的所有学生的学号
SELECT S1.S# FROM SC S1, SC S2
WHERE S1.S# = S2.S# AND S1.C# = '001' AND S2.C# = '002'
AND S1.Score > S2.Score;
```

### 5. 数据操纵（DML）

#### 5.1 INSERT INTO 插入数据

**单一元组新增**：
```sql
INSERT INTO 表名 [(列名 [, 列名] ...)]
VALUES (值 [, 值] ...);
```

- VALUES 后面值的排列须与 INTO 子句后面的列名排列**一致**
- 若省略所有列名，则 VALUES 后的值须与表中列名排列**一致**

```sql
-- 示例：追加学生表中的元组
INSERT INTO Student
VALUES ('98030101', '张三', '男', 20, '03', '980301');

INSERT INTO Student (S#, Sname, Ssex, Sage, D#, Sclass)
VALUES ('98030102', '张四', '女', 20, '03', '980301');

-- 示例：追加课程表（列名省略时顺序必须与定义一致）
INSERT INTO Course
VALUES ('001', '数据库', 40, 6, '001');

-- 示例：追加课程表（指定列名时，顺序与列名顺序一致）
INSERT INTO Course (Cname, C#, Credit, Chours, T#)
VALUES ('数据库', '001', 6, 40, '001');
```

**批数据新增（子查询形式）**：
```sql
INSERT INTO 表名 [(列名 [, 列名] ...)]
子查询;
```

```sql
-- 示例：将名字中含"伟"的同学新增到St表中
INSERT INTO St (S#, Sname)
SELECT S#, Sname FROM Student WHERE Sname LIKE '%伟';

-- 示例：将成绩及格的同学记录新增到SCt表中
INSERT INTO SCt (S#, C#, Score)
SELECT S#, C#, Score FROM SC WHERE Score >= 60;

-- 示例：将每个学生平均成绩新增到St表中
INSERT INTO St (S#, Sname, avgScore)
SELECT S#, Sname, AVG(Score) FROM Student, SC
WHERE Student.S# = SC.S#
GROUP BY Student.S#;
```

> 注意：新增元组时，DBMS 会检查完整性约束条件，不符合则不执行新增操作。

#### 5.2 DELETE 删除数据

```sql
DELETE FROM 表名 [WHERE 条件表达式];
```

- 如果省略 WHERE 条件，则删除**所有元组**（清空表，但保留表结构）

```sql
-- 删除SC表中所有元组
DELETE FROM SC;

-- 删除98030101号同学所选的所有课程
DELETE FROM SC WHERE S# = '98030101';

-- 删除自动控制系的所有同学（用子查询）
DELETE FROM Student WHERE D# IN
    (SELECT D# FROM Dept WHERE Dname = '自动控制');
```

#### 5.3 UPDATE 更新数据

```sql
UPDATE 表名
SET 列名 = 表达式 | (子查询)
    [[, 列名 = 表达式 | (子查询)] ...]
[WHERE 条件表达式];
```

- 如果省略 WHERE 条件，则更新**所有元组**

```sql
-- 将所有教师工资上调5%
UPDATE Teacher SET Salary = Salary * 1.05;

-- 将所有计算机系的教师工资上调10%
UPDATE Teacher SET Salary = Salary * 1.1
WHERE D# IN (SELECT D# FROM Dept WHERE Dname = '计算机');
```

| **SQL** | 结构化查询语言（Structured Query Language），1974年由 Boyce 和 Chamberlin 提出，是关系数据库的标准语言，具有综合统一、非过程化、面向集合等特点 |

## ④ 简答 / 写 SQL 题预测

### 【简答1】SQL 的主要特点是什么？

答：
1. **综合统一**：SQL 集数据定义（DDL）、数据操纵（DML）、数据控制（DCL）和数据查询（DQL）于一体，功能全面。
2. **非过程化**：用户只需描述"做什么"，无需说明"怎么做"，存取路径的选择由 DBMS 负责。
3. **面向集合**：SQL 的操作对象和操作结果都是元组的集合，不是逐条处理记录。
4. **简洁易学**：语言接近自然语言，关键词少，语法简单。
5. **统一标准**：所有主流 DBMS 都支持 SQL，是操作数据库的标准语言。

### 【简答2】char 和 varchar 有什么区别？

答：
- `char(n)`：**固定长度**字符串，无论实际内容多少，都占用 n 个字符的存储空间；若实际字符不足 n 个，系统会自动补空格。
- `varchar(n)`：**可变长度**字符串，最大长度为 n，实际占用存储空间等于实际字符串的长度（不补空格）。
- 区别：char 存储效率高但可能浪费空间；varchar 节约空间但有额外的长度信息开销。固定长度的数据（如学号、性别）用 char；长度变化大的数据（如地址、备注）用 varchar。

### 【写SQL1】查询"数据库"课成绩由低到高顺序显示所有同学姓名和成绩

```sql
SELECT Sname, Score
FROM Student, SC, Course
WHERE Student.S# = SC.S# AND SC.C# = Course.C#
AND Cname = '数据库'
ORDER BY Score ASC;
```

### 【写SQL2】查询03系的学生学号、姓名、性别和所在系编号

```sql
SELECT Sno, Sname, Ssex, Dno
FROM Student
WHERE Dno = '03';
```

### 【写SQL3】求"002"号课成绩比"003"号课成绩高的所有学生的学号

```sql
SELECT S1.Sno FROM SC S1, SC S2
WHERE S1.Sno = S2.Sno AND S1.Cno = '002'
AND S2.Cno = '003' AND S1.Score > S2.Score;
```

## ⑤ 易错点提醒

1. **DROP TABLE vs DELETE**：`DROP TABLE` 删除整张表（含结构+数据+相关视图）；`DELETE FROM 表名` 只删除数据，保留表结构。
2. **LIKE 中下划线是单字符通配符**：`'张_ _'` 表示三个字的名字（"张"+"任意字"+"任意字"），两个下划线之间**不加空格**，课件中写法请注意辨认。
3. **`IS NULL` 不能写成 `= NULL`**：空值判断必须用 `IS NULL` 或 `IS NOT NULL`。
4. **ORDER BY 必须在最后**：ORDER BY 子句必须在所有其他子句之后作为最后一个子句出现。
5. **多表连接时同名属性必须加表名前缀**：如 `Student.S#` 而不是直接写 `S#`。
6. **INSERT INTO 列名省略时，VALUES 中值的顺序必须与表定义顺序完全一致**。
7. **聚合函数不能用于 WHERE 子句**（见第四讲 HAVING vs WHERE 部分）。
8. **DDL 由 DBA 使用，DML 由用户/应用程序员使用**。

## ② 知识点逐条精讲

### 1. 聚合函数

#### 1.1 五个聚合函数

SQL 提供了五个作用在简单列值集合上的内置聚集函数（agfunc）：

| 函数 | 功能 | 参数类型 |
|------|------|---------|
| `COUNT(*)` | 求元组个数（行数） | 无特定类型 |
| `COUNT(列名)` | 求该列非 NULL 值的个数 | 任意 |
| `SUM(列名)` | 求该列所有值之和 | 数值型 |
| `AVG(列名)` | 求该列所有值的平均值 | 数值型 |
| `MAX(列名)` | 求该列最大值 | 可比较类型 |
| `MIN(列名)` | 求该列最小值 | 可比较类型 |

#### 1.2 NULL 在聚合函数中的处理规则（重要易错）

| 函数 | 对 NULL 的处理 |
|------|---------------|
| **COUNT(\*)** | 计入（统计所有行，包括含 NULL 的行） |
| **COUNT(列名)** | 不计入（只统计非 NULL 的值） |
| **SUM** | 忽略 NULL（不将其计入求和） |
| **AVG** | 忽略 NULL（含 NULL 的元组不参与计算均值，分子分母均不含该元组） |
| **MAX / MIN** | 忽略 NULL（NULL 不参与比较） |

> 总结：除 `COUNT(*)` 外，所有聚合函数都**忽略 NULL 值**。

#### 1.3 聚合函数示例

```sql
-- 求教师的工资总额
SELECT SUM(Salary) FROM Teacher;

-- 求计算机系教师的工资总额
SELECT SUM(Salary) FROM Teacher T, Dept
WHERE Dept.Dname = '计算机' AND Dept.D# = T.D#;

-- 求数据库课程的平均成绩
SELECT AVG(Score) FROM Course C, SC
WHERE C.Cname = '数据库' AND C.C# = SC.C#;
```

### 2. 分组查询与过滤（GROUP BY + HAVING）

#### 2.1 GROUP BY 子句

**语法**：
```sql
SELECT 列名 | expr | agfunc(列名) [[, ...]]
FROM 表名 [, 表名 ...]
[WHERE 检索条件]
[GROUP BY 分组条件];
```

- 分组条件可以是一个或多个列名
- GROUP BY 将检索到的元组按照某一（或多个）条件进行分类，具有相同条件值的元组划到一个组中，然后对每个组执行聚集运算

```sql
-- 求每一个学生的平均成绩（按学号分组）
SELECT S#, AVG(Score) FROM SC GROUP BY S#;

-- 求每一门课程的平均成绩（按课号分组）
SELECT C#, AVG(Score) FROM SC GROUP BY C#;
```

#### 2.2 HAVING 子句（分组过滤）

**语法**：
```sql
SELECT 列名 | expr | agfunc(列名) [[, ...]]
FROM 表名 [, 表名 ...]
[WHERE 检索条件]
[GROUP BY 分组条件 [HAVING 分组过滤条件]];
```

- `HAVING` 子句需要有 `GROUP BY` 子句支持（没有 `GROUP BY`，不能有 `HAVING`）
- `HAVING` 用于对**分组**（集合）进行条件过滤，满足条件的分组留下，不满足的剔除

```sql
-- 求不及格课程超过两门的同学的学号
SELECT S# FROM SC
WHERE Score < 60
GROUP BY S# HAVING COUNT(*) > 2;

-- 求有10人以上不及格的课程号
SELECT C# FROM SC
WHERE Score < 60
GROUP BY C# HAVING COUNT(*) > 10;
```

#### 2.3 HAVING vs WHERE 的区别（重点对比表）

| 比较维度 | WHERE | HAVING |
|----------|-------|--------|
| **作用对象** | 作用于关系中的**每一个元组（行）** | 作用于**每一个分组** |
| **执行时机** | 在 GROUP BY 之前执行，过滤元组 | 在 GROUP BY 之后执行，过滤分组 |
| **能否使用聚合函数** | **不能**使用聚合函数（WHERE 是对每行检查） | **可以**使用聚合函数 |
| **是否需要 GROUP BY** | 不需要 GROUP BY | **必须有** GROUP BY 子句才能使用 |
| **语法位置** | GROUP BY 之前 | GROUP BY 之后 |

> 口诀：WHERE 管行，HAVING 管组；WHERE 不能有聚合函数，HAVING 可以有。

### 3. 嵌套查询（子查询）

#### 3.1 子查询的概念

- **子查询（Subquery）**：出现在 `WHERE` 子句中的 `SELECT` 语句，子查询返回一个集合，通过与这个集合的比较来确定另一个查询的集合。
- 带有子查询的 `SELECT` 语句分为**外层查询**和**内层查询（子查询）**。

**三种类型的子查询**：
1. **(NOT) IN 子查询**：集合成员资格判断，某一元素是否是某一个集合的成员
2. **θ-some / θ-all 子查询**：集合之间的比较，某一个集合是否包含另一个集合
3. **(NOT) EXISTS 子查询**：集合基数的测试，测试集合是否为空

#### 3.2 (NOT) IN 子查询

**语法**：
```sql
表达式 [NOT] IN (子查询)
```

**语义**：判断某一表达式的值是否在子查询的结果中。

```sql
-- 列出张三、王三同学的所有信息
SELECT * FROM Student WHERE Sname IN ('张三', '王三');
-- 等价于：WHERE Sname = '张三' OR Sname = '王三'

-- 列出选修了001号课程的学生的学号和姓名
SELECT S#, Sname FROM Student
WHERE S# IN (SELECT S# FROM SC WHERE C# = '001');

-- 求既学过001号课程，又学过002号课程的学生的学号
SELECT S# FROM SC WHERE C# = '001'
AND S# IN (SELECT S# FROM SC WHERE C# = '002');

-- 列出没学过李明老师讲授课程的所有同学的姓名
SELECT Sname FROM Student
WHERE S# NOT IN (
    SELECT S# FROM SC, Course C, Teacher T
    WHERE T.Tname = '李明' AND SC.C# = C.C# AND T.T# = C.T#
);
```

> 注意：在非相关子查询中，`NOT IN` 和多表连接的效果不同！不能用 `WHERE T.Tname <> '李明'` 的多表连接来代替 `NOT IN` 子查询，因为含义不同。

#### 3.3 相关子查询 vs 非相关子查询

| 比较维度 | 非相关子查询 | 相关子查询 |
|----------|------------|-----------|
| **定义** | 内层查询**独立执行**，不涉及任何外层查询相关信息 | 内层查询**涉及外层查询**的属性值，依赖外层查询 |
| **执行方式** | 内层查询只执行**一次**，其结果用于外层查询 | 内层查询对外层查询**每一个元组都执行一次** |
| **典型形式** | 简单的 IN 子查询 | EXISTS 子查询；子查询中引用了外层表的属性 |
| **示例** | `WHERE S# IN (SELECT S# FROM SC WHERE C# = '001')` | `WHERE EXISTS (SELECT * FROM SC WHERE S# = Student.S#)` |

#### 3.4 θ-some / θ-all 子查询

**语法**：
```sql
表达式 θ SOME (子查询)   -- θ 是比较运算符：<, >, >=, <=, =, <>
表达式 θ ALL (子查询)
```

**语义**：
- `表达式 θ SOME (子查询)`：表达式的值与子查询结果中**至少一个**值满足 θ 关系，则为真
- `表达式 θ ALL (子查询)`：表达式的值与子查询结果中**所有**值都满足 θ 关系，则为真

```sql
-- 找出工资最低的教师姓名
SELECT Tname FROM Teacher
WHERE Salary <= ALL (SELECT Salary FROM Teacher);

-- 找出001号课成绩不是最高的所有学生的学号
SELECT S# FROM SC WHERE C# = '001'
AND Score < SOME (SELECT Score FROM SC WHERE C# = '001');

-- 找出001号课成绩最高的所有学生的学号
SELECT S# FROM SC WHERE C# = '001'
AND Score >= ALL (SELECT Score FROM SC WHERE C# = '001');

-- 找出98030101号同学成绩最低的课程号
SELECT C# FROM SC WHERE S# = '98030101'
AND Score <= ALL (SELECT Score FROM SC WHERE S# = '98030101');
```

**等价关系**：
- `表达式 = SOME (子查询)` 等价于 `表达式 IN (子查询)`
- **注意**：`<> SOME` 不等价于 `NOT IN`！（`<> SOME` 是"不等于某个值"，而 `NOT IN` 是"不等于所有值"）

#### 3.5 (NOT) EXISTS 子查询

**语法**：
```sql
[NOT] EXISTS (子查询)
```

**语义**：子查询结果中是否有元组存在（非空则为真）。

```sql
-- 检索选修了赵三老师主讲课程的所有同学的姓名（EXISTS 形式）
SELECT DISTINCT Sname FROM Student
WHERE EXISTS (
    SELECT * FROM SC, Course, Teacher
    WHERE SC.C# = Course.C# AND SC.S# = Student.S#
    AND Course.T# = Teacher.T# AND Tname = '赵三'
);

-- 检索学过001号教师主讲的所有课程的所有同学的姓名（NOT EXISTS 双重否定）
SELECT Sname FROM Student
WHERE NOT EXISTS (          -- 不存在这样一门课：
    SELECT * FROM Course
    WHERE Course.T# = '001' AND NOT EXISTS (  -- 该同学没有选过它
        SELECT * FROM SC
        WHERE S# = Student.S# AND C# = Course.C#
    )
);
```

> NOT EXISTS 的双重否定逻辑："不存在有一门001号教师主讲的课程该同学没有学过"= "该同学学过了001号教师主讲的所有课程"。

### 4. 集合操作

#### 4.1 三种集合运算

```sql
子查询 {UNION [ALL] | INTERSECT [ALL] | EXCEPT [ALL]} 子查询
```

| 运算 | 含义 | 等价的 SQL 写法 |
|------|------|----------------|
| `UNION` | 并运算，自动去重 | `OR` 或 `IN` |
| `UNION ALL` | 并运算，保留重复元组 | — |
| `INTERSECT` | 交运算，自动去重 | `AND + IN 子查询` |
| `EXCEPT` | 差运算，自动去重 | `NOT IN 子查询` 或 `NOT EXISTS` |

> 通常情况下自动删除重复元组（不带 ALL）；若要保留重复的元组，则带 ALL。

```sql
-- 求学过002号课的同学或学过003号课的同学学号（并运算）
SELECT S# FROM SC WHERE C# = '002'
UNION
SELECT S# FROM SC WHERE C# = '003';
-- 等价于：SELECT S# FROM SC WHERE C# = '002' OR C# = '003';

-- 求既学过002号课，又学过003号课的同学学号（交运算）
SELECT S# FROM SC WHERE C# = '002'
INTERSECT
SELECT S# FROM SC WHERE C# = '003';
-- 等价于：SELECT S# FROM SC WHERE C# = '002'
--         AND S# IN (SELECT S# FROM SC WHERE C# = '003');

-- 求没学过002号课程的学生学号（差运算，假定所有学生都有选课）
SELECT DISTINCT S# FROM SC
EXCEPT
SELECT S# FROM SC WHERE C# = '002';
```

### 5. 连接查询（JOIN）

#### 5.1 连接查询语法

```sql
SELECT 列名 [[, 列名] ...]
FROM 表名1 [NATURAL]
    [INNER | {LEFT | RIGHT | FULL} [OUTER]] JOIN 表名2
    {ON 连接条件 | USING (列名 [, 列名 ...])}
[WHERE 检索条件];
```

#### 5.2 四种连接类型对比

| 连接类型 | 关键字 | 结果说明 | 示例场景 |
|----------|--------|---------|---------|
| **内连接** | `INNER JOIN` | 只返回两表中**满足连接条件**的匹配行；不满足条件的行被丢弃 | 查询有任课记录的教师情况 |
| **左外连接** | `LEFT [OUTER] JOIN` | 返回左表的**所有行** + 右表中满足条件的匹配行；右表无匹配则填 NULL | 查询所有教师的任课情况（包括没有任课的教师） |
| **右外连接** | `RIGHT [OUTER] JOIN` | 返回右表的**所有行** + 左表中满足条件的匹配行；左表无匹配则填 NULL | 查询所有课程的教师情况（包括无教师的课程） |
| **全外连接** | `FULL [OUTER] JOIN` | 返回两表中**所有行**，无匹配的一侧填 NULL | 查询所有教师和所有课程的对应情况 |

```sql
-- 求所有教师的任课情况（有任课记录的教师）——内连接
SELECT Teacher.T#, Tname, Cname
FROM Teacher INNER JOIN Course
ON Teacher.T# = Course.T#;

-- 求所有教师的任课情况（没有任课的教师也需列在表中）——左外连接
SELECT Teacher.T#, Tname, Cname
FROM Teacher LEFT OUTER JOIN Course
ON Teacher.T# = Course.T#;
```

> 关键区别：内连接只保留匹配的行；左外连接保留左表全部行，右表不匹配的填 NULL；右外连接同理；全外连接两侧都保留全部行。

### 6. 视图（View）

#### 6.1 视图的概念

**视图（View）**：
- 是**虚表**，是从一个或几个**基本表**（或视图）导出的虚拟表
- **只存放视图的定义**，不会出现数据冗余（不存储数据本身）
- 基表中的数据发生变化，从视图中查询出的数据也**随之改变**
- 对应三级模式中的**外模式**，视图定义本身包含了**外模式—概念模式映像（E-C 映像）**

#### 6.2 创建视图（CREATE VIEW）

**语法**：
```sql
CREATE VIEW 视图名 [(列名 [, 列名] ...)]
AS 子查询
[WITH CHECK OPTION];
```

- 如果视图的属性名缺省，则默认为子查询结果中的属性名
- `WITH CHECK OPTION`：当对视图进行 INSERT、UPDATE、DELETE 时，要检查操作的元组是否满足视图定义中子查询的条件表达式

```sql
-- 定义视图 CompStud：计算机系的学生（屏蔽其他系）
CREATE VIEW CompStud AS
(SELECT * FROM Student
 WHERE D# IN (SELECT D# FROM Dept WHERE Dname = '计算机'));

-- 定义视图 Teach：教师任课情况（屏蔽工资等隐私信息）
CREATE VIEW Teach AS
(SELECT T.Tname, C.Cname, Credit
 FROM Teacher T, Course C
 WHERE T.T# = C.T#);

-- 定义分组视图 StudStat（含统计信息）
CREATE VIEW StudStat(S#, Sname, AvgS, MinS, MaxS, CNT) AS
(SELECT S#, Sname, AVG(Score), MIN(Score), MAX(Score), COUNT(*)
 FROM Student S, SC WHERE S.S# = SC.S#
 GROUP BY S.S#);
```

#### 6.3 查询视图

定义好的视图可以**像基本表一样**在 SQL 各种语句中使用：

```sql
-- 检索主讲数据库课程的教师姓名（使用视图 Teach）
SELECT T.Tname FROM Teach T WHERE T.Cname = '数据库';

-- 检索计算机系的所有学生（使用视图 CompStud）
SELECT * FROM CompStud;

-- 检索计算机系年龄小于20的学生（使用视图 CompStud）
SELECT * FROM CompStud WHERE Sage < 20;

-- 基于视图 StudStat 检索某学生平均成绩
SELECT Sname, AvgS FROM StudStat WHERE Sname = '张三';
```

#### 6.4 更新视图（重点：可更新性规则）

视图更新的本质：对视图的更新最终要反映到对基本表的更新上。但有时视图定义的映射不是可逆的，导致无法更新。

**不可更新的视图类型**（课件原话，五条规则）：

| 规则 | 说明 |
|------|------|
| 1 | 如果视图的 SELECT 目标列**包含聚集函数**，则不能更新 |
| 2 | 如果视图的 SELECT 子句使用了 **UNIQUE 或 DISTINCT**，则不能更新 |
| 3 | 如果视图中包括了 **GROUP BY 子句**，则不能更新 |
| 4 | 如果视图中包括经**算术表达式计算出来的列**，则不能更新 |
| 5 | 如果视图是由单个表的列构成，但**并没有包括主键**，则不能更新 |

**可更新的视图**：
- 由**单一基本表**使用选择、投影操作导出的视图
- 并且**包含了基本表的主键**

```sql
-- 不可更新的视图示例（含 GROUP BY 和聚合函数）
CREATE VIEW S_G(S#, Savg) AS
(SELECT S#, AVG(Score) FROM SC GROUP BY S#);
-- 下面的更新无法执行：
UPDATE S_G SET Savg = 85 WHERE S# = '98030101';
-- 原因：Savg 是聚合结果，无法反向更新 SC 表

-- 不可更新的视图示例（缺少主键）
CREATE VIEW ClassStud(Sname, Sclass) AS
(SELECT Sname, Sclass FROM Student);
-- 下面的插入无法执行：
INSERT INTO ClassStud VALUES ('张三', '980301');
-- 原因：缺少 Student 的主键 S#

-- 可更新的视图示例（含主键，单表选择+投影）
CREATE VIEW CStud(S#, Sname, Sclass) AS
(SELECT S#, Sname, Sclass FROM Student WHERE D# = '03');
-- 可以执行：
INSERT INTO CStud VALUES ('98030104', '张三丰', '980301');
-- 转换为：INSERT INTO Student VALUES ('98030104', '张三丰', NULL, NULL, '03', '980301');
```

#### 6.5 删除视图（DROP VIEW）

```sql
DROP VIEW 视图名;

-- 示例
DROP VIEW Teach;
DROP VIEW CompStud;
```

#### 6.6 视图的四大作用（简答常考）

1. **视图能够简化用户的操作**：将复杂的查询定义为视图，用户只需对视图进行简单查询，避免重复书写复杂语句。

2. **视图使用户能以多种角度看待同一数据（定制数据）**：不同用户可以通过不同的视图看到不同角度的数据，满足多样化需求。

3. **视图对重构数据库提供了一定程度的逻辑独立性**：若数据库逻辑结构发生改变（如一个表拆成两个表），可通过建立视图使外模式保持不变，从而对原表的查询程序不必修改。例：
   ```sql
   -- Student 拆成 SX(Sno,Sname,Sage) 和 SY(Sno,Ssex,Sdept) 后
   -- 建立视图恢复原有 Student 的外观：
   CREATE VIEW Student(Sno, Sname, Ssex, Sage, Sdept) AS
   SELECT SX.Sno, SX.Sname, SY.Ssex, SX.Sage, SY.Sdept
   FROM SX, SY WHERE SX.Sno = SY.Sno;
   ```

4. **视图能够对机密数据提供安全保护**：通过视图，用户只能看到和修改被授权访问的数据，其他数据不可见，从而实现安全控制。

### 7. SQL 查询的执行流程与查询优化

#### 7.1 SQL 查询执行流程

一条 SQL 语句的执行流程（三个层次）：
1. **语义检查**：检查 SQL 语句的语义是否正确（表名、列名是否存在，权限是否足够等）
2. **语法检查**：检查 SQL 语句的语法是否符合 SQL 语法规则
3. **执行**：将 SQL 语句编译成可执行的内部操作（关系代数表达式）并执行

#### 7.2 为什么需要查询优化

关系代数操作的执行次序对效率影响很大。即使表达相同的检索需求，不同的操作顺序（如选择先做还是连接先做）会导致处理的数据量差异极大。

#### 7.3 查询优化三个层面

| 优化层面 | 名称 | 内容 |
|----------|------|------|
| 第一层 | **语义优化** | 利用模型的语义及完整性规则，优化查询（如消除冗余条件） |
| 第二层 | **语法优化（逻辑层优化）** | 利用语法结构，优化操作执行顺序（如尽早执行选择和投影操作，减少中间结果） |
| 第三层 | **执行优化（物理层优化）** | 存取路径和执行算法的选择与执行次序优化（如选择索引扫描还是全表扫描） |

### 8. SELECT 语句完整格式

```sql
SELECT [ALL | DISTINCT] <目标列表达式> [别名] ...
FROM <表名或视图名> [别名] [, <表名或视图名> [别名]] ...
[WHERE <条件表达式>]
[GROUP BY <列名1> [HAVING <条件表达式>]]
[ORDER BY <列名2> [ASC | DESC]];
```

**子句执行顺序**（逻辑上）：
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

## ③ 名词解释清单

| 术语 | 完整定义 |
|------|----------|
| **聚合函数** | 作用在一组值（列值集合）上、返回单一值的内置函数，包括 COUNT、SUM、AVG、MAX、MIN，除 COUNT(*) 外均忽略 NULL 值 |
| **子查询（嵌套查询）** | 出现在 WHERE 子句中的 SELECT 语句，返回一个集合，用于外层查询的比较条件中 |
| **非相关子查询** | 内层查询独立执行，不涉及任何外层查询相关信息的子查询，内层只执行一次 |
| **相关子查询** | 内层查询涉及外层查询的属性值，依赖外层，内层对外层每一个元组都要执行一次 |
| **视图（View）** | 从一个或几个基本表（或视图）导出的虚拟表，只存放视图的定义不存储数据；对应三级模式中的外模式 |
| **外连接（OUTER JOIN）** | 包含 LEFT/RIGHT/FULL OUTER JOIN，在内连接结果之外，还包括一侧或两侧表中不满足连接条件的行，用 NULL 填充对侧 |
| **内连接（INNER JOIN）** | 只返回两表中满足连接条件的匹配行，不满足条件的行被丢弃 |
| **UNION / INTERSECT / EXCEPT** | SQL 的集合运算符，分别对应并集、交集、差集，默认去除重复元组，加 ALL 保留重复 |
| **GROUP BY** | 分组子句，将查询结果按指定列分组，具有相同列值的元组划为一组，常配合聚合函数使用 |
| **HAVING** | 分组过滤子句，用于对 GROUP BY 产生的分组进行条件过滤，可使用聚合函数，必须与 GROUP BY 同时使用 |

## ④ 简答 / 写 SQL 题预测

### 【简答1】WHERE 和 HAVING 有什么区别？

答：
1. **作用对象不同**：WHERE 对**每一个元组（行）**进行条件过滤；HAVING 对每一个**分组**进行条件过滤。
2. **执行时机不同**：WHERE 在 GROUP BY 之前执行，先过滤元组再分组；HAVING 在 GROUP BY 之后执行，先分组再过滤。
3. **能否使用聚合函数**：WHERE 子句中**不能**使用聚合函数（因为聚合函数是对集合操作，WHERE 是对单行操作）；HAVING 子句中**可以**使用聚合函数。
4. **使用前提不同**：WHERE 可以单独使用；HAVING 必须有 GROUP BY 子句才能使用。

### 【简答2】视图有哪些作用？

答：视图的四大作用：
1. **简化用户操作**：将复杂查询定义为视图，用户后续只需简单查询视图即可，减少重复操作。
2. **支持多角度看待数据**：不同用户可以通过不同视图看到数据的不同侧面，满足个性化需求。
3. **提供逻辑独立性**：当基本表结构改变时，可以通过修改视图来保持外模式不变，使应用程序不需要修改。
4. **保护机密数据安全**：通过视图只向特定用户展示特定列/行的数据，屏蔽敏感信息，实现安全控制。

### 【简答3】哪些视图不可更新？

答（课件总结5条）：
1. 视图的 SELECT 目标列包含**聚集函数**（如 AVG、SUM）——不能更新
2. 视图的 SELECT 子句使用了 **UNIQUE 或 DISTINCT**——不能更新
3. 视图中包括 **GROUP BY 子句**——不能更新
4. 视图中包括经**算术表达式**计算出来的列——不能更新
5. 视图由单个表的列构成，但**没有包括主键**——不能更新

可更新的视图：由**单一基本表**用选择、投影操作导出，且**包含了基本表的主键**。

### 【写SQL1】求每门课程不及格人数超过2人的课程号及不及格人数

```sql
SELECT C#, COUNT(*) AS 不及格人数
FROM SC
WHERE Score < 60
GROUP BY C#
HAVING COUNT(*) > 2;
```

### 【写SQL2】找出001号课成绩最高的所有学生的学号

```sql
-- 方法一：ALL 子查询
SELECT S# FROM SC WHERE C# = '001'
AND Score >= ALL (SELECT Score FROM SC WHERE C# = '001');

-- 方法二：NOT EXISTS
SELECT S# FROM SC SC1 WHERE C# = '001'
AND NOT EXISTS (
    SELECT * FROM SC WHERE C# = '001' AND Score > SC1.Score
);
```

### 【写SQL3】求学过001号课程但没有学过002号课程的学生学号

```sql
-- 方法一：NOT IN 子查询
SELECT S# FROM SC WHERE C# = '001'
AND S# NOT IN (SELECT S# FROM SC WHERE C# = '002');

-- 方法二：EXCEPT 集合运算
SELECT S# FROM SC WHERE C# = '001'
EXCEPT
SELECT S# FROM SC WHERE C# = '002';
```

### 【写SQL4】查询每个学生的学号、姓名及其所选课程数量

```sql
SELECT Student.S#, Sname, COUNT(C#) AS 选课数
FROM Student LEFT OUTER JOIN SC ON Student.S# = SC.S#
GROUP BY Student.S#, Sname;
```

### 【写SQL5】创建一个视图，显示所有选修了"数据库"课程且成绩>=90的学生学号和姓名

```sql
CREATE VIEW ExcellentDBStud(S#, Sname) AS
(SELECT Student.S#, Sname
 FROM Student, SC, Course
 WHERE Student.S# = SC.S# AND SC.C# = Course.C#
 AND Cname = '数据库' AND Score >= 90);
```

## ⑤ 易错点提醒

1. **COUNT(\*) 和 COUNT(列名) 的区别**：`COUNT(*)` 计入所有行（包括 NULL）；`COUNT(列名)` 只计入该列非 NULL 的行。
2. **聚合函数不能出现在 WHERE 子句中**：`WHERE AVG(Score) > 80` 是非法的，应改为 `HAVING AVG(Score) > 80`（配合 GROUP BY）。
3. **HAVING 必须有 GROUP BY**：没有 GROUP BY 就不能使用 HAVING。
4. **= SOME 等价于 IN，但 <> SOME 不等价于 NOT IN**：`<> SOME` 的意思是"与集合中某个值不等"（很容易为真）；`NOT IN` 的意思是"与集合中所有值都不等"。
5. **NOT EXISTS 的双重否定**：`NOT EXISTS (... NOT EXISTS ...)` 表达全称量词的逻辑，读法是"不存在 [某课程] 使得 [该同学没学过它]" = "该同学学过[该教师的]所有课程"。
6. **LEFT JOIN 保留左表全部行，RIGHT JOIN 保留右表全部行**，不匹配的一侧用 NULL 填充。
7. **视图不存储数据**：视图只存放定义（查询语句），查询视图时实际上是执行视图背后的查询。基表数据变化，视图查询结果也自动跟着变化。
8. **DROP VIEW 只删除视图定义，不影响基本表数据**。
9. **SELECT 语句执行顺序**（逻辑顺序）：FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY。不能在 WHERE 中引用 SELECT 里定义的别名（因为 WHERE 先于 SELECT 执行）。
10. **集合运算（UNION/INTERSECT/EXCEPT）默认去重**：要保留重复元组需加 `ALL`。

## 第四讲 补充：SELECT 完整语句执行案例分析

```sql
-- 题目：求计算机系学生中，至少选了3门课且平均成绩大于80分的学生学号和平均成绩
-- 按平均成绩降序排列

SELECT SC.S#, AVG(Score) AS 平均成绩
FROM Student, SC
WHERE Student.S# = SC.S#
  AND D# IN (SELECT D# FROM Dept WHERE Dname = '计算机')
GROUP BY SC.S#
HAVING COUNT(*) >= 3 AND AVG(Score) > 80
ORDER BY 平均成绩 DESC;
```

分析：
- `WHERE` 先过滤计算机系的学生（行级过滤）
- `GROUP BY SC.S#` 按学号分组
- `HAVING COUNT(*) >= 3 AND AVG(Score) > 80` 对分组过滤（至少3门课且均分>80）
- `ORDER BY` 对结果排序
