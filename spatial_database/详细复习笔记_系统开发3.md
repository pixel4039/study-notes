### 第2节 C++ 与 PostGIS 实现"附近餐厅推荐系统"（完整案例，重点）

这是本讲第一个、也是最完整的案例，串起了"空间数据库设计 + 空间索引 + 空间查询 + 应用编程"。考试可能考其中的**空间数据库设计要点**和**关键 PostGIS 函数**。

#### 2.1 功能需求（4 类）
1. **餐厅数据管理**：从 CSV 导入餐厅数据（名称、地址、菜系、评分、价格等级、经纬度）；支持增、删、改。
2. **用户位置管理**：记录并更新用户当前位置；保存用户搜索偏好（如默认搜索半径）。
3. **餐厅推荐**：基于用户当前位置推荐附近餐厅；支持按距离、评分、价格等级、菜系类型筛选；结果按距离排序。
4. **性能需求**：使用**空间索引**加速查询；支持高并发查询。

#### 2.2 数据库设计（★必看，空间建表范例）
```sql
CREATE TABLE restaurants (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    address VARCHAR(200),
    cuisine_type VARCHAR(50),
    rating FLOAT CHECK (rating >= 0 AND rating <= 5),
    price_level INTEGER CHECK (price_level >= 1 AND price_level <= 5),
    geom GEOMETRY(Point, 4326) NOT NULL          -- 空间字段：点几何，SRID=4326(WGS84)
);
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    last_location GEOMETRY(Point, 4326),
    last_search_radius INTEGER DEFAULT 1000
);
-- 为两张表的几何字段建 GiST 空间索引（加速空间查询的关键）
CREATE INDEX restaurants_geom_idx ON restaurants USING GIST(geom);
CREATE INDEX users_location_idx  ON users USING GIST(last_location);
```
**考点**：① 空间字段类型写法 `GEOMETRY(Point, 4326)`；② CHECK 约束做值域控制；③ 空间索引固定语法 `USING GIST(几何列)`。

#### 2.3 代码结构（三大核心模块）
项目目录：`restaurant_recommender/`（含 `CMakeLists.txt`、`include/`、`src/`、`data/restaurants.csv`）。系统分三个模块：

| 模块 | 职责 |
|------|------|
| **GeoUtils** | 处理地理空间数据的转换和计算：① WKT 格式与内部结构互转；② 用 **Haversine 公式**算球面距离 |
| **RestaurantRecommender** | 核心业务逻辑；**封装所有数据库操作**；提供餐厅管理和推荐功能接口 |
| **Main** | 系统入口；演示功能；处理用户界面展示 |

**数据流**：`CSV文件 → RestaurantRecommender → PostgreSQL/PostGIS → 查询结果 → 格式化输出`（用户位置/偏好设置作为输入注入到 Recommender）。

#### 2.4 GeoUtils 模块（几何工具）
定义 `struct GeoPoint { double longitude; double latitude; }`，命名空间 `GeoUtils` 提供三个函数：

| 函数 | 作用 |
|------|------|
| `pointToWKT(point)` | 将 GeoPoint 转为 **WKT 字符串**，示例 `POINT(116.404 39.915)`，用于 PostGIS 几何存储 |
| `wktToPoint(wkt)` | 反向转换，从 WKT 提取坐标点，处理查询结果中的几何数据 |
| `calculateDistance(p1,p2)` | 用 **Haversine 公式**算球面距离，考虑地球曲率，比平面距离更精确 |

**名词**：**WKT（Well-Known Text）**——OGC 定义的几何对象文本表示法，如 `POINT(x y)`。**Haversine 公式**——由两点经纬度计算球面大圆距离的公式。

#### 2.5 RestaurantRecommender 类（核心）
- `struct Restaurant` 字段：id、name、address、cuisineType、rating、priceLevel、`GeoPoint location`、`double distance`（与用户的距离/米）。
- 类的关键接口：
  - `initializeDatabase()`：数据库初始化——建餐厅和用户表、建**空间索引(GIST)**。
  - `importRestaurantsFromCSV(csvPath)`：逐行解析 CSV，**用参数化查询插入数据**（防注入）。
  - `addRestaurant()` / `deleteRestaurant(id)`：餐厅增删。
  - `updateUserLocation()` / `updateUserSearchRadius()`：用户位置管理。
  - **`findNearbyRestaurants(username, radius=1000, cuisineFilter="", minRating=0.0f, maxPriceLevel=5, limit=10)`**：核心推荐算法，**使用 PostGIS 的 `ST_DWithin` 进行距离筛选**。
  - 私有：`std::shared_ptr<pqxx::connection> conn_`（用 **pqxx/libpqxx** 连接库）、`executeSQL()`、`resultToRestaurant(row)`。
- **考点**：核心推荐用 **`ST_DWithin(geom, 用户点, 半径)`** 做"范围内"筛选；C++ 访问 PostgreSQL 用 **libpqxx（pqxx）** 库。

#### 2.6 main（结果展示）
`printRestaurants()` 用 `std::setw` 对齐输出表头（名称/菜系/评分/价格等级/距离(米)）并按行打印；空结果打印"没有找到符合条件的餐厅"。属于演示性代码，了解即可。

#### 2.7 main 主流程与推荐调用（体现空间查询用法）
```cpp
RestaurantRecommender recommender("dbname=restaurant_recommendation user=postgres password=...");
recommender.initializeDatabase();
recommender.importRestaurantsFromCSV("data/restaurants.csv");
recommender.updateUserLocation("john", {116.404, 39.915});          // 更新用户位置(北京王府井附近)

auto nearby   = recommender.findNearbyRestaurants("john", 500);                 // 例1: 附近500米内所有餐厅
auto chinese  = recommender.findNearbyRestaurants("john", 1000, "Chinese");     // 例2: 附近1000米内中餐厅
auto highRated= recommender.findNearbyRestaurants("john", 1500, "", 4.0f);      // 例3: 附近1500米内评分≥4.0
```
外层用 `try...catch(std::exception&)` 捕获数据库异常。**考点**：同一个 `findNearbyRestaurants` 通过改变半径/菜系/最低评分参数，就能表达"范围+属性"的组合空间查询——底层就是 `ST_DWithin` + `WHERE` 属性条件 + `ORDER BY 距离`。

### 第3节 深大校园全要素空间数据库设计实例（08，重点案例）

这是本讲第二个大案例，直接对应"空间数据库设计"的完整流程，**很可能作为论述/设计题考**。

#### 3.1 项目目标与设计要求
**深大校园全要素管理空间数据库设计与实现**，两条核心要求：
1. **参照数据库设计的标准流程**：需求分析 → 概念设计 → 逻辑设计 → 物理设计等；数据库结构及其关系定义清晰，完整性约束定义明确，形成内容完整的**数据库设计文档**。
2. **要素全覆盖**：要能管理深大校园**地上、地表、地下**所有要素数据，实现要素全覆盖；**要素分类编码可在现有国家地理信息分类编码标准基础上修改完善**。

#### 3.2 "全要素"的含义（地上—地表—地下三层）
课件用剖面图强调"全要素"=垂直方向全覆盖：
- **地上**：建筑、地形（DEM/DSM）等；
- **地表**：道路、地表建筑、绿地、水体等；
- **地下**：地下商场、地下停车场、地铁、人防工程、地下管线等。
**考点**："全要素"不仅是平面上要素类型齐全，更强调**三维/地上地下一体化**管理。

#### 3.3 数据基础（多源空间数据）
课件展示了该实例用到的多种数据形式：**电子地图（POI/兴趣点）**、**全要素矢量数据（点/线/面分层，带拓扑）**、**高分辨率遥感影像（正射影像）**、**DEM 数字高程模型**、**三维实景模型**等——体现空间数据库要整合矢量、栅格、三维等多源数据。

课件用大量成果图展示"全要素"数据形态（了解即可）：**倾斜摄影三维模型**、**航空/卫星正射影像**、**DEM 地形**、**数字城市三维平台**、**室内三维激光点云与三维重建**、**BIM 室内模型**、**地下管线三维模型**、**地质分层三维模型**——覆盖"地上建筑—地表—地下管线/地质"的地上地下一体化三维数据。

#### 3.4 全要素数据包括哪些数据内容？（★简答高频，P25）
**基础时空数据**内容至少包括：**矢量数据、影像数据、高程模型数据、地理实体数据、地名地址数据、三维模型数据、新型测绘产品数据及其元数据**（8 类，可直接背）。逐类要求：

| 数据类型 | 内容与精度要求（课件原话要点） |
|---|---|
| 矢量数据 | 1:500、1:1000、1:2000 大比例尺地形图至少覆盖规划区范围；1:5000 或 1:10000 应覆盖市辖范围 |
| 影像数据 | 0.1 米或 0.2 米影像至少覆盖规划区范围；0.5 米影像应覆盖市辖范围 |
| 高程模型数据 | 0.5 米、1 米格网至少覆盖规划区范围；2 米、5 米格网应覆盖市辖范围 |
| 地理实体数据 | 以地形图为基础，对**境界、政区、道路、水系、院落、建筑物、植被**等内容进行**实体化**，并**赋予唯一编码**，作为与其它行业和专题数据进行**关联的基础** |
| 地名地址数据 | 扩充自然村以上的行政地名，建立**市（地区/自治州/盟）级、县（区/县级市）级、乡（镇/街道）级和行政村（社区）级**四级区划单元，实现市辖范围精细化地名地址全覆盖 |
| 三维模型数据 | 至少分等级实现市辖范围全覆盖：**一级模型**——政治/经济/文化/交通/旅游等地标性中心区、CBD 及特定区域；**二级模型**——除上述外的各类中心区域、高档住宅、公寓；**三级模型**——其他中心区域、普通住宅；**四级模型**——城中村、棚户区、工厂厂房区、远郊、农村地区 |
| 新型测绘产品数据 | 宜涵盖**全景及可量测实景影像、倾斜影像、激光点云数据、室内地图数据、地下空间数据、建筑信息模型（BIM）**数据等 |

深大校园全要素数据

#### 3.6 基础地理要素的分层与几何类型设计（P27，逻辑设计核心，★重点）
每类要素落到具体图层时明确**几何类型（点/线/面）**：

| 要素大类 | 图层（几何类型） |
|---|---|
| 水系 | 湖泊（面） |
| 居民地 | 房屋（面） |
| 道路 | 道路中心线（线）、道路面（面） |
| 植被 | 草地（面）、林地（面）、行道树（点） |
| 边界 | 校园边界（面） |
| 地名地址 | 地名地址（点） |
| 地貌与土质 | 等高线（线）、土质（面） |
| 定位基础 | 测量控制点（点） |
| 管线（6种） | 电力、信息与通信、给水、排水、燃气、热力——每种均分**管点（点）+ 管线段（线）** |
| 部件设施—公共设施类 | 井盖、路灯、电力设施、通信设施、体育设施、治安岗亭、休闲设施、监控摄像头（均为点） |
| 部件设施—交通设施类 | 停车场（面）、交通标牌（点）、交通信号灯（点）、小巴站点（点） |
| 部件设施—市容环境类 | 垃圾箱（点）、垃圾间（面） |

**考点**：①管线要素统一采用"**管点+管线段**"二元建模；②同一现实要素可拆多图层（道路=中心线+面）；③部件设施按"公共设施/交通设施/市容环境"三类组织。

#### 3.7 栅格/影像数据的组织：金字塔 + 分块（P28–30，★名词解释）
- 三维地形模型数据（DOM/TDOM、DEM、DSM、倾斜摄影真三维实景模型）中的栅格数据按 **"金字塔 + 均匀分块、边缘补零"** 组织：**对原始影像构建分辨率递减的金字塔结构（Pyramid Level 0/1/2…），并在每一级金字塔上对栅格数据进行分块**。
- 课件以 **Oracle GeoRaster** 存储模型为例说明落库方式：`CITY_IMAGES` 表每行存一个 **SDO_GEORASTER 对象**（含 GeoRaster 类型、影像空间范围 SDO_GEOMETRY、栅格数据表名、Raster ID、元数据 XMLType）；实际像素按块存入 **Raster data table**，每块一行 **SDO_RASTER**（含 Raster ID、金字塔层级、该块 MBR、该块影像数据 BLOB）。
- UML 关系（P30）：地形数据集 ↔ 空间参考集；地形数据集 1—* DEM/DOM 数据集；DEM/DOM 数据集 1—* 金字塔层数据集；金字塔层数据集 1—* 数据块。
- **考点串联**：这正是第七讲"栅格数据在 DBMS 中分块存储（BLOB）"与第九讲"空间索引/MBR"的落地应用。

#### 3.8 三维要素模型的组织（P31）
- 三维要素模型共 10 类：建筑、交通、水系、地质体、植被、场地、管线、地下空间设施、城市部件、其他。
- 建筑要素按 **LOD（细节层次）多级表达**：从体块（LOD1）→ 外立面细化 → 带门窗 → 室内结构（示意图为一栋房屋四级递进）。
- UML 结构：**数据集** 分为语义拓扑数据集、三维模型数据集、空间参考等；三维模型数据集下分**三维要素类**与**对象类**（经**关联类**连接）；三维要素类聚合**三维空间实体**（对应**三维几何模型(LOD)** 与**属性数据**），非空间对象只挂属性；三维几何模型关联**共享模型、材质（简单/复杂）、纹理（多级纹理）**。
- **考点**：三维数据模型 = **语义（要素类/对象类）+ 几何（LOD 多级）+ 表观（材质/纹理）** 三部分组织，与第十二讲"三维数据模型三层结构（语义/尺度/几何）"呼应。

#### P33
**深大校园全要素空间数据库设计实例 —— 实体之间关系怎么构建（管线举例）**

- 举例：一个天然气设备连接一条或多条天然气管线。
- 通过设施图片（阀门、消防栓、管道）分别映射为空间几何要素类：
  - `GeoPoint`：`int m_FetureID`, `Point3f m_Pos`（点状要素，如阀门、消火栓）
  - `Mesh`：`int m_FetureID`, `Point3f m_StartPos`, `Point3f m_EndPos`, `float m_Diameter`（管状要素）
- 类图（E-R/UML 关系）：
  - `_PipelineObject`（管线对象）属性：PipelineID、PipelineName、PipelineType、TotalPipelinePoints、TotalPipelineSegments、PipelineLength、BuriedType（埋设方式）、FlowDirection（流向）、ProjectType

**深大校园全要素空间数据库设计实例 —— 如何管理时空数据？**

- **静态GIS**：静态GIS大多不具备处理数据的时间动态性，只是描述时间的瞬时状态。数据发生变化时，新数据将代替旧数据，即成了另一个瞬时状态，旧数据将会消失，无法对数据的更新变化进行分析，更不能预测未来的趋势。
- **时态GIS**：通过建立合适的时间与空间联合的数据模型，将空间要素随时间的变化建模，将地理事物和现象的时间、空间和属性特征有机结合，丰富了地理事物和现象的表达内容。

**1.5 深大校园全要素空间数据库设计实例 —— 时空数据模型三种（图示对比）**

|---|---|---|
| **时空立方体模型** | 用二维图形沿时间第三维发展变化的过程表达现实世界平面位置随时间的演变。给定一个时间位置值，就可以从三维立方体中获得相应界面、立方体的状态。也可以扩展表达三维空间沿着时间变化的过程。 | 直观表达位置随时间演变 |

#### P42
**阿里数据库架构演进分享 —— 基本概念**

- **分布式**：系统中的多个模块在不同服务器上部署，即可成为分布式系统，如Tomcat和数据库分别部署在不同的服务器上，或者两个相同的Tomcat分别部署在不同的服务器上。
- **高可用**：系统中部分节点失效时，其他节点可以接替他继续提供服务，则可认为系统具有高可用性。
- **集群**：一个特定领域的软件部署在多台服务器上并作为一个整体提供一类服务，这个整体称为集群；如Zookeeper中的Master和Slave分别部署在多台服务器上，共同组成一个整体提供几种配置服务。在常见的集群中，客户端往往能链接任意一个节点获得服务，并且当集群中一个节点掉线时，其他节点往往能够接替他继续提供服务，这时候说明集群具有高可用性。
- **负载均衡**：请求发送到系统时通过某些方式把请求均匀的分发到多个节点上，使系统中每个节点能够均匀的承担请求负载，则认为系统是负载均衡的。
- **正向代理和反向代理**：系统内部要访问外部网络时，统一通过一个代理服务器把请求转发出去，在外部网络来看就是代理服务器发起的访问，此时代理服务器实现的是**正向代理**；当外部请求进入系统时，代理服务器把该请求转发到系统中某台服务器上，对外部请求来说，与之交互的只有代理服务器，此时代理服务器实现的是**反向代理**。简单来说，正向代理就是代理服务器代替系统内部来访问外部网络的过程，反向代理就是外部系统请求访问系统时通过代理服务器转发到内部服务器的过程。

#### P46
**4. 第三次演进：引入反向代理实现负载均衡**

- 做法：在多台服务器上分别部署 Tomcat，使用反向代理软件（如 Nginx）把请求均匀分发到每个 Tomcat 中。
- 数值示例：单个 Tomcat 最多支撑 100 并发，Nginx 最多支持 50000 并发，理论上 Nginx 把请求分发到 500 个 Tomcat 就能抗住 50000 并发。
- 涉及技术：**Nginx、HAProxy**——两者都是工作在网络第七层（应用层）的反向代理软件，主要支持 HTTP 协议；还会涉及 session 共享、文件上传下载的问题。
- 架构图：浏览器 → Nginx → 多个 Tomcat（172.168.0.1/0.2/0.3）→（分布式缓存 + 数据库，单份）。
- **遇到的问题（→下一步演进的动因）**：反向代理使应用服务器可支持的并发数大大增加，但并发量的增长也意味着更多的请求穿透到数据库，**单机的数据库成为瓶颈**。

名词解释：
- **反向代理（Reverse Proxy）**：代理服务器代表后端服务器接收客户端请求，并将请求转发到内部服务器，客户端只感知代理、不感知真实后端，同时可实现负载均衡。

#### P47
**5. 第四次演进：数据库读写分离**

- 做法：把数据库划分为**读库**和**写库**，读库可以有多个，通过同步机制把写库的数据同步到读库。
- 针对"需要查询最新写入数据"的场景：可以在缓存中多写一份，通过缓存获取最新数据（避免主从同步延迟导致读不到最新写入）。
- 涉及技术：**Mycat**——数据库中间件，可用来组织数据库的读写分离和分库分表，客户端通过它来访问下层数据库；同时涉及数据库同步、数据一致性问题。
- 架构图：浏览器 → Nginx → 多 Tomcat →（分布式缓存 + 数据库(写) ⇄同步⇄ 数据库(读)）。
- **遇到的问题**：业务逐渐变多，不同业务之间的访问量差距较大，不同业务将直接竞争数据库资源，相互影响性能。

名词解释：
- **Mycat**：一款开源的数据库中间件，位于应用与数据库之间，负责 SQL 解析、路由，实现读写分离、分库分表等功能，对客户端透明。
- **读写分离**：写操作只发往主库（写库），读操作发往从库（读库），通过主从同步保持数据一致，用以分摊数据库压力。

---

#### P48
**6. 第五次演进：数据库按业务分库**

- 做法：把不同业务（如商品业务、支付业务）的数据库拆分保存到不同的数据库实例中，使业务之间的资源竞争降低；对于访问量大的业务，可以单独部署更多服务器来支撑。
- 副作用：跨业务的表无法直接关联分析（join），需要通过其他途径解决（如数据同步到数仓）。
- 架构图：浏览器 → Nginx → 多 Tomcat →（分布式缓存 Redis + 商品业务库(读/写同步) + 支付业务库(读/写同步)，各业务库独立）。
- **遇到的问题**：随着用户的增长，单机的写库会逐渐达到性能瓶颈。

名词解释：
- **分库（垂直拆分/按业务拆分）**：按照业务模块把数据分散到不同数据库实例中，降低单库压力，实现业务间资源隔离。

#### P49
**7. 第六次演进：大表拆分为小表（分表）**

- 做法：
  - 针对评论数据：可按照**商品 ID 进行 hash**，路由到对应的表中存储；
  - 针对支付记录：可按照**小时创建表**，每小时表继续拆分为小表，使用用户 ID 或记录编号来路由数据。
- 原理：只要实时操作的表数据量足够小，且请求能够足够均匀地分发到多台服务器上的小表，数据库就能通过**水平扩展**的方式提高性能。
- Mycat 同样支持大表拆分为小表情况下的访问控制（分表路由）。
- 架构图：在业务库内部，写库进一步按月/hash 拆成多张小表（如支付业务库拆成 1月/2月/…/N月）。
- **遇到的问题**：数据库和 Tomcat 都能够水平扩展，可支撑的并发大幅提高；但随着用户数的增长，**最终单机的 Nginx 会成为瓶颈**。

名词解释：
- **分表（水平拆分）**：把一张大表按某种规则（hash、时间范围等）拆成多张结构相同的小表，分散单表的数据量和访问压力，属于水平扩展手段。
- **路由（Routing）**：根据某个键（商品ID、用户ID、时间等）决定数据应该落在哪个库/表中的过程。

#### P50
**8. 第七次演进：使用 LVS 或 F5 来实现多个 Nginx 的负载均衡**

- 动因：瓶颈转移到了 Nginx，单个 Nginx 无法通过"两层 Nginx"结构来实现多 Nginx 的负载均衡（Nginx 本身也需要被均衡）。
- 做法：在 Nginx 前面再加一层四层负载均衡设备/软件——**LVS 或 F5**，通过**虚拟 IP（VIP）**对外提供服务，LVS/F5 采用主备（主/备）模式保证高可用，再把流量分发到多个 Nginx。
- 架构图：浏览器 → 虚拟IP(10.102.4.1) → LVS/F5(主) + LVS/F5(备) → 多个 Nginx → 多个 Tomcat → 后端存储。
- **遇到的问题**：LVS 本身也是单机的（主备中同一时刻只有一台工作），随着并发数增长到几十万时，LVS 服务器最终也会达到瓶颈；此时用户数达到千万甚至上亿级别，用户分布在不同地区，与服务器机房距离不同，导致访问延迟明显不同（单机房无法就近服务全国/全球用户）。

名词解释：
- **LVS（Linux Virtual Server）**：基于 Linux 内核实现的四层（传输层）负载均衡软件，工作在 IP/传输层，性能高。
- **F5**：商用的硬件负载均衡设备，功能与 LVS 类似，也工作在四层，性能与稳定性更强（收费）。
- **虚拟 IP（VIP）**：对外暴露的一个逻辑 IP 地址，实际流量被转发到后面的一组物理服务器，用于实现高可用与负载均衡。
- **四层 vs 七层负载均衡**：四层（如 LVS/F5）基于 IP+端口做转发，不解析应用层内容，性能更高；七层（如 Nginx/HAProxy）可解析 HTTP 内容做更细粒度的路由（如按 URL、Session）。

#### P51
**9. 第八次演进：通过 DNS 轮询实现机房间的负载均衡**

- 做法：在 DNS 服务器中配置一个域名对应多个 IP 地址，每个 IP 地址对应到不同机房里的虚拟 IP。当用户访问 www.taobao.com 时，DNS 服务器使用**轮询策略**（或其他策略）来选择某个 IP 供用户访问。
- 效果：实现了**机房间**的负载均衡；系统可以做到机房级别的水平扩展，千万级到亿级的并发量都可以通过增加机房来解决，系统入口处的请求并发量不再是问题。
- 架构图：浏览器 ⇄ DNS服务器（记录 www.taobao.com → 10.102.4.1 / 10.102.4.2，对应不同机房虚拟IP）→ 各机房内部仍是 LVS/F5 → Nginx → Tomcat → 存储。
- **遇到的问题**：随着数据的丰富程度和业务的发展，检索、分析等需求越来越丰富，**单单依靠关系数据库无法解决如此丰富的需求**（如全文检索、海量文件、多维分析等）。

名词解释：
- **DNS 轮询（DNS Round Robin）**：DNS 服务器为同一域名配置多条 A 记录（多个 IP），按顺序依次返回给不同的查询请求，从而把用户流量分摊到不同的机房/服务器集群，实现地域级的负载均衡。

#### P52
**10. 第九次演进：引入 NoSQL 数据库和搜索引擎等技术**

- 背景问题：当数据库中的数据多到一定规模时，关系数据库不适用于复杂查询，往往只能满足普通查询场景；统计报表在数据量大时不一定能跑出结果，且跑复杂查询会导致其他查询变慢；对于全文检索、可变数据结构等场景，数据库天生不适用。
- 针对特定场景引入合适的解决方案：

| 场景 | 解决方案 |
|---|---|
| 海量文件存储 | 分布式文件系统 **HDFS** |
| key-value 类型数据 | **HBase、MongoDB、Neo4j、Redis** 等 NoSQL |
| 全文检索 | 搜索引擎，如 **Lucene、ElasticSearch、Solr** |
| 多维分析 | **Kylin 或 Druid** 等 |

- 架构图：整体架构下层扩展为"分布式缓存 / 分布式数据库 / 分布式文件系统(HDFS) / NoSQL数据库(HBase/MongoDB/Neo4j) / 搜索引擎(Lucene/ES/Solr)"并列的存储层。
- **遇到的问题**：引入更多组件解决了丰富的需求，业务维度能够极大扩充；但随之而来的是一个应用中包含了太多的业务代码，**业务的升级迭代变得困难**。

名词解释：
- **NoSQL**：非关系型数据库的统称，相较传统关系数据库，通常牺牲部分事务/一致性能力换取更好的扩展性、灵活的数据结构（如键值、文档、列族、图）与更高性能，适合海量、半结构化/非结构化数据场景。
- **HDFS（Hadoop Distributed File System）**：Hadoop 生态的分布式文件系统，用于海量文件的可靠存储。
- **HBase**：基于 HDFS 的分布式、面向列的 NoSQL 数据库，适合海量 key-value/稀疏数据存储。
- **ElasticSearch/Solr/Lucene**：基于倒排索引的全文检索引擎（Lucene 是底层库，ES/Solr 是其上的搜索服务）。

#### P53
**11. 第十次演进：大应用拆分为小应用**

- 做法：按照业务板块来划分应用代码（如淘宝应用、天猫超市应用、天猫书店应用、天猫户外应用……），使单个应用的职责更清晰，相互之间可以做到独立升级迭代。
- 公共配置问题：应用之间可能涉及一些公共配置，可以通过分布式配置中心 **Zookeeper** 来解决。
- 架构图：Tomcat 层之上拆分为多个独立业务应用，各自连接到下方共享的存储层（分布式缓存/分布式数据库/HDFS/NoSQL/搜索引擎）。
- **遇到的问题**：不同应用之间存在共用的模块，若由各应用单独管理会导致相同代码存在多份，导致**公共功能升级时全部应用代码都要跟着升级**。

名词解释：
- **Zookeeper**：分布式协调服务，常用于分布式配置中心、服务注册与发现、分布式锁、集群选主等场景。

#### P54
**12. 第十一次演进：复用的功能抽离成微服务**

- 做法：如用户管理、订单、支付、鉴权等功能在多个应用中都存在，把这些功能的代码单独抽取出来形成一个独立的服务来管理，这样的服务就是所谓的**微服务**。应用和服务之间通过 HTTP、TCP 或 RPC 等多种方式来访问公共服务，每个单独的服务都可以由单独的团队来管理。
- 可通过 **Dubbo、SpringCloud** 等框架实现服务治理、限流、熔断、降级等功能，提高服务的稳定性和可用性。
- 架构图：在应用层与存储层之间新增一层"微服务"（用户管理服务、订单服务、支付服务……）。
- **遇到的问题**：不同服务的接口访问方式不同，应用代码需要适配多种访问方式才能使用服务；此外应用访问服务、服务之间也可能相互访问，**调用链变得非常复杂，逻辑变得混乱**。

名词解释：
- **微服务（Microservice）**：将系统拆分为多个可独立开发、部署、扩展的小服务，服务间通过轻量级协议（HTTP/RPC）通信，各自管理自己的数据。
- **Dubbo / SpringCloud**：常见的微服务/RPC治理框架，提供服务注册发现、负载均衡、限流、熔断、降级等能力。
- **限流/熔断/降级**：限流控制请求速率防止过载；熔断在下游故障时快速失败避免雪崩；降级在系统压力大时关闭非核心功能保核心可用。

#### P55
**13. 第十二次演进：引入企业服务总线 ESB 屏蔽服务接口的访问差异**

- 做法：通过 **ESB（企业服务总线）**统一进行访问协议转换，应用统一通过 ESB 来访问后端服务，服务与服务之间也通过 ESB 来相互调用，以此降低系统的耦合程度。
- 概念辨析（课件原话）：这种"单个应用拆分为多个应用，公共服务单独抽取出来管理，并使用企业消息总线解除服务间耦合"的架构，就是所谓的 **SOA（面向服务架构）**。SOA 与微服务架构容易混淆，因为表现形式十分相似。
  - 个人理解（课件观点）：**微服务架构**更多是指把系统里的公共服务抽取出来单独运维管理的思想；**SOA 架构**则是指一种拆分服务并使服务接口访问变得统一的架构思想，SOA 架构中包含了微服务的思想。
- 架构图：应用层与微服务层之间插入 "ESB企业服务总线"作为统一访问入口。
- **遇到的问题**：业务不断发展，应用和服务都会不断变多，应用和服务的部署变得复杂；同一台服务器上部署多个服务还要解决**运行环境冲突**的问题；此外对于如大促这类需要动态扩缩容的场景，需要水平扩展服务性能，就需要在新增的服务器上准备运行环境、部署服务等，**运维变得十分困难**。

名词解释：
- **ESB（Enterprise Service Bus，企业服务总线）**：一种中间件，作为统一的通信总线，负责协议转换、消息路由，屏蔽各服务接口差异，降低系统间耦合。
- **SOA（Service-Oriented Architecture，面向服务架构）**：一种将系统功能拆分为独立、可复用服务，并通过标准化接口/总线对外提供服务的架构思想。

**SOA 与微服务对比：**

| 维度 | SOA | 微服务 |
|---|---|---|
| 核心思想 | 拆分服务+统一接口访问方式（常配合ESB） | 把公共服务抽取出来独立运维管理 |
| 通信方式 | 通常经由 ESB 总线 | 服务间直接通信（HTTP/RPC），去中心化 |
| 关系 | 范围更广，包含微服务思想 | 可视为 SOA 思想的一种轻量级实践 |

#### P56
**14. 第十三次演进：引入容器化技术实现运行环境隔离与动态服务管理**

- 做法：目前最流行的容器化技术是 **Docker**，最流行的容器管理服务是 **Kubernetes（K8S）**。应用/服务可以打包为 Docker 镜像，通过 K8S 来动态分发和部署镜像。
- Docker 镜像可理解为一个能运行你的应用/服务的**最小操作系统**，里面放着应用/服务的运行代码，运行环境根据实际需要设置好；把整个"操作系统"打包为一个镜像后，就可以分发到需要部署相关服务的机器上，直接启动 Docker 镜像就可以把服务起起来，使服务的部署和运维变得简单。
- 架构图：新增"K8S 编排容器"层，管理各服务器上的 DK（Docker）容器（用户管理服务、支付服务、订单服务、商品推荐服务等按需组合部署在不同服务器上）。
- **遇到的问题**：使用容器化技术后服务动态扩缩容问题得以解决，但机器本身还是需要公司自身来管理；在非大促的时候，还是需要闲置着大量的机器资源来应对大促，**机器自身成本和运维成本都极高，资源利用率低**。

名词解释：
- **Docker**：容器化技术，将应用及其依赖打包为轻量级、可移植的镜像，实现"一次构建，到处运行"，提供进程级别的运行环境隔离。
- **Kubernetes（K8S）**：容器编排管理平台，负责容器的自动化部署、弹性伸缩、调度、故障自愈等。
- **镜像（Image）与容器（Container）**：镜像是静态的、只读的应用运行环境模板；容器是镜像的运行实例。

- 本页为演进路径的最后一步，未列出"新问题"（云平台承载已是该分享给出的终态方案）。

名词解释：
- **公有云（Public Cloud）**：由第三方云服务商提供、面向公众开放的计算资源池，用户按需租用、按使用量付费。

#### P59
**架构设计的原则**（共 11 条，逐条含义如下）

| 原则 | 含义 |
|---|---|
| **N+1 设计** | 系统中的每个组件都应做到没有单点故障（至少有一个备份/冗余节点） |
| **回滚设计** | 确保系统可以向前兼容，在系统升级时应能有办法回滚版本 |
| **禁用设计** | 应该提供控制具体功能是否可用的配置，在系统出现故障时能够快速下线功能 |
| **监控设计** | 在设计阶段就要考虑监控的手段 |
| **多活数据中心设计** | 若系统需要极高的高可用，应考虑在多地实施数据中心进行多活，至少在一个机房断电的情况下系统依然可用 |
| **采用成熟的技术** | 刚开发的或开源的新技术往往存在很多隐藏的 bug，出了问题没有商业支持可能会是一个灾难 |
| **资源隔离设计** | 应避免单一业务占用全部资源 |
| **架构应能水平扩展** | 系统只有做到能水平扩展，才能有效避免瓶颈问题 |
| **非核心则购买** | 非核心功能若需要占用大量的研发资源才能解决，则考虑购买成熟的产品 |
| **使用商用硬件** | 商用硬件能有效降低硬件故障的机率 |
| **快速迭代** | 系统应该快速开发小功能模块，尽快上线进行验证，早日发现问题大大降低系统交付的风险 |
| **无状态设计** | 服务接口应该做成无状态的，当前接口的访问不依赖于接口上次访问的状态 |

名词解释：
- **单点故障（SPOF, Single Point of Failure）**：系统中一旦失效就会导致整体不可用的组件；N+1 设计即通过冗余消除单点故障。
- **无状态服务（Stateless Service）**：服务本身不保存客户端的会话状态，每次请求都携带所需的全部信息，便于水平扩展和故障转移（与之相对的是有状态服务，如需要 session 保持）。
- **多活数据中心（Multi-Active Data Center）**：多个地理位置分散的数据中心同时对外提供服务（而非仅一主一备），任一机房故障不影响整体可用性。

#### P61
**标题：研究背景-项目需求**

- 随着互联网、物联网、社交网络的发展，智慧城市智能设施管理建设过程中产生了大量的**多模态场景数据**。
- 这些数据具有以下特征：
  - **多元**：人机物三元空间及其相互作用
  - **多维**：三维空间、时间、高维属性
  - **多尺度**：宏观与微观、概略与精细、户外与室内
  - 以及**全生命周期管理**的需求，给多模态场景数据组织带来挑战。
- 云存储中心无法支持边缘侧实时性需求强的应用（例如VR/AR应用），**需要发展面向特定场景可视分析任务的高效组织管理与调度**。
- 图示：边缘计算（Edge Computing）与云数据中心（Data Center/Cloud）的架构图；联网设备数量增长趋势图（美国计算机技术工业协会预测2020年达到501亿）；智慧城市多模态场景数据应用场景图（涉及机场、医院、购物、安防、物流、娱乐、地铁等）；"人类认知-社会空间/现实世界-物理空间/信息获取-信息空间"三元空间关系图；应用领域列举：智能工业、智慧交通、智能家具、智能电网、多元社交、可穿戴设备、智能医疗、智慧物流、智能环保。

#### P64
**标题：国内外研究现状——基础框架数据组织**

| 典型方法 | 存储系统 | 索引算法 | 查询算法 |
|---|---|---|---|
| 基于关系数据库（Oracle Spatial、PostGIS） | 底层存储针对空间数据优化较好，但不支持横向扩展，无法存储大数据量的基础框架数据 | 使用内置的R树、四叉树等空间索引，具有高效准确的空间数据检索能力，但索引结构复杂，维护成本高 | 支持丰富的空间查询类型，在数据量较小时查询效率较高 |
| 基于非关系数据库（Key-Value、Document、Column store） | 存储系统具有高扩展性和可用性，但底层系统对空间数据支持较弱 | 使用用户自定义的多层次空间索引，方法具有多样性且针对特定应用领域，但空间索引能力参差不齐 | 得益于多层次的索引结构，查询速度较快，但支持的空间查询类型有限 |
| 基于分布式文件系统（HDFS） | 常作为通用型的数据仓库，存储系统具有高扩展性和可用性，但底层系统对空间数据支持较弱 | 使用用户自定义的空间索引，算法结构简单，空间索引能力差 | 能够快速定位目标范围，但查询精度较低，支持的空间查询类型也有限 |

小结：现有的基于SQL、NOSQL和分布式文件系统的基础框架数据组织方案，难以支持多模态场景数据和多样化可视分析任务，**需要发展SQL、NOSQL和分布式文件系统混合的关联存储方法，实现优势互补**。

#### P65
**标题：国内外研究现状——动态过程数据组织**

| 典型方法 | 存储系统 | 索引算法 | 查询算法 |
|---|---|---|---|
| 存储组织系统：NOSQL-based | 具有高可扩展性，能够支持实时存储与更新，底层存储未针对时空数据进行优化 | 针对特定应用领域设计的算法，通用型不强且时空索引的查询模式固定 | 能够满足特定应用的实时查询与更新以及后续的分析需求 |
| 计算分析系统：Hadoop-based | 适合作为数据仓库，支持离线批处理，底层存储系统未对时空数据进行优化 | 时空索引支持计算并行，但时空索引的查询模式固定，基于磁盘的索引具有较高延迟，无法满足实时应用 | 对历史数据查询较快，但是对实时数据查询支持较差 |
| 计算分析系统：Spark-based | 适合作为实时在线分析系统，支持计算密集型应用，但底层存储系统未对时空数据进行优化 | 同上（磁盘索引延迟高，无法满足实时应用） | 对实时数据查询较快，支持复杂的在线时空分析查询，但对历史数据查询较慢 |

小结：面向实时计算分析的动态过程数据组织，因为传统树型索引针对外存I/O优化，不适用于内存，**亟需设计一种以内存为主的动态过程数据计算分析系统和面向内存的支持多种查询模式的多模态、多层次混合的时空索引方法**。

#### P66
**标题：国内外研究现状——关联关系数据组织**

| 典型方法 | 存储系统 | 索引算法 | 查询算法 |
|---|---|---|---|
| 关系数据库（PostgreSQL、MySQL） | 模式固定，适合管理结构化数据，通过外键表达数据之间的关系，不支持动态关系的组织 | 采用B-tree及其变种作为主要索引，索引维护成本高导致其不支持动态关系的组织 | 支持复合关键词查询，但是多张表连接（JOIN）操作效率低，不适合查询高度关联的数据 |
| 图数据库-原生（Neo4j、Titan，GraphDBMS） | 无模式，适合半结构化和非结构化数据，底层存储系统针对图数据进行优化，适合存储动态变化的关系数据 | 图的节点通过关系与其他节点相连，可以沿着关系快速访问其他节点，对索引依赖低 | 支持基于图的匹配查询以及可达性查询，能够支持复杂场景中多样化的查询任务 |
| 图数据库-非原生/多模型（OrientDB、ArangoDB） | 无模式，适合半结构化和非结构化数据，底层存储系统将关系数据序列化成其他形式存储，对于动态关系的维护成本高 | 图数据的序列化与反序列化，对索引依赖严重，导致其对于动态关系组织的资源开销大 | 同上（支持复杂场景中多样化的查询任务） |

小结：动态关联关系数据的组织管理是GIS领域的薄弱点，为快速有效地从多模态时空大数据中发现价值，诊断问题，检验预测和探索未知规律，**亟需研究基于图模型的动态时空关系数据组织方法，发展基于图匹配技术的时空对象查询算法**。

## ③ 名词解释清单（本讲精选）

1. **WKT（Well-Known Text）**：OGC 定义的几何对象文本表示格式，如 `POINT(116.404 39.915)`，用于空间数据的存储与交换。
2. **Haversine 公式**：由两点经纬度计算球面大圆距离的公式，考虑地球曲率，比平面距离更精确。
3. **ST_DWithin(geomA, geomB, d)**：PostGIS 空间函数，判断两几何对象距离是否在 d 之内，常配空间索引做"附近查询"的高效过滤。
4. **全要素空间数据库**：对某区域地上、地表、地下所有要素（基础地理要素+三维地形模型+三维要素模型）进行一体化管理的空间数据库，要素分类编码在国家地理信息分类编码标准基础上修改完善。
5. **地理实体数据**：以地形图为基础，对境界、政区、道路、水系、院落、建筑物、植被等内容实体化并赋予唯一编码，作为与其它行业和专题数据关联的基础。
6. **影像金字塔（Pyramid）**："金字塔+均匀分块、边缘补零"——对原始影像构建分辨率递减的多级结构，并在每级上对栅格数据分块存储，加速多尺度浏览。
7. **LOD（细节层次）**：三维模型按精细程度分级表达（体块→立面→门窗→室内），按区域重要性建一~四级模型。
8. **时态GIS（TGIS）**：在静态GIS基础上加入时间维，管理和分析随时间变化的地理实体与现象。
9. **时空数据模型三种**：时空立方体模型、连续快照模型、基态修正模型（基态+变化量）。
10. **读写分离**：主库负责写、从库负责读，通过中间件（如 Mycat）路由，提升数据库并发能力。
11. **分库分表**：按业务拆库、按数据量拆表，解决单库单表数据量过大导致的性能瓶颈。
12. **反向代理/负载均衡**：Nginx/HAProxy 将用户请求分发到多台应用服务器；更大规模用 LVS/F5（网络四层）与 DNS 轮询（机房级）。
13. **微服务**：将复用功能抽离为独立服务（Dubbo/Spring Cloud），服务间通过接口调用。
14. **ESB（企业服务总线）**：SOA 架构中统一的服务接入与消息转换总线。
15. **容器化（Docker+K8s）**：应用打包为镜像按容器部署，K8s 负责编排，配合云平台实现弹性伸缩。

## ④ 简答/论述题预测

## ⑤ 易错点提醒

1. `GEOMETRY(Point, 4326)` 中 4326 是 **SRID（WGS84 经纬度）**，别写成投影坐标系再直接用米做距离；ST_DWithin 在 geography 类型/投影坐标下单位才是米。
2. 空间索引写法是 `CREATE INDEX ... USING GIST(geom)`，**不是** B-tree；索引只做粗过滤，精确判断仍需空间函数（过滤—精炼两步）。
3. "全要素"强调**地上+地表+地下**垂直全覆盖，不只是平面要素齐全。
4. 管线要素建模固定套路：**管点(点)+管线段(线)** 成对出现。
5. 三维模型分级：一级最精细（地标/CBD），四级最粗（城中村/远郊）——方向别记反。
6. 金字塔 Level 0 是**原始最高分辨率**，层级越高分辨率越低。
