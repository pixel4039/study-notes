## 一、4.1 GIS总体开发介绍

### 1. GIS总体开发的两大体系（P4）

GIS应用系统开发从技术路线上分为两大类：
1. **基于商业GIS软件的开发体系**（以Esri ArcGIS为代表）
2. **基于开源GIS软件的开发体系**（PostGIS、GeoServer、QGIS、OpenLayers等组合）

这是本节的总纲，后续内容分别展开这两条线。

### 2. ArcGIS 产品体系（P6，重点图示）

ArcGIS围绕"以地图为中心"构建了一套四层产品体系：

```
Apps（应用）        —— ArcGIS平台的入口：桌面、浏览器、移动设备
   ↓
Portal（门户）      —— 平台的访问控制中枢，支持公有云/私有云/混合云
   ↓（ArcGIS Enterprise 内）
Server（服务器）    —— GIS Server / GeoAnalytics Server / Image Server / GeoEvent Server
   ↓
Data Store（数据存储） —— 平台内容的重要支撑
```

- **Apps**：ArcGIS平台的入口，用户通过桌面、浏览器、移动设备接入。
- **Portal（Portal to Portal）**：平台的访问控制中枢，提供"以地图为中心"的内容管理模式、分布式WebGIS构建、多维内容管理、跨部门协作分享、精细化访问控制、便捷的发现和使用资源。
- **Server（GIS/GeoAnalytics/Image）**：包括GIS Server、GeoAnalytics Server（大数据分析）、Image Server（影像服务）、GeoEvent Server（实时事件流处理），对接"在线资源和服务"，提供丰富内容、开放标准支持、构建云GIS平台、支撑大数据存储。
- **Data Store**：平台内容的重要支撑，即下面单独讲解的ArcGIS Data Store。

**ArcGIS用户/产品的三个层次**（P6右侧小字，属于易忽略但可能考的细节）：
| 层次 | 说明 | 举例 |
|---|---|---|
| 专业型 | 由Esri开发，供GIS专业人员使用 | ArcGIS Desktop、ArcGIS Pro、Orthomapping工具集 |
| 通用型 | 由Esri开发，为用户提供即拿即用的Apps及生产Apps的工具 | ArcGIS Earth、Explorer for ArcGIS、Operations Dashboard、Collector for ArcGIS、Survey123、Navigator for ArcGIS 等 |
| 业务型 | 由开发商定制开发完成 | Web APIs、ArcGIS Runtime SDKs、ArcGIS Engine、ArcGIS Pro for .NET |

### 3. ArcGIS Data Store（P7，名词解释重点）

**定义（课件原话，可直接背诵）**：
> ArcGIS Data Store是一个独立的数据库应用程序，是新一代的Web GIS平台的数据存储组件，让用户无需专业的数据库知识即可轻松配置出Web GIS平台的数据，主要用于存储Portal for ArcGIS背后托管的GIS Server的数据和图层，以及包括Portal for ArcGIS发布的托管要素/场景图层，以及各种服务器（实时、矢量大数据和栅格大数据）分析处理的结果图层等。

- **三种存储类型**（对应下方数据源关系图）：
  1. **关系型**（Relational）——对应关系型数据存储；
  2. **切片缓存型**（Tile Cache）——对应SceneView等缓存展示场景；
  3. **时空大数据型**（Spatiotemporal Big Data）——对接ArcGIS GeoEvent Server（实时数据）、ArcGIS Image Server（栅格大数据）、ArcGIS GeoAnalytics Server（矢量大数据），以及Insight for ArcGIS（分析工具）。

即：Data Store屏蔽了底层数据库操作细节，是ArcGIS Enterprise架构里"Portal—Server—Data Store"三段式中最底层的数据落地组件。

### 4. ArcGIS Runtime（P8，名词解释重点）

**定义（课件原话）**：
> ArcGIS Runtime是伴随ArcGIS 10.1发布而诞生的一个基于服务架构的全新产品，是一款轻量级桌面移动跨平台开发产品。

- **支持平台**：.NET、iOS、OS X、Android、JAVA SE、Qt QML（六大平台一套底层）。
- **分层架构**：
  ```
  .NET / iOS / OS X / Android / JAVA SE / Qt QML  （各平台特定层）
              ↓
      Runtime Common API   （跨平台统一API层）
              ↓
      C++ Runtime core     （C++核心运行时）
              ↓
   Mapping and GIS Services （地图与GIS服务能力）
  ```
- **特点标签**：轻量级、跨平台。核心能力包括：Map & Data（地图与数据）、3D（三维）、编辑、分析、操纵Portal。

### 5. ArcGIS Engine VS ArcGIS Runtime（P9，★高频对比题★）

| 对比项 | ArcGIS Engine | ArcGIS Runtime |
|---|---|---|
| 优点 | 可以实现ArcMap的大部分功能，包括操作复杂对象（如拓扑、几何网络），还可以专业级制图 | Esri大力发展，64位，部署简单，跨平台，展示效果好，支持矢量切片、三维场景、操纵Portal等 |
| 缺点 | 只能32位，部署繁重，不支持矢量切片、三维场景、Portal等 | 对于复杂对象和专业级制图的支持稍显不足 |
| 位数 | 仅32位 | 64位 |
| 部署 | 繁重 | 简单 |
| 跨平台 | 不支持 | 支持 |
| 新特性支持（矢量切片/三维场景/Portal） | 不支持 | 支持 |

> **记忆口诀**：Engine是"旧而全"（老技术但功能全，能做复杂拓扑分析和专业制图），Runtime是"新而轻"（新技术、轻量化、跨平台，但复杂对象支持弱）。

## 二、4.2 空间数据库应用系统开发介绍

### 1. 空间数据库系统的体系结构（P16，★重点图示，务必会画★）

```
                客户端应用   客户端应用   客户端应用
                     \         |         /
                      分布式空间数据库管理系统 —— 全局元数据
                               |
                          计算机网络
              ┌───────────────┼───────────────┐
        空间数据库管理系统  空间数据库管理系统  空间数据库管理系统
              |                  |                  |
          空间数据库          空间数据库          空间数据库
```

- 上半部分（虚线以上）是**逻辑/控制层**：多个客户端应用通过**分布式空间数据库管理系统**统一入口访问，该系统维护一份**全局元数据**（描述各分布节点的数据分布、模式信息）。
- 下半部分（虚线以下）是**物理/存储层**：由多个**空间数据库管理系统（各自独立管理一个空间数据库）**通过**计算机网络**互联，构成分布式存储节点。
- **考点**：该图体现的是**分布式空间数据库**的体系结构思想——客户端不直接面对具体某个物理数据库，而是通过"分布式空间数据库管理系统 + 全局元数据"这一中间层进行透明访问，屏蔽底层数据的物理分布细节。

### 2. 空间数据库管理系统（SDBMS）功能（P17，★六大模块，简答高频★）

课件图示为一个星形结构，SDBMS（空间数据库管理系统）居中，围绕的六个功能模块及两个外部接口分别是：

| 功能模块 | 作用 |
|---|---|
| 空间数据定义 | 定义空间数据的结构（数据库、工作区、要素层、地理要素对象四个层次，见下节） |
| 空间数据操作处理 | 对空间数据进行增删改查等操纵 |
| 空间关系处理与建立 | 建立要素之间的空间关系（如拓扑关系） |
| 空间数据定义（图中重复出现，强调其贯穿性） | 同上 |
| 空间数据可视化查询 | 支持地图可视化方式的数据查询 |
| 数据库维护 | 数据库的转储、恢复、重组织、性能监视分析 |

外部两个接口：**外部数据装载**（从外部导入数据进入SDBMS）、**空间数据备份**（从SDBMS导出备份）。所有模块最终服务于底层的**空间数据库**存储。

> 记忆技巧：这套功能模块本质上和第4部分要讲的"空间数据库操作"是同一逻辑的两种呈现方式——一个是功能架构图，一个是具体操作清单，二者可以互相印证记忆。

### 3. 空间数据库定义的四个层次（P18，★名词解释+简答高频★）

**总定义（课件原话）**：
> 根据应用的地理信息需求，数据库中的数据被组织成结构化的集合。数据按照这些预先定义的数据结构被放入数据库。这些数据结构使用数据库定义功能加以定义。

四个层次由粗到细：

| 层次 | 定义内容 |
|---|---|
| 1. 空间数据库定义 | 数据库名称、元数据，包含**工作区数**和**工作区名称** |
| 2. 工作区定义 | 包含**数据块数**、**数据块大小**和**数据块名称** |
| 3. 空间数据要素层定义 | 定义要素层（图层）的结构 |
| 4. 基本地理要素对象定义 | 定义最底层的地理要素对象 |

> 层次关系：**数据库 ⊃ 工作区 ⊃ 要素层 ⊃ 地理要素对象**，从粗粒度到细粒度逐级细化，这是空间数据组织的经典分层思想（类似文件系统的"磁盘-分区-目录-文件"层次）。

### 4. 空间数据库操作（P19-P21，★四大类操作，简答/填空高频★）

#### （4）空间数据维护操作
**定义（课件原话）**：
> 包括数据库的转储、恢复功能，数据库的重组织功能和性能监视、分析功能等。所有数据库的操作都是在控制程序的统一管理下进行，以保证数据的安全性、完整性以及多用户对数据库的并发使用。

- 核心功能：**转储、恢复、重组织、性能监视、性能分析**。
- 目的：保证数据的**安全性、完整性**以及**多用户并发使用**。

> **易错点**：四大类操作的层次与"空间数据库定义四层次"一一对应（数据库操作↔数据库定义，工作区操作↔工作区定义，要素层操作↔要素层定义），但**数据维护操作是横切各层次的管理性功能**，不是"基本地理要素对象"这一层的操作，容易与第四个定义层次混淆，注意区分"定义的第4层是对象定义"而"操作的第4类是维护操作"，两套分类角度不同。

### 2. C++ APIs（libpqxx，P24，★表格记忆★）

libpqxx是PostgreSQL官方的C++客户端库（对libpq的面向对象封装）。核心API：

| 序号 | API | 描述 |
|---|---|---|
| 1 | `pqxx::connection C(const std::string & dbstring)` | 用于连接数据库的类型定义。dbstring提供连接参数，例如：`dbname=testdb user=postgres password=pass123 hostaddr=127.0.0.1 port=5432`。连接成功后创建连接对象C，提供各种公共函数。 |
| 2 | `C.is_open()` | 连接对象的公共方法，返回布尔值：连接活动返回true，否则返回false。 |
| 3 | `C.disconnect()` | 断开用此方法打开的数据库连接。 |
| 4 | `pqxx::work W(C)` | 使用连接C创建一个**事务对象**，用于在事务模式下执行SQL语句。创建成功后分配给变量W。 |
| 5 | `W.exec(const std::string & sql)` | 事务对象的公共方法，用于执行SQL语句。 |
| 6 | `W.commit()` | 事务对象方法，用于**提交事务**。 |
| 7 | `W.abort()` | 事务对象方法，用于**回滚事务**。 |
| 8 | `pqxx::nontransaction N(C)` | 使用连接C创建一个**非事务性对象**，用于在非事务模式下执行SQL语句。 |
| 9 | `N.exec(const std::string & sql)` | 非事务性对象方法，用于执行SQL语句，返回一个结果对象（本质是记录的迭代器）。 |

> **易考对比点**：`pqxx::work`（事务模式，需要commit/abort）vs `pqxx::nontransaction`（非事务模式，直接执行，无需提交/回滚）——二者的核心区别在于**是否需要显式提交/回滚控制**。

### 3. libpq 完整连接代码示例（P25，代码题重点，需理解每一行）

```cpp
#include<windows.h>
#include "framework.h"
#include "postdbtest1.h"
#include <iostream>
#include<libpq-fe.h>

void ConnectPostgisdb()
{
    int lib_ver = PQlibVersion();
    //printf("Version of libpq: %d\n", lib_ver);

    const char* hostaddr = "localhost";      // 主机ip地址
    const char* port = "5432";               // 默认端口号
    const char* dbname = "postgres";         // 数据库名，默认为此值
    const char* user = "postgres";           // 用户名，默认为此值
    const char* password = "vgegis";         // 数据库密码
    const char* connect_timeout = "5";       // 连接等待时间，单位是秒，尝试超过此时长连接便不再连接

    // 该行应根据个人情况进行修改：使用PQsetdbLogin建立连接
    PGconn* conn = PQsetdbLogin(hostaddr, port, NULL, NULL, dbname, user, password);

    if (PQstatus(conn) == CONNECTION_BAD) {
        fprintf(stderr, "Connection to database failed: %s\n", PQerrorMessage(conn));
        PQfinish(conn);
    }

    int ver = PQserverVersion(conn);
    //printf("Server version: %d\n", ver);
    MessageBox(NULL, TEXT("数据库连接成功"), TEXT("数据库连接"), MB_OK);
    PQfinish(conn);   // 关闭连接，释放资源
}
```

**逐行考点解析**：
- `PQlibVersion()`：获取libpq库的版本号。
- `PQsetdbLogin(hostaddr, port, options, tty, dbname, user, password)`：libpq提供的**建立数据库连接**函数，返回一个 `PGconn*` 连接句柄。是本代码的核心API。
- `PQstatus(conn) == CONNECTION_BAD`：**判断连接是否失败**的标准写法，`CONNECTION_BAD`是libpq定义的连接状态常量。
- `PQerrorMessage(conn)`：获取连接失败的错误信息字符串。
- `PQserverVersion(conn)`：获取所连接的服务器版本号。
- `PQfinish(conn)`：**关闭连接并释放资源**，任何用PQsetdbLogin/PQconnectdb建立的连接最终都必须调用此函数，否则造成资源泄漏。

> 该函数体现了libpq最基本的编程范式：**建立连接→检查连接状态→执行操作→关闭连接**，这是C/C++直接使用libpq进行数据库开发的固定模板，务必掌握。

### 案例二：基于PostgreSQL+GeoServer+OpenLayers的台风路径WebGIS案例（P33-P41，★重点实操案例★）

#### 1. 环境要求（P34）

| 组件 | 说明 |
|---|---|
| ① PostgreSQL | 安装PostgreSQL后，还需安装对应版本的空间数据存储扩展**PostGIS**才能支撑空间数据的管理 |
| ② GeoServer | **定义**：GeoServer是OpenGIS Web服务器规范的**J2EE实现的社区开源项目**，利用GeoServer可以方便地发布地图数据，允许用户对特征数据进行**更新、删除、插入**操作。获取方式：官网下载zip压缩包，解压后双击`bin`目录下的`startup.bat`即可启动（**前提需要安装JDK**，因为是J2EE实现） |
| ③ OpenLayers | 使用5版本，可从官网下载压缩包或直接用CDN |
| ④ QGIS | 跨平台开源桌面GIS软件，本案例中主要用于**对底图进行符号化标注**（生成SLD文件） |

#### 2. 数据准备（P35）

- **底图数据**：世界地图、中国行政区划图、九段线、南海诸岛（经过处理，原始数据网上可下载）。
- **台风路径数据**：各个时刻台风的经纬度、风力、风速等信息。

#### 3. GeoServer图层发布（P35）

数据处理完成后，使用GeoServer进行地图服务发布。GeoServer管理界面中，可通过【Layer Preview】对已发布的图层进行多种格式预览（**OpenLayers、GML、KML等**）。

#### 4. GeoServer图层符号化——SLD（P36，★名词解释高频★）

**背景**：GeoServer默认发布的图层样式（默认灰色边界）与预期的美观效果（分省份彩色渲染）差距很大，因此需要**符号化**。

**SLD定义（课件原话，可直接背诵）**：
> SLD是**风格化图层描述器（Styled Layer Descriptor）**的简称。GeoServer图层风格支持ZIP和SLD两种格式，推荐使用SLD，更加方便快捷。GeoServer支持手写SLD也支持使用工具生成（QGIS就可以做到）。

**用QGIS生成SLD的操作流程**：
1. 打开QGIS，将数据文件加载到QGIS中；
2. 右键图层 → 选择【属性】；
3. 对【符号化】和【标注】进行设置，达到想要的显示效果；
4. 在符号化面板左下角点击【样式】下拉框 → 选择【保存样式】→ 保存为 **SLD格式**文件；
5. 将该SLD文件上传/关联到GeoServer中对应图层的样式（Style），即可让GeoServer按照QGIS中设计好的样式渲染输出。

> **考点提炼**：GeoServer本身只管"发布服务"，**样式（美观程度）的设计工作是交给QGIS完成、以SLD文件的形式传递给GeoServer**，这是两个开源软件分工协作的典型例子。

#### 5. GeoServer图层组发布（Layer Group）（P37-P38，★简答考点★）

**问题背景**：发布图层后GeoServer默认只能预览单个图层，而实际底图是"世界地图+行政区划+九段线+南海诸岛"等**多个图层叠加显示**的效果。

**解决方案——图层组（Layer Group）**：
> 可以同时调用多个图层进行图层的叠加，更好的办法是使用GeoServer的**图层组**，对多个图层进行叠加组合后**以单个服务的形式发布**。

**操作步骤**：
1. 单击【图层组】→【添加新的图层组】；
2. 设置Name（名称）、工作区（Workspace）、坐标参考系（**EPSG:4326**，即WGS84经纬度坐标系）；
3. 在【添加图层】中依次勾选要叠加的图层（如country、nanhai、nine_dotted_line、province、river、waring_line等），按先后顺序排列后保存；
4. 保存后在Layer Preview中会多出一个图层组条目，选择OpenLayers预览即可看到多图层叠加的最终效果。

> **考点**：**图层组的本质作用是"以单个服务的形式对外提供多图层叠加的组合效果"**，这样前端OpenLayers调用WMS时只需引用一个图层组名称，而不必自己在客户端逐一叠加多个图层请求，简化了前端开发。

#### 7. 前端渲染——OpenLayers调用GeoServer WMS服务（P40，★代码题重点★）

```javascript
var map = new ol.Map({
    layers: [
        new ol.layer.Tile({
            // 调用geoserver
            source: new ol.source.TileWMS({
                url: 'http://localhost:8080/geoserver/typhoon/wms',
                params: { 'LAYERS': 'typhoon:base_map', 'TILED': true },
                serverType: 'geoserver',
                crossOrigin: 'anonymous',
                transition: 0,
            })
        })
    ],
    target: 'map',
    view: new ol.View({
        projection: 'EPSG:4326',
        center: [120, 33],
        zoom: 4.5,
    }),
    controls: [new ol.control.MousePosition()]
});
```

**逐项解析**：
- `ol.Map`：OpenLayers的地图主对象。
- `ol.layer.Tile` + `ol.source.TileWMS`：**通过WMS协议、以瓦片(Tile)方式请求GeoServer地图服务**的标准写法。
- `url`：指向GeoServer的WMS服务地址，格式固定为 `http://<host>:<port>/geoserver/<工作区>/wms`。
- `params.LAYERS`：指定要请求的图层（或图层组）名称，格式为 `工作区:图层名`（此处即之前建的图层组`base_map`）。
- `params.TILED: true`：启用瓦片缓存请求方式，提升性能。
- `view.projection: 'EPSG:4326'`：与GeoServer发布图层组时设置的坐标参考系保持一致，这是**前后端坐标系必须匹配**的重要考点。
- `ol.control.MousePosition()`：显示鼠标当前地理坐标的控件。

#### 8. OpenLayers台风路径动态可视化（P41，★代码题重点★）

**背景**：台风数据存放在PostgreSQL中，属于**动态业务数据**（区别于底图这种静态WMS服务图层），因此不能直接用WMS方式请求，而需要"**后端提供API + 前端Ajax请求 + 前端矢量图层动态绘制**"的方式。

```javascript
// 添加一个绘制点线使用的Layer
var drawLayer = new ol.layer.Vector({
    // Layer所对应的source
    source: new ol.source.Vector(),
});
// 把Layer加入到地图图中
map.addLayer(drawLayer);

$.ajax({
    type: "get",
    dataType: "json",
    url: "http://localhost:8081/api/typhoon/find_typhoon?name=莫兰蒂",
    contentType: "application/json; charset=utf-8",
    success: function (result) {
        for (var i = 0; i < result.data.length; i++) {
            addPoint(result.data[i].lng, result.data[i].lat);
        }
    },
    error: function () {
        console.log("请求失败")
    }
});

function addPoint(lng, lat) {
    // 创建一个点
    var point = new ol.Feature({
        geometry: new ol.geom.Point([lng, lat]),
    });
    drawLayer.getSource().addFeatures([point]);
}
```

**逐项解析**：
- `ol.layer.Vector` + `ol.source.Vector()`：**矢量图层**，专门用于承载前端动态添加的点/线/面要素（区别于上面WMS的栅格瓦片图层）。
- `$.ajax(...)`：使用jQuery的Ajax异步请求，向自建的后端API（`http://localhost:8081/api/typhoon/find_typhoon`）请求指定名称台风的数据，后端从PostgreSQL中查询后以JSON格式返回。
- `addPoint(lng, lat)`：将每一条返回的经纬度记录，用 `ol.Feature` + `ol.geom.Point` 创建为一个OpenLayers点要素，再调用 `drawLayer.getSource().addFeatures([point])`加入矢量图层，从而在地图上渲染出台风路径的一系列点。

> **核心考点**：本案例展示了WebGIS开发中**两种不同的数据服务模式**：
> 1. **静态/半静态底图数据** → 通过GeoServer发布WMS服务 → 前端用`ol.source.TileWMS`直接请求瓦片图像；
> 2. **业务/动态数据（如台风路径）** → 自行开发后端API直接查询数据库 → 前端用Ajax获取JSON数据 → 用`ol.source.Vector`在前端动态绘制矢量要素。
>
> 这两种模式的选择依据是：**数据是否需要频繁按业务条件（如台风名称）筛选、是否需要前端灵活处理坐标点**——WMS更适合大范围通用底图展示，矢量动态绘制更适合业务专题数据的交互式展示。

### 2. 三维空间数据管理系统的三大组成部分（P46，★简答高频★）

   数据内容：地质模型数据、地下设施数据、地上模型数据、DEM数据、DOM数据、属性数据

- **三维空间数据库**：底层物理存储，可以是数据库管理系统（如Oracle）、并行数据库集群，也可以是普通文件系统，三者都可以通过各自的**SQL引擎**被上层访问。存储的数据内容涵盖地质模型、地下设施、地上模型、DEM（数字高程模型）、DOM（数字正射影像）、属性数据。

- **三维空间数据管理工具**：最上层的管理端应用，供管理员进行数据建库管理、数据更新、数据备份、并行管理、数据安全管理、日志管理。

**指南要求（课件原话，可能考填空）**：
> 三维数据管理支持**主流商业数据库和文件系统两种数据管理模式**，三维空间数据管理系统能够管理与动态调度**10万个以上三维静态目标**。

### 3. 三维GIS统一表示的数据模型（P47，★三层结构，核心考点★）

三维GIS的数据模型自底向上分为三层：

| 层次 | 内容 | 作用（课件原话提炼） |
|---|---|---|
| **几何层** | 点、线、面、体等基本几何元素；点由节点/复合点/引用对象等构成，线由折线段/复合线/线环等构成，面由多边形/参数曲面/多边形网格等构成，体由体元/多面体/复合体等构成，还包括TIN DEM、Grid DEM等地形几何表达 | **通过点、线、面、体等基本元素实现地上地下三维空间实体几何表达的统一** |
| **尺度层** | 三维对象（1）与三维模型（*）之间通过**多细节层次模型（LOD, Level of Detail）**关联，一个三维对象可对应多个不同细节层次的三维模型 | **通过多细节层次模型实现地上地下、室内室外三维空间对象的多尺度表达** |
| **语义层** | 对地形（数字正射影像、数字高程模型）、道路（交叉口、路段、车道带、行车道）、建筑（室内空间、通道、墙面、附属设施、室内/室外建筑）、地质（岩体、构造、地层）、管线（管点、管段）等专题模型进行面向专题语义的扩展 | **对地形、建筑物、道路网络、管线以及地质体等专题模型进行面向专题语义的扩展，对各种专题模型丰富的语义拓扑信息进行表达** |

> **考点提炼**：这是一个典型的"**几何—尺度—语义**"三层数据模型设计范式：
> - 几何层解决"**长什么样**"（形状统一表达）；
> - 尺度层解决"**在不同缩放级别下该展示多详细**"（LOD多细节层次，兼顾性能与精度）；
> - 语义层解决"**这是什么、有什么属性和拓扑关系**"（专题语义与拓扑扩展）。
>
> 三层自底向上通过"几何→（多细节层次模型/三维对象）→语义"逐级抽象，是理解整个三维空间数据库设计的关键框架，务必能够默写这张表并画出层次关系。

### 4. 三维空间数据一体化数据库模型（P48，ER图，理解为主）

以"三维空间数据库"为根，向下依次分解：

```
三维空间数据库
  └── 工程（Project）── 分区（Region）
         └── 数据集（Dataset）
               ├── 语义拓扑数据集
               ├── 三维模型数据集
               │      ├── 三维要素类 ── 三维空间实体 ── 三维几何模型(LOD) / 属性数据
               │      │        （共享模型、材质、纹理 单独管理，通过关联表提高关联检索效率）
               │      └── 对象类（关联类）── 非空间对象
               ├── 空间参考集
               └── 地形数据集
                      ├── DEM数据集 → DEM金字塔层数据集 → DEM数据块
                      └── DOM数据集 → DOM金字塔层数据集 → DOM数据块
```

**四个设计要点（课件原话提炼）**：
1. **三维空间数据语义拓扑**：实现复杂语义层级关系、语义拓扑关系等的管理；
2. **面向对象技术**：采用面向对象技术，以**三维要素类**实现三维空间数据的分类管理、直观表达以及自定义扩展；
3. **共享资源单独管理**：共享模型、材质、纹理单独管理，通过**关联表**提高关联检索效率（避免重复存储相同模型/材质/纹理，多个要素可共享引用）；
4. **地形数据金字塔分块**：地形数据采用**金字塔分层 + 均匀分块**的成熟方法进行管理（这与传统栅格金字塔/瓦片金字塔思想一致，用于支持多级LOD快速调度）。

#### （2）客户端-服务器数据调度流程（P53，★时序图，简答高频★）

以"三维可视化的数据查询请求"为例，完整流程为：

1. 客户端发起**三维可视化的数据查询请求** → 统一访问接口 → 客户端缓存与调度模块；
2. 客户端先在**客户端缓存**中查找：
   - 若**命中**：直接返回缓存中的数据，流程结束（最快路径）；
   - 若**未命中**：将请求转发给客户端/可视化集群的通信模块 → 发送数据请求消息到服务器端通信模块；
3. 服务器端收到请求后，数据请求线程先在**服务器端缓存**中查找：
   - 若**命中**：直接返回结果数据集；
   - 若**未命中**：交给**多线程调度模块**——进行**请求任务分解、SQL转换、多线程分配**，从**三维空间数据库**中获取数据结果集，并放入服务器端缓存；
4. 服务器空闲时，还会**预测下一步可能需要的数据**，主动启动**预调度多线程**，提前将预测数据放入服务器端缓存中（**预调度/预取机制**，提升后续响应速度）；
5. 结果数据集通过通信模块逐级返回：服务器端缓存 → 服务器端通信模块 → 客户端通信模块 → 客户端缓存（同时缓存一份）→ 最终返回给上层三维可视化请求。

> **核心考点（论述题）**：该流程体现了三个关键设计思想：
> 1. **多级缓存机制**（客户端缓存 + 服务器端缓存），优先命中缓存以减少数据库直接访问，提升响应速度；
> 2. **多线程任务分解与并行调度**，将复杂查询请求分解为多个可并行执行的子任务（SQL转换后分线程执行）；
> 3. **预调度/预测机制**，利用服务器空闲时段预测并预取下一步可能需要的数据，是一种典型的**空间数据缓存预取（Prefetching）优化策略**，在海量三维模型数据的实时可视化场景中尤为重要（因为三维模型数据量巨大，若等到用户请求时才现查数据库，会造成明显卡顿）。

| `CT3DPD_OraDBModel` | 地物模型数据库管理模块。实现数据库中三维地物模型所有类型数据的插入、删除、清空、查询等全部接口的实现 |
| `CT3DPD_OraDBTerrain` | 数据库地形管理模块。实现了数据库地形数据表空间的操作、数据表的操作，以及数据的插入和读取等操作 |
| `CT3DPD_OraDBSemanTopology` | 数据库语义拓扑管理模块。实现了数据库中语义拓扑建库和查询统一接口的实现，包括语义拓扑表空间、数据表的操作 |
| `CT3DPD_OraSpatialIndex` | 数据库空间索引管理模块，实现了各种类型空间索引在数据库的创建、删除、更新以及查询等功能。提供了丰富的空间索引管理接口，实现不同类型的空间索引在不同层次的创建和删除等操作 |

层次关系：`CT3DPD_OraDBInterface`（统一接口）→ 下接 `CT3DPD_OraDBModel`（模型管理）与`CT3DPD_OraDBTerrain`（地形管理）→ 二者都基于 `CT3DPD_OraDBBase`（数据库管理基础模块，已实现用户管理、初始化数据库环境等全部基本功能）→ 再往下是文件和数据库公用的底层模块 `GV3dFeature`、`GV3dIndex` → 最底层是 `GV3dCore`（核心）。

## ③ 名词解释清单

> 以下定义均以课件原话为主，可直接用于考试背诵。

1. **ArcGIS Data Store**：一个独立的数据库应用程序，是新一代的Web GIS平台的数据存储组件，让用户无需专业的数据库知识即可轻松配置出Web GIS平台的数据，主要用于存储Portal for ArcGIS背后托管的GIS Server的数据和图层，以及Portal for ArcGIS发布的托管要素/场景图层，以及各种服务器（实时、矢量大数据和栅格大数据）分析处理的结果图层等。

2. **ArcGIS Runtime**：伴随ArcGIS 10.1发布而诞生的一个基于服务架构的全新产品，是一款轻量级桌面移动跨平台开发产品，支持.NET、iOS、OS X、Android、JAVA SE、Qt QML六大平台。

3. **ArcGIS Engine**：Esri提供的可实现ArcMap大部分功能（包括操作拓扑、几何网络等复杂对象及专业级制图）的开发产品，仅支持32位，部署繁重，不支持矢量切片、三维场景、Portal等新特性。

4. **数据库应用系统（DBAS）**（本讲承接上一讲的概念，此处以空间数据库应用系统体现）：为完成特定任务，把与该任务相关的数据以某种数据模型进行存储，并围绕这一目标开发的应用程序，数据、数据模型以及应用程序整体称为一个数据库应用系统。

5. **空间数据库定义**：根据应用的地理信息需求，数据库中的数据被组织成结构化的集合，数据按照预先定义的数据结构被放入数据库，这些数据结构使用数据库定义功能加以定义，分为空间数据库定义、工作区定义、空间数据要素层定义、基本地理要素对象定义四个层次。

6. **空间数据维护操作**：包括数据库的转储、恢复功能，数据库的重组织功能和性能监视、分析功能等，所有数据库的操作都是在控制程序的统一管理下进行，以保证数据的安全性、完整性以及多用户对数据库的并发使用。

7. **PostGIS**：PostgreSQL对象关系数据库的空间数据库扩展程序，它增加了对地理对象的支持，允许在SQL中运行位置查询。

8. **Elasticsearch（ES）**：为所有类型的数据（结构化文本、非结构化文本、数字数据、地理空间数据）提供近乎实时的搜索和分析的引擎。

9. **MongoDB**：是为现代应用程序开发人员和云时代构建的基于文档的通用分布式数据库，具有丰富的JSON文档、强大的查询语言，对mesh（网格/三维模型）数据存储具备天然优势。

10. **Redis**：C语言开发的一个开源的可基于内存亦可持久化的日志型、Key-Value数据库，并提供多种语言的API，主要用于数据缓存。

11. **GeoServer**：OpenGIS Web服务器规范的J2EE实现的社区开源项目，利用GeoServer可以方便地发布地图数据，允许用户对特征数据进行更新、删除、插入操作。

12. **SLD（Styled Layer Descriptor，风格化图层描述器）**：用于描述地图图层渲染样式的一种格式规范，GeoServer图层风格支持ZIP和SLD两种格式，SLD可手写也可由QGIS等工具生成。

13. **图层组（Layer Group）**：GeoServer中将多个图层进行叠加组合后以单个服务的形式发布的机制，解决了默认只能预览单个图层、无法一次性获得多图层叠加效果的问题。

14. **Npgsql**：PostgreSQL的一个.NET数据提供程序，是C#/.NET环境下连接PostgreSQL数据库的标准驱动，依赖`Npgsql.dll`和`Mono.Security.dll`。

15. **多细节层次模型（LOD, Level of Detail）**：三维GIS数据模型"尺度层"的核心机制，通过一个三维对象对应多个不同细节层次的三维模型，实现地上地下、室内室外三维空间对象的多尺度表达。

16. **三维空间数据库统一访问接口（`CT3DPD_OraDBInterface`）**：数据库统一访问接口实现模块，用于实现数据统一访问接口，是三维空间数据引擎与其他组件和工具发生关系的唯一通道。

17. **预调度（预取）机制**：三维空间数据引擎在服务器空闲时，主动预测下一步可能需要的数据并启动预调度多线程提前将数据放入服务器端缓存中，以缩短后续实际请求的响应时间。

## ④ 简答/论述题预测

**1. 简述空间数据库系统的体系结构，并说明"分布式空间数据库管理系统"与各"空间数据库管理系统"之间的关系。**
- 参考要点：画出P16结构图；说明客户端应用通过分布式空间数据库管理系统统一访问，该系统维护全局元数据描述各节点数据分布；下层多个空间数据库管理系统各自独立管理一个物理空间数据库，通过计算机网络互联；体现"对上透明、对下分布"的分布式数据库设计思想。

**2. 空间数据库管理系统（SDBMS）有哪些功能模块？请结合空间数据库操作四大类逐一说明。**
- 参考要点：六大功能模块（空间数据定义、空间数据操作处理、空间关系处理与建立、空间数据可视化查询、数据库维护、外部数据装载/备份接口）；对应到具体操作层面即数据库操作（登录/装载/关闭/删除）、工作区操作（新建/打开/关闭/删除/修改范围/修改属性结构/输出结构/压缩）、要素层操作（新建/打开/关闭/删除/改名/合并/分离/重组/排序/建立空间关系/属性输入）、数据维护操作（转储/恢复/重组织/性能监视分析）。

**3. 对比ArcGIS Engine与ArcGIS Runtime的优缺点，并说明二者的适用场景。**
- 参考要点：见对比表；Engine适合需要专业级制图和复杂对象（拓扑、几何网络）操作的桌面GIS专业应用；Runtime适合需要跨平台、轻量级部署、三维场景和矢量切片展示的移动/桌面应用。

**4. 请描述一个基于开源软件的WebGIS系统从数据制作到最终展示的完整技术流程（以PostGIS台风路径案例为例）。**
- 参考要点：QGIS编辑生成shape文件 → 通过PostGIS将shape文件导入PostgreSQL → GeoServer连接数据库发布WMS服务（并用QGIS生成的SLD做符号化，用图层组合并多图层）→ 前端OpenLayers通过`ol.source.TileWMS`调用WMS服务展示静态底图，通过Ajax请求自建后端API获取动态业务数据（台风路径）并用`ol.source.Vector`动态绘制矢量要素。需要说明底图与动态业务数据两条技术路线的差异及各自适用场景。

**5. 什么是SLD？为什么GeoServer开发中常常需要借助QGIS来完成符号化工作？**
- 参考要点：SLD定义；GeoServer默认发布的样式简陋（灰色边框），而QGIS提供了强大直观的符号化和标注设计界面，设计好效果后可直接导出为SLD格式供GeoServer使用，是"专业工具各司其职"的设计分工体现。

**6. 简述三维GIS统一表示数据模型的三层结构（语义层、尺度层、几何层），并说明各层的作用。**
- 参考要点：见前表；几何层统一表达地上地下三维实体几何（点线面体）；尺度层通过LOD多细节层次模型实现多尺度表达；语义层对各专题模型（地形/建筑/道路/管线/地质）进行语义扩展与拓扑关系表达。三层自底向上、逐级抽象。

**7. 三维空间数据管理系统由哪几部分组成？各部分的作用是什么？**
- 参考要点：三维空间数据库（存储层，可为数据库管理系统/并行数据库/文件系统三种物理形态）、三维空间数据引擎组件（中间层，提供统一访问接口，含通信/索引/调度/缓存四模块）、三维空间数据管理工具（管理层，供管理员做建库/更新/备份/并行管理/安全/日志六项管理）。

**8. 请描述三维空间数据引擎中，一次三维可视化数据查询请求从客户端发出到最终获得数据的完整调度流程，并说明其中使用了哪些性能优化机制。**
- 参考要点：见P53时序图详解；性能优化机制包括：客户端缓存+服务器端缓存的多级缓存、请求任务分解与多线程并行执行、服务器空闲时的预调度（数据预取）机制。

**9. 三维空间数据一体化数据库模型中，为什么要将共享模型、材质、纹理单独管理并通过关联表关联？地形数据为什么采用金字塔分层+均匀分块的方法？**
- 参考要点：单独管理+关联表可以避免相同模型/材质/纹理在多个要素间重复存储，提高存储效率和关联检索效率（多对多关系用关联表建模是关系数据库的标准做法）；金字塔分层+均匀分块是海量地形/影像数据支持多级LOD快速调度、避免一次性加载整幅地形导致性能问题的成熟工程方法。

**10. 简述PostGIS在大湾区城市群EMPR多源数据库架构中扮演的角色，并说明为什么该系统不使用单一数据库而要混合使用ES、MongoDB、PostGIS、Redis四种技术。**
- 参考要点：PostGIS负责标准矢量空间数据的存储与SQL空间查询；ES负责全文检索与近实时分析；MongoDB负责文档型数据尤其是三维模型（mesh）数据存储；Redis负责高频访问数据的内存缓存。因为不同数据类型（全文检索、文档/网格数据、矢量空间数据、缓存热数据）各有最适合的数据库技术，采用"多元异构存储、各司其职"（Polyglot Persistence）的架构可以兼顾功能覆盖面和性能。

## ⑤ 易错点

1. **不要把"空间数据库定义的四个层次"与"空间数据库操作的四大类"记混**：前者是"定义/建模什么"（数据库→工作区→要素层→地理要素对象，纯粹的静态结构层次），后者是"能做哪些操作"（数据库操作/工作区操作/要素层操作/数据维护操作），数据维护操作**不对应**"基本地理要素对象"这一层，而是横跨全局的管理性功能，两套分类的划分角度不同，不能简单一一对应。

2. **ArcGIS Engine与ArcGIS Runtime的位数与部署方式容易记反**：Engine是**32位、部署繁重**（老技术），Runtime是**64位、部署简单**（新技术）。切勿记反。

3. **PostGIS的定位容易与GeoServer混淆**：PostGIS是**数据库扩展**（负责空间数据存储和SQL空间查询能力），GeoServer是**地图服务器**（负责把数据库里的数据发布成OGC标准的地图服务，如WMS/WFS）。二者是"存储层"与"服务层"的关系，不是同一层级的竞品。

4. **`pqxx::work`（事务）与`pqxx::nontransaction`（非事务）容易混用**：事务模式下必须显式调用`commit()`或`abort()`才能提交/回滚，而非事务模式下`exec()`执行后即直接生效，没有提交/回滚的概念，考试中若给出代码片段要能准确判断当前使用的是哪种模式。

5. **WMS静态底图与前端矢量动态绘制两种数据服务方式容易混淆**：WMS（`ol.source.TileWMS`）适用于**GeoServer已发布的、相对静态的底图图层**，前端拿到的是**服务器渲染好的瓦片图片**，无法在前端单独获取/操作某个具体地理要素；而`ol.source.Vector`+Ajax方式适用于**需要按业务条件筛选、需要前端灵活处理具体坐标点/要素**的动态业务数据（如按台风名称查询路径点）。二者不能互相替代，实际项目中经常同时使用（底图用WMS，业务数据用矢量动态绘制），这也是本讲台风路径案例前端代码分两段（P40底图+P41业务数据）呈现的原因。

6. **三维数据模型"语义层-尺度层-几何层"的顺序容易记反**：课件图示中语义层在最上（最抽象、面向应用），几何层在最下（最基础、纯几何形状），尺度层居中（通过LOD连接几何模型与语义对象）。答题时应按"几何层是基础，尺度层向上扩展多细节表达，语义层最终赋予专题含义"的逻辑来组织，而不是死记图片的上下方位。

7. **图层组（Layer Group）不是"新建了一种新图层"，而是"多个已有图层的组合发布方式"**：图层组本身不包含新数据，只是将多个已发布图层按顺序叠加、以单一服务名称对外统一提供，这一点容易被误解为"又生成了一份新数据"。

8. **不要把ArcGIS Data Store和ArcGIS Runtime混淆**：Data Store是**服务器端的数据存储组件**（存数据），Runtime是**客户端/移动端的开发SDK**（消费/展示数据），两者所处的架构层级和作用对象完全不同。

9. **本讲两份同名PDF文件的核对结论**：经检查工作目录，仅存在一份第十二讲PDF文件（`11-第十二讲 空间数据库系统开发(2) (1).pdf`），文件名中虽带有"(1)"字样容易让人误以为是"第二份重复文件"，但实际目录中并不存在不带"(1)"的另一份同名文件，因此**不存在需要去重的重复内容**，本笔记内容来自对这唯一一份58页PDF的完整通读。
