#### P10 热点之一：大数据处理技术的需求
特点：**海量、多结构、高负载、高可用、高可靠和低成本**
- 传统数据处理技术面临挑战——扩展性和成本
- 大数据技术**不能等同于**Hadoop和NoSQL
- 结构化数据与非结构化数据的融合：
  - 传统企业以结构化数据为主；互联网企业以半结构化/非结构化数据为主
  - 原有数据与新数据的融合
  - 内部数据与外部数据的融合
  - 高价值密度与低价值密度数据的融合
- 关系数据库技术在不断发展和积极创新，以支持大数据处理需求
- **单一技术无法解决好所有问题**（重要结论，易考简答）

#### P11 Hadoop与关系型数据库对比（雷达图，取长补短、优势互补）
对比维度：工具成熟度、强约束性、ACID事务、Security、数据格式多样化、数据稀疏性、ETL简化、数据存储经济性、吸收率、直通处理(STP)

| 维度 | RDBMS 优势 | Hadoop 优势 |
|---|---|---|
| 工具成熟度、强约束性、ACID事务、Security | 强 | 弱 |
| 数据格式多样化、数据稀疏性、ETL简化、数据存储经济性、吸收率、直通处理 | 弱 | 强 |

结论：
- 每个系统有自己的优势
- Hadoop不断进化：趋势——更完善的资源管理、更丰富的计算框架、最大限度使用内存、减少IO消耗
- 问题：平台组件多，开发和管理难度大，驾驭这样一个技术环境只能依靠强大的开发和运营团队，成本转移到日常运营

#### P12 案例：中国移动BSS核心系统数据处理平台的选择
- **数据特点**：几乎全部为结构化数据；实体众多，"强关系"特征
- **应用特点**：OLTP型复杂应用，少量生产报表；性能、可靠性、敏捷性是核心诉求；一定的伸缩性要求
- **IT模式**：依赖于SI（系统集成商）开发建设
- **结论/选择**：
  1. 以**传统RDBMS**为主，在部分应用探索结合NoSQL的使用
  2. 对于海量数据处理、话单稽核等应用可考虑结合数据特点采用Hadoop等新技术

#### P13 在适合与可控前提下探索分布式数据处理平台的应用
以传统RDBMS为核心平台，在部分领域探索Hadoop/NoSQL的应用，实现互补与集成：
- 历史详单进入NoSQL平台（如HBase+Hadoop架构：HMaster/HRegionServer/DataNode），提供直接查询——简单的数据关系、简单的查询与统计
- 对历史详单的统计分析结果可输入到RDBMS

**优缺点分析**：极好的扩展性与不错的性能，但功能简单，场景受限，对开发要求很高。
**可考虑的应用**：分布式历史详单存储与查询、少量基于详单的全量统计、大型文件备份。
**技术**：分布式数据平台（如Hadoop平台）、NoSQL（如HBase、Oracle NoSQL）

#### P14 技术创新驱动业务价值（三分体系对比表，重要）

| 技术 | 定位 | 特点 |
|---|---|---|
| Hadoop | **改变业务** | 横向扩展，低成本存储；采集任何数据；Map-reduce, SQL on Hadoop；批量处理，分析类应用 |
| NoSQL | **扩展业务** | 横向扩展，低成本存储；采集键值数据；通过键值发现数据；Web应用 |
| Relational（关系型） | **运行业务** | 横向扩展和纵向扩展；采集任何数据；SQL优势；企业级的交易型和分析型应用；安全和高可靠性 |

核心思想：三者互为补充（Hadoop + NoSQL + Relational），使用正确的技术支持业务需求。

#### P15 突破创新：融合的大数据解决方案——Big Data SQL
核心理念：统一的数据分析环境，**一条SQL可以访问所有数据源**（Bigdata SQL）
可访问的数据源包括：Hadoop、Oracle、Oracle NoSQL Database、NoSQL（MongoDB、Cassandra等）

#### P16 热点之二：全球集成基础架构和平台强劲增长
关键词：**集成设计，卓越性能**
IDC明确区分两类系统：

| 类型 | 定义 | 代表 |
|---|---|---|
| 集成平台（Integrated Platforms） | 硬件+软件；附加预集成打包软件与定制工程优化系统一同出售的系统 | Oracle是最大的集成平台系统提供商 |
| 集成基础架构（Integrated Infrastructure） | 没有针对特定工作负载优化；通用型、支持不同水平工作负载 | VCE是2013年排名第一的集成基础架构提供商 |

引用：IDC认为"集成系统是企业数据中心变革中最重要的发展之一"，该市场将继续保持增长势头。（来源：IDC Worldwide Integrated Infrastructure & Platforms Tracker, April 29, 2014）

#### P17 数据库云服务器成为高端计算池主流（趋势演进）
高端计算池发展趋势（自上而下）：数据仓库 → 联机交易处理 → 数据库整合 → 以闪存为中心 → **数据库云（DBaaS）**

# 第十五讲《空间数据库发展前沿》复习笔记（P19–P35）

#### P19　数据库多租户（Database Multitenant）
- 案例：云端的 **Oracle Database 12c 架构**，在 PDB（Pluggable Database，可插拔数据库）中实现数据库虚拟化。
- 核心思想：多个业务数据库（ERP、CRM、DW 等）共享同一套底层硬件（CPU、内存），是对"虚拟机"方案的补充。
- 三大好处：
  1. **应用无需修改**——在 PDB 中实现数据库虚拟化；
  2. **更低的运营成本**——像管理一个数据库一样管理多个数据库（统一升级、HA 高可用、备份），支持细粒度控制，数据移动/克隆/升级更容易；
  3. **更低的资本性支出**——单个服务器运行更多数据库，共享内存和后台进程。

#### P20　热点之四：内存技术成就实时企业
- **新技术突破：行、列双格式的数据库内存计算**（如 Oracle Database In-Memory）。
- 关键特性：
  - 同一张表在内存中**同时支持行格式和列格式**两种存储；
  - 两种格式**同时激活并保持事务一致性**；
  - 分析和报表使用**新的列格式**，OLTP 使用久经考验的**行格式**；
  - 可以**基于 OLTP 数据直接做分析**，消除数据搬迁的时延；
  - 压缩比例达 **2x～20x**，支持 RAC（Real Application Clusters）。
- 结论：**简单易部署，无需更改应用即可获得 2～100 倍性能提升。**

#### P21　不同技术适用于不同的数据和应用特点（对比表）
| 维度 | 传统集中式数据库 | MySQL+Sharding+x86 | NoSQL+x86 | HDFS+Mapreduce+x86 |
|---|---|---|---|---|
| 数据结构 | 结构化为主，模式固定 | 结构化为主，模式固定 | 半/非结构化，模式灵活 | 半/非结构化，任意模式 |
| 数据量 | 通常TB级及PB | 通过分片扩展至PB | PB级支持 | PB级支持 |
| 数据密度 | 高密度 | 高密度 | 低密度 | 低密度 |
| 数据关系 | 关系复杂 | 关系较简单 | 关系简单 | 关系简单 |
| 并发 | 高 | 高 | 高 | 低 |
| 实时性 | 高 | 高 | 高 | 弱 |
| 负载类型 | 混合型 | 混合性 | 随机访问 | 分析型 |
| 事务与一致性 | 强 | 强 | 弱 | 弱 |
| 索引能力 | 强 | 较强 | 一般 | 弱 |

| **代表应用** | 企业核心应用 | 电商前端交易 | 搜索、社交 | Web日志分析 |

#### P22　数据库技术发展的愿景——持续创新
- **海量数据的高性能与分布式扩展**；
- **一体化软件与硬件集成**；
- **一条 SQL 访问所有数据源**（Hadoop/NoSQL/数据库），简化应用开发；
- **不同的技术适用于不同的业务场景**，今天的 IT 是个"混合持久化"的态势，需要在业务需求与成本控制之间做平衡（重点结论，可能考简答/论述）。

#### P23　SQL、NoSQL、NewSQL 数据库演进时间线
- **传统 SQL 数据库**：
  1. 严格遵守 **ACID** 设计原则，一般很难实现硬件存储设备的"横向扩展"（多集群服务）；
  2. 严格的结构化数据类型。
  代表：MySQL、Oracle、SqlServer、DB2（1970s起）。
- **NoSQL 数据库**：
  1. 扩展能力强（分布式多集群服务）；
  2. 方便多用的数据类型承载能力（支持键-值、列族、文件集、图形数据等）；
  3. 数据服务与云计算紧密融合；
  4. **不支持 ACID 操作**；
  5. **符合 CAP 理论**。
  代表：Redis、HBase、Cassandra、MongoDB（2010起）。
- **NewSQL 数据库**：综合了 NoSQL 和 SQL 数据库优势，**同时具备 SQL 数据库的 ACID 操作及 NoSQL 对海量数据足够优秀的扩展和并发处理能力**。
  代表：Google Spanner、TiDB、Oceanbase（2015至今）。

#### P24　Google Spanner（Cloud Spanner）
- **定义（可背）**：Spanner 是谷歌公司研发的、**可扩展的、多版本、全球分布式、同步复制**数据库，支持**外部一致性的分布式事务**。
- 四大特点：
  1. **分片的关系型数据库**——手工分片困难，Cloud Spanner 是关系型数据库并直接提供横向扩展能力；
  2. **可扩展的关系型数据库**——是可扩展的关系型数据库而非 NoSQL，使用 Cloud SQL，从关系型数据库迁移到具备更好扩展性的关系型数据库；
  3. **管理性/高可用性**——高度自动化，支持在线 Schema 修改和打补丁，无计划停机时间，可达 **99.999%** 高可用 SLA；
  4. **跨 region**——写一次后自动复制数据到多个不同 region，可用单 region 或跨 region 实例满足全球数据库需求。

#### P26　2.1 实时GIS数据高性能管理——时空过程模拟与实时GIS系统
- 引入**传感器网络监测技术**，将实时观测数据直接接入相应的 GIS 数据库，能够快速提升 GIS 的**数据实时服务与分析能力**。
- 迫切需要突破：**实时GIS数据模型、时空索引、实时传感数据接入**等关键技术瓶颈，提高我国新一代GIS的核心技术水平和国际竞争力。
- 满足复杂自然与人文过程综合模拟及突发事件应急响应需要，提供全面技术支撑。
- 架构图（示范系统）：物联感知协同观测网 → 应急反应专家库和资源库 → 城市内涝应对方案（广州番禺水文模拟）；瓦斯监测预警→瓦斯危险源预警→瓦斯事故应急响应→瓦斯事故应急指挥（煤矿巷道瓦斯监测）；非点源污染驱动因子分析→流域用地变化模型→非点源污染模型→突发事件应急响应方案（滇池流域非点源污染模拟）。
- 底层支撑：**实时GIS软件平台** + **时空过程综合模拟系统**，二者共同依赖"实时数据集成表示的时空GIS模型"（通用时空GIS数据模型、传感观测实时数据的信息模型、扩展的时空GIS数据模型）。
- 实时GIS的关键技术（四层）：分布式时空数据访问机制、分布式时空数据的高效索引、分布式时空数据存储与管理、传感器地图实时接入与融合。
- 时空过程综合模拟关键技术：通用模拟模型、基础模拟模型、专业模拟模型、动态模拟技术、实时时空数据集成分析技术。

#### P27　实时GIS关键技术（细化）
- **设计完整的时空数据存储、管理、索引和访问机制**；
- **研发多源多时相传感数据的接入、融合工具**；

#### P29　时空大数据多态存储组织架构（图2-2）
- 核心思路：结合**分布式/并行文件系统、空间数据库、NoSQL数据库和大容量内存**，形成**多态存储组织架构**方案，解决大规模时空数据高效存储与快速访问问题。
- 要点：
  - 利用分布式/并行系统成熟的大文件组织管理技术存储海量大文件；
  - 利用内存随机读取、读写速度快的特点，用于矢量数据和地图瓦片的快速访问及服务；
  - 结构化的地理空间元数据，可用空间数据库集群管理；
  - 内存中直接存放"热点"数据，针对内存断电丢失数据的特点，需定时备份到硬盘文件或数据库作灾备，并建立内存可靠性保护，实现多副本管理。
- 架构层次（自下而上）：时空大数据存储模型（并行/分布式文件系统、内存、空间数据库、NoSQL数据库）→ 多态缓存机制（地图瓦片缓存、热点数据缓存、计算结果缓存、冗余备份缓存）→ 存储调度代价模型与地理数据负载均衡 → 虚拟访问接口 → 时空大数据逻辑模型 → 分布式计算/实时计算/批处理计算/地图渲染。

#### P30　实时GIS数据高性能管理——数据模型与存储引擎细节
- 数据源类型：矢量数据、栅格数据、网络数据、时空流数据、三维数据。
- **数据模型层**（借鉴多模型数据库 ArangoDB/Aerospike/Datomic 实现思路）：元数据、空间类型、索引、投影、分析操作，通过 JSON/BSON 与存储引擎交互。
- **多存储引擎层**（借鉴 MySQL 插件式存储引擎实现思路）：内存管理、存储管理、索引管理；下分三类引擎：
  - RDBMS引擎：PostGIS、MySQL、Oracle Spatial；
  - NoSQL引擎：Redis、MongoDB；
  - 文件系统：HDFS、GPFS。
- 逻辑映射示例：landuse属性表(ID,landuse)—geometry(面/线要素)—road属性表(ID,road)，通过逻辑映射建立关联；
- 存储模型示例：GeoHash堆存储 + 连接索引（ID-ID对照），通过物理映射实现空间索引查找。

#### P32　智慧城市时空大数据管理——全空间信息模型
- 目标：**建立全空间信息模型，实现地上下、室内外、二三维、虚实时空大数据一体化管理**，克服非关系数据库存储时空大数据存在的存储与访问效率低下问题，难以满足高并发、大数据量下的实时性要求，充分发挥非关系数据库的性能优势。
- 结构：以"地面"为中心，向上连接**地上实体**（室内、室外——建筑物n、道路n、植被n，各带时间标识/地址标识/空间位置/几何形状/名称类别等属性，并做时间序化、地址匹配），向下连接**地下实体**（地下管线、地下人防、地下交通，各带管线/仓储/停车场等信息，同样做时间序化、地址匹配）；地上实体"通过坐标承载"，地下实体"通过坐标悬挂"；两侧分别连接**物理世界**与**虚拟世界**。

- 核心论点：GIS 相当于智慧城市的"操作系统"，**基于虚拟城市地理空间环境集成融合城市数据**。
- 通过虚拟城市三维场景，将产权（房产证）、社保（社保卡）、户籍（户口本）等**人的属性数据**，与纳税（营业执照）、法人、交通（车辆）、手机、视频（监控）、卫星等**多源城市感知数据**统一挂接到地理空间位置上，实现城市数据的空间化集成与融合管理。

## 名词解释清单（本讲精选）

1. **DBaaS（数据库即服务）**：把数据库以云服务形式提供，按需分配、弹性伸缩、统一运维（课件给出五阶段演进路线）。
2. **数据库多租户/PDB**：Oracle 12c 可插拔数据库，多个业务库共享同一套硬件与实例资源，应用无需修改，降低运营与资本成本。
3. **内存计算（行列双格式）**：同一张表在内存中同时保持行格式（OLTP）与列格式（分析），事务一致，免数据搬迁，2x–20x 压缩。
4. **NewSQL**：兼具 NoSQL 可扩展性与关系数据库 ACID 事务的新型数据库（代表：Google Spanner）。
5. **Google Spanner**：谷歌全球分布式数据库，跨数据中心外部一致性事务（TrueTime），SQL+ACID+水平扩展。
6. **混合持久化（Polyglot Persistence）**：不同技术适用不同业务场景，RDBMS/NoSQL/Hadoop 并存互补，在业务需求与成本间平衡。
7. **实时 GIS**：对传感器等实时数据流进行接入、管理、分析与可视化的 GIS 技术（多态存储、流处理）。
8. **智慧城市时空大数据/全空间信息模型**：以统一时空框架整合城市多源数据，GIS 充当"智慧城市操作系统"。

## 简答/论述题预测

1. **数据库技术的四大热点方向**：大数据处理、集成系统（平台/基础架构）、数据库云（DBaaS/多租户）、内存计算——各展开一句。
2. **Hadoop/NoSQL/关系数据库三者的定位与互补**（Hadoop改变业务、NoSQL扩展业务、Relational运行业务；对比表见 P21）。
3. **SQL → NoSQL → NewSQL 的演进逻辑**：关系库强一致但难扩展 → NoSQL 弃事务换扩展 → NewSQL 兼得（Spanner 为例）。
4. **谈谈空间数据库的发展前沿**：实时 GIS 高性能管理 + 智慧城市时空大数据 + 分布式/云化 + 多态存储。
