# 阅读指南(Obsidian 阅读通道 A)

> 说明:本页由 Sonnet 按 Claude 拟定的合同(`阅读通道 A` 交付合同)代笔执行落地,frontmatter 按仓库 SCHEMA 要求署 `author: claude`;文字与配置内容已由 Claude 验收。

本仓库(`study-notes`)选定的阅读方式是「通道 A」:电脑本地 Obsidian + git clone + Obsidian Git 插件,让 GitHub 仓库本身就是 Obsidian vault,笔记与留言通过 git 提交/推送同步,不额外引入云盘或第三方笔记服务。

## PC 端(优先)

1. 安装 [Obsidian](https://obsidian.md)。
2. 用 `git clone git@github.com:pixel4039/study-notes.git`(或 GitHub Desktop 图形化 clone)把仓库拉到本地一个目录。
3. 打开 Obsidian → "Open folder as vault" → 选择上一步 clone 出来的目录本身作为 vault(不要再套一层子目录)。
4. 打开 Obsidian 设置 → Appearance → CSS snippets,确认 `reading` 已勾选启用(仓库已在 `.obsidian/appearance.json` 里预置 `enabledCssSnippets: ["reading"]`,正常情况下打开 vault 就是启用状态;如果显示未启用,手动勾选一次即可)。
5. 安装社区插件 **Obsidian Git**(仓库本身不预置任何社区插件配置文件,插件安装与个人偏好设置都在你自己电脑上完成,不会写回仓库配置)。建议配置:
   - Auto pull:vault 启动时自动 pull;
   - Auto push:每 10 分钟自动 push,或改成手动点击 push(两种都可以,看你希望多实时)。
6. 日常阅读入口 = 每条视频对应的**连续阅读**稿(`sources/bilibili/<MID>/transcripts/<year>/...__连续阅读.md`),不是逐句加时间戳的 canonical 字幕稿(那份是不可变原始层,用来核对用词,不作为默认阅读版面)。
7. 留言写法(两种,都可以用):
   - **随文留言(推荐)**:见下方「随文留言写法」一节,直接在连续阅读稿原文段落下方写,服务器收割脚本会自动搬进 annotation 页并带回链。
   - **直接写 annotation 页**:也可以直接打开该视频对应的 annotation 页(`annotations/bilibili/<MID>/<BVID>.md`),把想法追加到「留言栏(用户原话)」段落末尾,不新建文件、不改已有留言。
8. 写完存盘后,让 Obsidian Git 插件把改动 push 回 GitHub(自动或手动都行),服务器 cron 收割脚本或 Codex/Claude 下次处理入库时会读到。

## 随文留言写法

想在读到某一段时直接写想法,不用切到 annotation 页,可以在**连续阅读稿**里对应段落的下一行,用 Obsidian callout 语法写:

```markdown
> [!留] 这里是你的想法,可以多行,继续用 `>` 开头就行
> 第二行想法
```

- 语法:必须是 `> [!留]` 开头(注意"留"字,和 `[!note]`/`[!ai-note]` 等系统 callout 区分);后面接的文字就是留言正文,支持多行(每行都要以 `>` 开头)。
- 位置:写在你想评论的那一段原文**正下方**(隔一个空行也可以,脚本只认"上方最近的一段正文"),不用管前后有没有别的留言或 callout。
- 收割节奏:服务器 cron(装好后)大约每 30 分钟跑一次收割脚本;跑过之后,原位置会自动变成一行收录指针:
  ```markdown
  > [!留] ✓ 已收录 → [[../../../../../annotations/bilibili/<MID>/<BVID>.md#^an-<bvid>-p01-00N]]
  ```
  真正的留言正文(带日期、原文引文、canonical 时间戳回链)已经被搬进对应视频的 annotation 页,点开指针链接就能跳过去看;连续阅读稿上不会再重复保留一份正文。
- 批准蒸馏:如果这条留言不是普通感想,而是"这一段值得沉淀成 Wiki 候选",就写成:
  ```markdown
  > [!留] 批准蒸馏 简单说一句为什么值得
  ```
  收割脚本会把它单独放进 `wiki/_candidates/_distill-queue.md` 排队(而不是进 annotation 留言栏),原位置替换为 `> [!留] ✓ 已入蒸馏队列`。队列本身**不会**自动生成 Wiki 正文——真正合并仍要走 D0–D5 审批,由 Sol/Claude/Codex 在你开口处理这批队列的时候才消耗 token。
- 主人亲笔的随文留言直接算确认态,不会被标 `tentative`(`tentative` 只用于 AI 转写口述的场合)。

## 边读边问怎么问(带记忆的陪学通道)

想边读边问、需要 AI 当场回应或讨论的,不用等收割脚本,直接在 Obsidian 旁边开你平时用的 QQ/微信窗口找 Hermes(Sol)聊——这条通道本来就带 Scope Recall 记忆,也能读到 study-notes 当前现场,不需要另外新建任何聊天入口。

- 提问习惯:带上 **BVID** 和你想问的那句/那段**原文引文**,比如"BV1syNg6DEFi 里说'给 AI 项目地图'那段,是不是也适用于……",这样 Sol 能直接定位到具体段落,不用你手动截图或复制大段字幕。
- 对话本身会自然沉淀进 journal → episode 链路(已批的陪学设计),不需要你额外整理。
- 目前**不**内嵌聊天窗口或开公网 API 到 Obsidian 里(成本高、记忆容易割裂、安全面也更大);如果以后这条摩擦真的积累到必须解决,会重新评估已规划的 Hermes Pocket PWA 方案,单独立项,不在本次范围内。

## 移动端(iPad / 手机,体验有限,PC 优先)

- iPad/手机可以装 Obsidian,再打开同一个 GitHub 仓库作为 vault。
- iOS 上 git 同步推荐两条路:
  - **Working Copy**(专门的 iOS git 客户端,配合 Obsidian 用文件 Provider 打开同一份 clone);
  - 或 **Obsidian Git 插件的移动端版本**(功能和易用性都弱于桌面版,冲突处理更麻烦)。
- 这条路线目前只作为补充,遇到冲突或体验问题优先回到 PC 端处理,不必强求移动端功能对等。

## 分支说明

远端默认分支是 **master**,不是 `main`;clone、pull、push 都对着 `master` 操作即可,不需要额外切分支。
